

# 第2章：適用条件と準備作業

## 2.1 本手法がエージェントプラットフォームに要求する最低条件

実際に作業を始める前に、使用しているエージェントプラットフォームが以下の能力を備えているか確認してください。これらはアーキテクチャを実装するための物理的基盤です。

### 2.1.1 必須能力（いずれか一つでも欠けると完全実装できません）

| 能力               | 説明                                   | 用途                                                     |
| :----------------- | :------------------------------------- | :------------------------------------------------------- |
| **永続化メモリ**     | セッション間で格納・読み取り・更新が可能なメモリシステム       | L1メタ認知フレームワーク、L2ユーザーロジック、L3シーン規則、積分制御テーブルを格納 |
| **モジュール化機能**     | 作成・呼び出し・更新が可能なスキル/Skill/GPTs 等 | 実行層（デリバリーゲート、前フィードバック起動、クローズドループ整理など）を格納             |
| **十分なコンテキストウィンドウ** | 推奨100kトークン以上                  | セッション内のフィードフォワードリスト、回避記録、タスク実行プロセスを収容             |

もしプラットフォームが以上の3つをすべて備えていれば、本チュートリアルの全内容を完全に実装できます。

### 2.1.2 加点能力（あればより強固、なければダウングレード運用）

| 能力         | 説明                                         | 不足時の代替策                                 |
| :----------- | :------------------------------------------- | :----------------------------------------------- |
| **セッション検索** | 過去のセッションの内容と修正記録を検索               | 修正のたびに重要な教訓を手動でメモリに書き込み、前フィードバード素材とする |
| **ツール呼び出し** | エージェントが外部ツール（検索、コード実行など）を能動的に呼び出せる | 範囲は限定されますが、コアアーキテクチャへの影響はありません                       |

### 2.1.3 プラットフォームにスキル機能がない場合の対処法

ChatGPTの基本版や簡単なチャットインターフェースなど、一部のプラットフォームにはスキルモジュールがありません。この場合は「メモリシミュレーション法」を採用できます：

- スキルの内容を構造化されたメモリエントリとして格納する（例：`[Skill-デリバリーゲート] 内容：...`）
- L1メタ認知フレームワークに強制ルールを追加する：「デリバリー前に、[Skill-*]とマークされたすべてのメモリエントリを読み取り、実行する必要がある」
- 欠点：メモリ容量を占有し、エージェントが能動的に検索する必要がある
- 利点：ロジックは完全に維持され、軽度の使用シーンに適している

## 2.2 自己評価：あなたは本チュートリアルを使用するのに適していますか

### 2.2.1 強く推奨されるユーザープロフィール

- **高精度協力ユーザー**：エージェントの出力に対して安定した品質要求がある（執筆スタイル、コード規約など）
- **反復タスクユーザー**：エージェントと同種のタスクを頻繁に行っている（毎日/毎週の類似ワークフローなど）
- **システム思考ユーザー**：初期導入コストを投資して長期的な協力効率を得る意欲がある
- **境界明確ユーザー**：好みと紅線を明確に表現でき、エージェントがミスをしたときに指摘することを厭わない

### 2.2.2 あまり適さないユーザー

- 一回限りで再利用の必要性がないランダムな質問応答のみを行う
- 使用シーンが非常に広範にわたり、安定した規則をまとめるのが困難
- エージェントとの長期的な協力関係を構築したくない

## 2.3 開始前の準備作業

正式にデプロイする前に、以下の4つの準備を完了すると、その後の調整コストを大幅に削減できます。

### 2.3.1 現在のエージェント状態を把握する

以下の2つのコマンドを実行し、現在の状態をベースラインとして記録してください：

```
現在保持しているすべてのスキル、およびその内容の要約を列挙してください。
```

```
メモリに現在格納されているすべての内容を列挙し、利用率を評価してください。
```

【記録のポイント】：
- 現在いくつのスキルを持っていますか？ それらはまだ使用されていますか？
- メモリの利用率は何％ですか？ 内容は混在していませんか？
- 大量の古い情報がありますか？

### 2.3.2 固定化したい好みとルールを列挙する

簡単な表を使って、エージェントに長期的に守らせたい原則を列挙してください。このステップは、曖昧な「感覚」から操作可能なルールを抽出するのに役立ちます：

| 次元     | あなたの原則               | 例                   |
| :------- | :--------------------- | :--------------------- |
| コミュニケーションスタイル | 簡潔で直接的、定型的な挨拶は使用しない   | 「お役に立てればと思います」を出力しない | 
| 出力フォーマット | コード優先、検証方法を添付  | ソリューションを出力する際、テストステップを含む | 
| 意思決定の好み | リスクを先置きし、問題を先に露出させる | 最初にリストアップしてからメリットを語る | 
| 美的志向 | 冷徹な語り口、AIのまとめを拒否 | 小説の結末は余白を残し、高揚させない |

一度に完全に網羅する必要はなく、4〜6条のコア原則で十分です。エージェントはその後のクローズドループの中で継続的に補充してくれます。

### 2.3.3 頻繁に実行する3〜5種類のタスクタイプを思い出す

- 例：短編小説を書く / コードレビュー / 運用トラブルシューティング / アーキテクチャ設計 / データ分析
- 各タイプについて一つの文章を試みに書き、この種のタスクを「正しく」行うための重要な規則について記述してください

【例】：
- 短編創作：「情緒の定調はプロットの展開よりも優先され、結末は必ず余白を残す必要がある。」
- コードレビュー：「最初にアーキテクチャの問題を見つけ、次に命名規約を見る。一種類の問題を見つけるたびに修正の提案を与える。」
- トラブルシューティング：「最初にログを見て根本原因を特定し、憶測で待たない。不確定なときは『検証待ち』とマークする。」

これらはあなたのL3シーン規則の初期コンテンツとなります。

### 2.3.4 現在の状態をバックアップする

デプロイを開始する前に、**現在のメモリとスキルの状態をバックアップしてください**。具体的な方法はプラットフォームに依存します：

- プラットフォームがエクスポートをサポートしている場合、まずエクスポートしてください
- サポートしていない場合、現在のメモリとスキルリストを手動でローカルドキュメントにコピーしてください
- バージョン管理機能（一部のプラットフォームのメモリスナップショットなど）がある場合は、復元ポイントを作成してください

このステップの目的は、デプロイプロセス中に混乱が生じた場合、初期状態に復元してやり直せるようにすることです。

## 2.4 準備完了のサイン

以下の3つの質問に答えられるとき、準備フェーズは完了しており、第3章に進むことができます：

1. 私のエージェントプラットフォームは永続化メモリ＋スキルモジュールを備えていますか？（はい / いいえ / 部分的に備えている）
2. いくつかのコアの好みと原則を整理しましたか？（推奨：≥4条）
3. 何種類かの高頻度タスクタイプを列挙し、各タイプについて初步的な「規則判断」をしましたか？（推奨：≥3個）
