# 第7章：よくある質問と拡張思考

これはチュートリアルの最終章で、デプロイと実行中に遭遇する可能性のある典型的な障害とその解決方法、およびこのフレームワークをより複雑なシナリオに拡張するアイデアの2つの部分をカバーします。

---

## 7.1 デプロイ関連の問題

### Q1：L1を書き込んだ後、Agentが5つの鉄則に従って実行しません。どうすればよいですか？

**よくある原因**：
- メモリがトップに固定されておらず、その後に書き込まれた内容で上書きまたは希薄化された
- 長い対話によりコンテキストウィンドウが圧迫され、L1が切り捨てられた
- Agentプラットフォームがメモリ書き込みのタイミングに遅延がある

**トラブルシューティング手順**：
1. AgentにL1の5つの鉄則を繰り返させます。完全に繰り返せない場合、メモリの書き込みまたは読み取りに問題があります。
2. 正しく繰り返しても実行しない場合、新しいセッションの開始時に次のように宣言します：``今回のセッションを開始する前に、まずmemory中のL1メタ認知フレームワークの5つの鉄則を復習してください。``
3. 問題が続く場合、L1のコアルールを1〜2文に精簡し、プラットフォームの「システム指示」または「system prompt」領域（サポートされている場合）に書き込むことを検討してください。二重保険となります。

---

### Q2：Agentは`skill_manage`機能または類似のスキル管理ツールを持っていないと言います

**2つの解決策**：

**解決策A（推奨）**：メモリでスキルをシミュレートします。
- memoryに`[Skill-スキル名]`とマークされたメモリエントリを作成し、内容がスキルの操作手順です。
- L1にルールを追加します：``各スキルの実行が必要な場合、memoryから対応する[Skill-*]エントリを検索し、手順に従って厳密に実行してください。``
- 欠点：メモリ容量を占有する；長所：ロジックが完全です。

**解決策B**：アーキテクチャを簡素化し、L1 + L2 + L3 + スコア表のみを使用します。
- スキル層の自動化を諦め、タスク開始時に手動でAgentに自己チェック手順に従うようリマインドします。
- 軽度の使用または短期間の協力に適しています。

---

### Q3：L2/L3を初期化する際、Agentが私の要約が完全に不正確です

これはアーキテクチャに問題があることを意味するのではなく、むしろAgentが以前もあなたの理解に偏りがあったことを示しています——これこそがこのチュートリアルが解決すべき問題です。

**対処方法**：
- Agentの要約を全面的に受け入れないでください。一つずつ修正し、これを「校正対話」として扱います。
- 修正後、Agentに明確に伝えます：``あなたは以前私の理解に偏りがあり、私はL2/L3の内容を修正しました。memoryでは私の修正を基準としてください。``
- その後の各ループレビューで、Agentは自動的に理解を更新します。通常、2〜3回のレビューの後、L2/L3の精度が大幅に向上します。

---

### Q4：スコア制御表のエントリが増え続けています。どうすればよいですか？

これはシステムが正常に動作しているシグナルです——あなたが実際に継続的に協力しており、Agentが継続的に学習しています。

**維持戦略**：
1. 各「離陸」レビュー時に、スコア表を確認します。
2. 既に下沉しており安定して動作しているエントリについては「固化済み」としてアーカイブし、アクティブエントリ数を減らします。
3. 分類が間違っている、または適用されなくなったエントリは手動で削除します。

```
スコア制御表の「終了AIまとめ」（已下沉、6ヶ月間再発なし）をアーカイブ領域に移動し、アクティブエントリに計上しない。
```

---

### Q5：デプロイ後にフィードフォワードチェックリストが長すぎて効率に影響します

フィードフォワードチェックリストの長さはタスクの複雑度に比例しているべきです。単純なタスクでも長いチェックリストを出力する場合、調整方法：

**簡略化指示**：
```
feedforward_startupを修正します：単純なタスク（翻訳、短いQ&Aなど）の場合、フィードフォワードチェックリストを1行に圧縮します——「避坑：xxx」。複雑なタスクのみ完全な起動チェックリストを出力します。単純なタスクかどうかは実行時に判断して私に伝えます。
```

タスク開始時にいつでも「チェックリストをスキップして直接実行」と言ってもAgentが実行します。

## 7.2 実行中によくある問題

### Q6：Agentが私が2回訂正しなかった同種類の間違いを連続で犯しましたが、下沉がトリガされませんでした

**直接トラブルシューティング**：
```
スコア制御表の現在の完全な状態を出力してください。
```

**可能な原因**：
- Agentがあなたの最初のフィードバックを正しく分類しませんでした（例：D類偶発に分類）。
- Agentが2回のフィードバックを異なるエラーパターンとして分類し、同じエントリにしませんでした。

**修正方法**：
```
[2回のフィードバックを記述]を同じエラーパターンに統一分類します：「[8文字以内のパターン名]」。スコア表を更新し、下沉閾値に達したかどうかを確認してください。
```

---

### Q7：Agentはデリバリーゲートレポートで常に「全員通過」と言いますが、問題を発見しました

これはゲートのチェック項目が細かくないか、Agentがゲートを実行する際におざなりにしていることを示しています。

**強化方法**：
1. 問題を発見した時点で指摘します：``delivery_gate第3段階で「注目に値」という定型句を発見できませんでした。理由を説明し、ゲートの検出基準を更新してください。``
2. 特定の問題が繰り返し発生する場合、ゲートにターゲットを絞ったチェックルールを追加します：
```
delivery_gate第3段階のチェック項目に追加する：「ある程度」「言える」などのあいまいな限定語が含まれていますか？発見したら削除してください。``
```

---

### Q8：ループレビューの出力がテンプレート化すぎて、真に価値のある根本原因分析が欠如しています

これAgentが長い期間内に真の問題に遭遇していないか、プロセス設定によりAgentが「考え」よりも「記入」に傾いている可能性があります。

**活性化方法**：
- 「離陸」時に具体的な指示を追加します：``今回のレビューは[特定の問題]に重点を置き、深刻な根本原因分析を行ってください。``
- 長期間問題がない場合、Agentに伝えます：``今回のループで修正が必要な問題が発見されない場合、3文の簡易報告に短縮でき、完全な9工程は不要です。``

---

### Q9：Agentのメモリがだんだん増えていき、最終的に制限を超えることを心配しています

これはアーキテクチャ設計で既に考慮された問題です。3つの組み込みメカニズムが膨張を抑制します：

1. **階層分離自体が膨張抑制設計**：原則と規則のボリュームは操作手順と事例よりはるかに小さいです。
2. **L2/L3エントリ上限**：L2上限10条、L3各シーン上限3条。超過時にクリアを促します。
3. **ループレビュー中の浄化ステップ**：各レビューで混乱した内容をチェックして移行します。

心配な場合、定期的に手動でクリアできます：
```
memory中で過去30日間アクセスされなかったエントリをリストアップし、削除できるか評価してください。
```

## 7.3 拡張思考

### 7.3.1 単一Agentから複数Agent協力へ

このアーキテクチャのコア——制御論4概念 + 階層分離——は自然に複数Agentシーンに適用されます。

**拡張アイデア**：
- **マスターAgent**（オーケストレーター）：L1メタ認知フレームワークを持ち、タスク配布と品質管理に責任を持ちます。
- **サブAgent**（実行者）：各々が特定のシーンのL3規則と対応するSkillを持ちます。
- **共有メモリ領域**：L2ユーザー基礎ロジックは全てのAgentに見え、異なる役割が統一した美的と原則に従うことを確保します。
- **統一スコア表**：全てのAgentのネガティブフィードバックが同一スコア表に集約され、マスターAgentが下沉決定を実行します。

これは本質的に「階層分離」の「戦略/実行」境界を、「マスターAgent / サブAgent」の物理的境界にマッピングするものです。

---

### 7.3.2 他の理論フレームワークの導入

チュートリアル全体で銭学森エンジニアリング制御論を基礎フレームワークとして使用しました。しかしL1はクローズドシステムではありません——あなたの学習移行能力を通じて「移行提案」ステップで新たな理論を継続的にシステムに導入できます。

コメント欄で既に現れた拡張の試み：

| 導入理論 | 応用方向 | 対応L1の可能性のある拡張 |
| :-------------------- | :--------------- | :----------------------------------------- |
| 積極的な投資管理 | 意思決定とリソース配分 | L1に「動的再平衡」原則を追加 |
| ベイズ推論 | 不確実性下の判断 | スコア制御に「事前確率」修正フィードバック重みを追加 |
| ケリー基準 + シャノン情報論 | 情報価値評価 | 自己監視に「情報密度」チェックを追加 |
| TRIZ発明原理 | 矛盾解決 | 根本原因分析に「物理矛盾 vs 技術矛盾」判定を追加 |

新理論の導入方法：「離陸」レビュー時、Agentが移行提案をする場合、または明確なアイデアがある場合、次のように言います：
```
L1に[理論名]を導入したいです。原則は[一言説明]です。既存の4概念フレームワーク下で第6条鉄則を追加する方法を設計し、Skill実行にどのように影響するか示してください。
```

### 7.3.3 フレームワークの適用境界の再考

コメント欄に真剣に扱うべき批判がありました：**「エンジニアリング制御論は現在のAgentにとって必須ではないかもしれない、あるいはこれは単にコンテキストにLLMにステップを追加するだけで、強い制約ではない」**

このチュートリアルの完全展開を経て、この批判に対して次の答えを出せます：

1. **「LLMにステップを追加する」自体が有効な制約**です。LLMはプロンプト可能なシステムであり、構造化されたコンテキスト制約は実際にその行動分布を変えることができます。効果はハードコードされた強制性から来るのではなく、継続的な構造化ガイダンスから来ます。

2. **アーキテクチャの価値は単回実行にではなく、長期安定性にあります**。単回で見ると、Agentに何のフレームワークも与えず、言いつだけで適格な結果を得られます。しかし連続50回のタスク後、アーキテクチャのあるシステムはより良く収束し、ドリフトは小さいです。これは保険であり、燃料ではありません。

3. **全てのシーンに必要ではありません**。軽度な、一回限りの、高度にランダムな対話はこのアーキテクチャを必要としません。その適用境界は：**長期協力 × 高精度要求 × 規納可能な規則のタスクタイプ**。あなたがたまたまこの領域にいる場合、投入した構築コストは繰り返しの間違い修正を減らす利益で回収されます。

---

### 7.3.4 フレームワークの自己終了条件

健全なシステムは「いつ簡素化されるべきか」を認識できるべきです。

**終了または降格のシグナル**：
- スコア制御表が3ヶ月連続で新規エントリがない（協力が完全に安定したか、協力頻度が大幅に下がったことを示す）。
- 各「離陸」レビューで実質的な発見がない。
- 協力中にフィードフォワードチェックリストが不要になり、好みが高度に内部化されたと気づく。

これらのシグナルが現れた時、ループ頻度を下げる（例：毎週から毎月へ）、またはアーキテクチャを簡素化する（例：スコア表を削除し、フィードフォワードと自己監視を残す）ことができます。

これはシステムの自己謙虚な設計選択です：**それは永遠に存在することを追求せず、必要な時に有効に動作し、不要な時に優雅に縮小します。**

## 7.5 最後のステップ

チュートリアルはこれで終了です。第五章に従ってデプロイを完了した場合、今のあなたは3つのことだけを行えばよいです：

1. **通常使用**——通常通りタスクを送信し、通常通りフィードバックを与えます。
2. **定期レビュー**——「離陸」と言うことを覚えます。
3. **必要時調整**——問題に遭遇した時、本章を再確認します。

あなたのHermesはもう絶えず間違いを修正する必要のあるツールではなく、使えば使うほどあなたを理解し、継続的に原則に調整する協力エンジンです。これはSFではなく、20世紀のシステム科学の思想を21世紀のAI Agentアーキテクチャに実装したものです。

**この7つの章の全過程を一緒に歩めて嬉しく思います。**
