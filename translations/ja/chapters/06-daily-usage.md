# 第6章：日常使用と検証方法

デプロイ完了後、本章では2つの疑問に答えます:**このシステムを毎日どのように使用するか**、および**正常に動作しているかどうかをどう確かめるか**。

---

## 6.1 日常コラボレーションの新しいリズム

デプロイ後、あなたとAgentのコラボレーション・パターンは微妙だが重要な変化を遂げます——あなたは「修正者」から「確認者」へと切り替わります。

| 段階   | 以前の発生順序                           | 現在の発生順序                                               |
| :---   | :--------------------------------------- | :----------------------------------------------------------- |
| 起動   | あなたがタスク送付 → Agentが直接実行                  | あなたがタスク送付 → Agentが起動チェックリスト出力 → あなた確認（または「直接実行」言） → 開始 |
| 実行   | Agentが出力 → あなたが確認 → あなたが修正             | AgentがSkillに従って実行、回避すべき項目を自動的に注入                          |
| 納品   | あなたが確認 → 決まり文句やフォーマット問題に気づき → 指示 → 修正 | Agentが5つのゲートを自動的に通過 → あなたがクリーンな結果を取得                    |
| フィードバック | あなたが一言 → Agentは時々修正しすぎ時々忘れる        | あなたが一言 → Agentが積算表に記録、2回目からルールに変更                    |
| 振り返り | あなたが感覚でまとめ                             | あなたが「起飞」と言う → Agentが9ステップ振り返りを自動実行                            |

**核心的変化**：以前、あなたはフィードバック・コントローラーとしてループ内にいましたが、現在は大部分のフィードバック・ループがAgent内でフィードフォワードと自己監視として内面化されています。

---

## 6.2 典型的な使用シナリオと対話例

### シナリオA：新規タスクの開始

**あなたが言う**：
> 「秋の最後の日」についての短編を書いてください。約2000文字。

**Agentが自動実行**：追加指示不要：
1. `feedforward_startup`を呼び出し
2. 以下の起動チェックリストを出力：

```
起動チェックリスト
タスクタイプ：短編創作
シーン規則L3：感情調子をプロットより優先、結末は余白を残して昇華しない
ユーザー原則L2：冷感叙述、AI要約拒否；簡潔に形容詞を積まない
歴史的教訓：前回「画面感不足」、今回は環境描写を強化
リスク予測：
- P0：文字数が大きく逸脱
- P1：結末が無意識の要約
- P2：対話の比率が高すぎて叙述が貧弱
回避チェックリスト：冒頭で画面を作り、結末は自然に締め切ってテーマを明記しない；全編冷感、形容詞を積まない。

（確認待ちまたは「直接実行」を言うまで）
```

**あなたが言う**：
> 直接実行

**Agent**: 執筆を開始し、納品前に自動的に `delivery_gate` を通過。

> *タスクが非常に単純な場合（例：「ある一文を翻訳してください」）、フィードフォワード・チェックリストも簡素化し、長文を強制しません。いつでも「チェックリストをスキップして直接実行」と言えます。*

---

### シナリオB：ネガティブ・フィードバックの提供

**あなたが言う**：
> 今回も結末で「秋はいずれ過ぎてゆく」と要約しました。好きではありません。

**Agentが内部で自動実行**（追加指示不要）：
1. 分類：A類（L2審美原則違反）
2. 積算制御表で検索：既に「結末AI要約」項目があれば回数+1；なければ新規作成。
3. 判断：
   - 回数=1 → 今回のみ修正、返答：「記録済み、今回修正しました。」
   - 回数=2 → 自動的に沈降をトリガー、返答：「'結末AI要約'が2回出現したため、このルールをL2審美原則に沈降させます：‘いかなる形式の要約的締め切りも拒否’。同意しますか？」

> *フィードバックが具体的であるほど（例：「結末がくどい」）、Agentはそれを正確なエラー・パターンとして抽象化しやすくなります。あいまいなフィードバック（例：「書き方が悪い」）は避けてください。*

---

### シナリオC：毎日/毎週の振り返り

**あなたが言う**：
> 起飞

**Agentが自動的に `control_loop_review` を呼び出し**、9ステップのプロセスで出力し、最後に以下のような結果を見ます：

```
[閉ループ整理完了]

- フィードフォワード審査：今週 feedforward 5回、すべて正常に呼び出し
- 積算沈降：1項目が沈降済み（結末AI要約 → L2）
- 階層浄化：混在なし
- ゲートアップグレード：P1に「連続対話段落は6行を超えない」チェック項目を追加することを提案します。同意しますか？
- 根本原因分析：略...
- 再利用抽出：skill: dialogue_density_check の作成を提案
- 移行提案：今回不要（または：シャノン情報論を導入して対話密度を制御することを提案）
```

あなたが各項目を確認/否決した後、Agentは記憶に書き込み、人物像メンテナンスを呼び出します。

---

### シナリオD：システム状態の随時照会

**あなたが言う**：
> 今の記憶は何階層に分かれていますか？それぞれ何を保存していますか？

**Agent** は、L1の5項目、L2の次元、L3のシーン・リスト、積算表の項目数を正確に回答。

**あなたが言う**：
> 積算制御表には現在どのような項目がありますか？

**Agent** は、現在の表の状態を出力。

## 6.3 システム健全性チェックリスト

毎週2分かけて以下のチェックリストを確認し、システムが静かに劣化していないことを確かめます。

### 自己検査項目（観察のみ、追加指示不要）

| #    | チェック点   | 正常な表現                                | 異常信号                         |
| :--- | :------- | :-------------------------------------- | :------------------------------- |
| 1    | フィードフォワード起動 | 新規タスクのたびにAgentが起動チェックリストを出力         | Agentが直接実行を開始し、チェックリストをスキップ   |
| 2    | 納品品質 | 納品物が清潔でAIの決まり文句がなく、フォーマットが正確        | また「総じて」「特筆すべきは」などが見える |
| 3    | 積算制御 | 同じ種類の問題を2回目に言ったとき、Agentが沈降を提案    | 3回目でも同じ問題を修正する必要がある         |
| 4    | 階層分離 | memoryに操作手順がなく、skillに原則がない | あるとき記憶を開けると中が手順で一杯 |
| 5    | 閉ループ実行 | 「起飞」と言った後、Agentが完全な9ステップを実行             | 振り返り出力が適当で、根本原因分析をスキップ       |

### 高速修復コマンド

チェックポイントが異常な場合、対応するコマンドで高速修復：

- フィードフォワード失効 → `今後、各タスク前に必ず feedforward_startup を呼び出してください。`
- 積算失効 → `積算制御表を確認し、すべての ≥2 回項目が沈降済みかどうか確認してください。`
- 階層混在 → `階層浄化スキャンを1回行ってください。`
- ゲート緩和 → `delivery_gate プロセスを完全に再実行し、結果を報告してください。`

## 6.4 システム正常稼動の核心指標

以下の4つの定量指標でシステムが本当に有効かどうかを判断できます（たまに観察すればよく、毎日統計する必要はありません）。

### 指標1：再修正率

旧パターン：同じ種類の問題を平均3-5回修正するまで繰り返します。
新パターン（積算制御有効後）：同じ種類の問題が2回目に出現した時点で沈降し、3回目からは出現しなくなります。

**判定**：沈降済みの問題が3回目に出現した場合、沈降の実行が失敗したか、あるいはSkillが更新されていないことを示し、閉ループ調査を開始する必要があります。

### 指標2：フィードフォワード・チェックリスト命中率

Agentが出力した起動チェックリストのうち、P1リスク予測があなたによって「そうです、まさにそれです」と確認される頻度。

**判定**：フィードフォワード・チェックリストのP1予測の少なくとも50%以上があなたが実際に気にする点に属している場合、フィードフォワード・システムは良好に稼動しています。5回連続タスクでP1がすべて「違う」と言われた場合、L2/L3の更新が必要か、あるいはAgentがあなたを理解するのに偏差があることを示します。

### 指標3：納品前ブロック回数

delivery_gate のゲート報告で、AI痕跡、構造欠損がブロックされる頻度を観察します。

**判定**：初期はブロック回数が多い（Agentが自己監視を通じて学習・修正中）。継続稼動後、P1類ブロック回数はゼロに漸近すべき——これはフィードフォワードと積算制御が源頭からこれらの問題を排除したことを示し、ゲート失効ではありません。

### 指標4：積算表沈降転化率

積算表中の「沈降済み」項目数 / 「累積≥2回」項目総数の比率。

**判定**：比率は長期的に100%を維持すべき。≥2回の項目が長期間沈降しない場合、積算制御の自動沈降メカニズムがトリガーされていないことを示し、調査が必要です。

## 6.5 しきい値と設定の調整

システムはあなたのコラボレーション・スタイルに合わせてパラメータ調整を許可しています。

### 6.5.1 積算しきい値の調整

しきい値デフォルト = 2。あなたが：

- 安定性を非常に重視し、極めて稀なフィードバックがシステムを乱さないことを望む → 3に調整
- コラボレーションテンポが速く、Agentがあなたの好みをより早く学ぶことを望む → 2のまままたは1に調整（1に調整することは推奨しない、ノイズ耐性が失われる）

**調整コマンド**：
```
積算制御表のトリガーしきい値を2から3に変更してください。これから、同じ種類のフィードバックが≥3回出現したときに沈降します。
```

### 6.5.2 ゲートチェック項目の調整

あなたのニーズの変化に伴い、ゲートの5つは増減できます。

**例**：頻繁に純粋なTXTでヘッダーなしの出力を求めるようになり、一つ追加したい場合：
```
delivery_gate の第5ゲートの後に、第6ゲート [P2] 出力フォーマットを追加：純粋なTXTでありヘッダー情報がないかチェック。不合格なら浄化。
```

### 6.5.3 新規シーンのL3規則

新しいタスクタイプを開始するとき、Agentにはまだ対応するL3がありません。

**方法**：そのタイプのタスクを数回完了した後、「起飞」振り返りの際、Agentは自動的にL3規則を追加することを提案します。また、直接言うこともできます：
```
これは新しいタスクタイプです：[名称]。今回のタスク完了後、私のためにL3の核心規則を1つ抽出し、記憶に保存してください。
```

## 6.6 初週の稼動提案

デプロイ後の初週、以下のリズムで稼動させ、システムを十分に適合させることを推奨します。

| 日        | アクション                                                         | 目的           |
| :-------- | :----------------------------------------------------------- | :------------- |
| 1日目   | デプロイ完了（第5章）、当日のすべてのタスクを通常通り実行。フィードフォワード・チェックリストが出現するか観察。 | コンポーネント稼動確認   |
| 2日目   | 通常タスク。意図的に同じ種類のネガティブ・フィードバックを2回与え、沈降がトリガーされるか観察。 | 積算制御検証   |
| 3日目   | 通常タスク。「起飞」と言い、本格的な振り返りを初回実行。               | 閉ループ・プロセス検証   |
| 4-5日目 | 通常使用、追加テストなし。                                     | システムにデータを蓄積 |
| 6日目   | 積算制御表の状態を一度確認し、すべてのしきい値到達項目が沈降済みであることを確認。           | 積算表のメンテナンス     |
| 7日目   | 完全な「起飞」振り返りを1回実行。人物像 v1 が v2 に更新が必要かチェック。      | 週間閉ループ形成     |

一週間後、このシステムがあなたのコラボレーション・スタイルに適しているか判断できます。もし何らかの部分があまりに複雑に感じる場合（例：毎回チェックリストを確認する）、「直接実行し、私が「まずチェックリスト」と言ったときのみフィードフォワードを出力」に調整できます。

## 6.7 本章のまとめ

日常使用において、このシステムを利用する秘訣はただ3つ：

1. **通常タスクは通常通り送付し、追加操作不要**——フィードフォワードとゲートは自動的に稼動します。
2. **フィードバックがあるときは具体的に言う**——あなたが言うすべての一言の批判は積算表によって捕捉されます。
3. **定期的に「起飞」と言う**——これこそシステムが自己進化する唯一のボタンです。

デプロイは終点ではなく、システムが自己進化を始める起点です。一週間ごと、振り返りのたびに、それは前週よりもさらにあなたに「似て」ゆきます。
