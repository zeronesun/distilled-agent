# 第四章：システムアーキテクチャ詳解――三層記憶体 + スキル層

前三章ではそれぞれ「なぜ」と「何か」の問題を解決しました。本章ではそれらを完全な、実際に実行可能な**システムアーキテクチャ**に組み立て、4つのサイバネティックス概念をAgentの記憶とスキル体系の中で明確に配置し、データフローの方向性を定めます。

---

## 4.1 アーキテクチャ全体像

本アーキテクチャの核心原則は：**論理的階層化で認知境界を定義し、物理的階層化で記憶と実行を分離する**ことです。

```
┌────────────────────────────────────────────┐
│          あなたのAgent（操作されるシステム）    │
├────────────┬───────────────────────────────┤
│  論理層     │  物理記憶                       │
├────────────┼───────────────────────────────┤
│  L1 メタ認知│  記憶トップエリア（自律修正不可） │
│  L2 ユーザー論理│  記憶主体エリア              │
│  L3 シーン規律│  記憶主体エリア              │
│  積分制御表 │  記憶主体エリア              │
├────────────┼───────────────────────────────┤
│  実行層     │  スキルモジュール               │
│  フィードフォワード清单│   セッション内コンテキスト（一時）│
│  歴史修正記録│ セッションアーカイブ（長期読み取り専用）│
└────────────┴───────────────────────────────┘
         ↑ ↓ 制御フローとデータフロー
    ┌─────┴──────┐
    │  あなたのタスク │ ← 被制御システム
    └────────────┘
```

各層には厳格な情報准入基準があり、以下で順番に展開します。

---

## 4.2 L1：メタ認知フレームワーク――システム憲法

### 4.2.1 位置づけ

L1はAgentの「思考憲法」であり、**情報をどう処理するか、自らの行動をどう評価するか、意思決定をどう行うか**を定義します。特定のタスクやユーザーへの個別対応ではなく、Agent自体の動作方式に対する強制的な制約です。

### 4.2.2 物理記憶

- **場所**：`memory`のトップエリア（例：Hermesの`memory(target='memory')`の最前部）。
- **属性**：**自律修正不可**。Agentはクローズドループ整理の中で修正を*提案*できますが、ユーザー確認を経る必要があります。これはシステム骨構がAgent自身の進化プロセスにより偶発的に捻じ曲げられるのを防ぐためです。

### 4.2.3 コンテンツ構造

L1には4つのサイバネティックス概念の具体的な行動指針を保存し、抽象的な定義ではありません。各指針は以下の条件を満たす必要があります：
- あなたは～しなければならない、あるいは～てはならないで始まる。
- 明確なトリガー条件と実行アクションを含む。
- 一度にコンテキストに読み込み可能な長さである。

**標準エントリ例**：

```
① フィードフォワード：新規タスク起動時、まずL2、L3、セッションアーカイブを検索し、
   「起動チェックリスト」を出力しP0-P3のリスクを予測しなければならない。
② 積分制御：悪いフィードバックを受けた時、積分制御表に記録しなければならない。
   同種類が累積2回以上になったら、減格案を提出しユーザー確認を求める。
③ 階層分離：memoryに書き込む内容は「原則/規律」のみで、
   操作手順を含んではならない。操作手順はskillに書くこと。
④ 自己モニタリング：どの結果を引き渡す前にも、delivery_gateを実行しなければならない。
   5関門すべて通過したら最終結果を出力する。
⑤ 根本原因分析：クローズドループ整理の中で、各問題に対して「規律ミス」か「実行ミス」の層まで問い詰めなければならない。
```

エントリ総数は5-7件以内に抑え、Agentが一度に完全に思い出せるようにする。

## 4.3 L2：ユーザー基礎論理――あなたの「原則ライブラリ」

### 4.3.1 位置づけ

L2には異なるタスクにまたがるあなたの**安定した嗜好と核心原則**を保存します。これらの嗜好は特定のシーンに限定されず、Agentとあなたの協力時の基本的な行動方向を定義します。

L1がサイバネティックスの外殻なら、L2は**あなたのパーソナル化された内核**であり、Agentにとってあなたの判断パターンの計算可能なモデルです。

### 4.3.2 物理記憶

- **場所**：`memory`の主体エリア、L1の直後。
- **更新権限**：積分減格やクローズドループ整理時にAgentが書き込み、あなたの確認を経る。
- **容量制御**：各エントリ30字以内、総数10件以内（調整可能）まで。L2が「メモ」に膨れ上がるのを防ぎ、シーンをまたがる安定原則のみを保持するよう強制する。

### 4.3.3 コンテンツ構造と例

**組織方式**：次元別にグループ化し、検索しやすくする。

| 次元     | 保存例                               | 説明                                  |
| :------- | :----------------------------------- | :------------------------------------ |
| コミュニケーションスタイル | ユーザー嗜好：简洁直接、礼儀表現不要   | 「お役に立てれば幸です」を出力しない   |
| 出力標準 | ユーザー要求：コード優先、検証方法付き | 方案を与えるだけでなく「どう検証するか」も伝える |
| 意思決定モード | ユーザー意思決定：リスク先行、完全なリスクを先に暴露 | まずリストしてから利益を話す       |
| 美的指向 | ユーザー美学：冷感叙事、AI式总结拒否 | 小説の結末は余白を残し、価値的な总结をしない |

**重要な制約**：
- 具体的シーンを含まない：「短編創作はどうあるべきか」のようなものはL3に入れるべき。
- 操作手順を含まない：「第3段階で总结を削除」のようなものはSkillに入れるべき。
- 各エントリはユーザー判断の抽象化であり、タスク結果の描写であってはならない。

### 4.3.4 他の層とのインタフェース

- **L1フィードフォワードによって呼び出される**：タスク起動時、AgentはL2から関連次元の原則を読み込み、回避 avoidance リストに組み込む。
- **積分制御によって書き込まれる**：ある美的原則が2回のフィードバックを蓄積したら、AgentはそれをL2に減格させる。
- **クローズドループ整理によって監査される**：各回の整理でL2が時代遅れか、新たな原則を追加する必要があるかを検査する。

## 4.4 L3：シーン核心規律――タスク「成功パスワード」

### 4.4.1 位置づけ

L3には特定のタスクタイプの**成功規律**を保存します。この層が答える問いは：「この種のタスクを行う時、どんな規律に従えば正しくできるか」。

L2はシーンをまたがる「公理」、L3はシーンに関連した「定理」です。一つのL3エントリはあるタスクタイプの濃縮された知恵であり、操作マニュアルではありません。

### 4.4.2 物理記憶

- **場所**：`memory`の主体エリア、L2の下側、タスクタイプ別にノード分け。
- **更新権限**：L2と同じく、積分減格やクローズドループ整理時に書き込み、あなたの確認を経る。
- **容量制御**：各タスクタイプ1-3件の規律、各エントリは一文で「このシーン下の最核心的な矛盾あるいは最も重要な原則」を言い表す。

### 4.4.3 コンテンツ構造と例

**組織方式**：タスクタイプ別にノード分け、各ノード下に1-3件。

```
[L3·シーン核心規律]

- 短編創作：
  • 感情定調が情節の展開よりも優先
  • 結末は余白を残し、価値的な总结をしない
- コードレビュー：
  • まず構造的合理性と境界条件を検査し、その後に命名規範を検査
  • 各種類の問題に対して修正提案を与え、エラーを指摘するだけにしない
- 問題トラブルシューティング：
  • まずログの根本原因を位置づけ、経験で推測しない
  • 不確定な点は「検証待ち」と表記し、結論を下さない
- アーキテクチャ設計：
  • まずシステム境界とデータフローを定義し、その後に技術スタックを決定
  • 拡張性と失敗モードを評価し、ハッピーパスだけを書かない
```

**重要な制約**：
- 「規律」はSkill設計を駆動できるものでなければならない：一つのL3エントリがSkillにチェック項目を書かせることができないなら、まだ十分に抽象化されていないか、本質に到達していないことを示す。
- ファイル名、ツール名など期限付きの情報を含めてはならない。
- 「第3段階で何をすべきか」などの操作手順を含めてはならない。

### 4.4.4 スキル層との関係

L3はスキル層の**設計仕様**です。あるSkillは対応するL3規律から導出された実行パイプラインでなければなりません。クローズドループ整理で実行結果に問題が見つかったら、まずSkillステップ（実行層）を検査し、Skillに問題がなければ、L3規律の抽出ミスか（戦略層）を検証します。

## 4.5 積分制御表――システム「フィードバック閾値調整器」

### 4.5.1 位置づけ

積分制御表はサイバネティックス第二概念（積分制御）の物理的担い手です。軽量な構造化されたログであり、**繰り返し現れるエラーパターンとその処理状態**を記録します。

### 4.5.2 物理記憶

- **場所**：`memory`の主体エリア、独立したノード。
- **操作**：Agentが悪いフィードバックを受けて自動的に保守し、あなたが手動で管理する必要はありません。

### 4.5.3 データ構造

| 項目     | 説明                                              | 例               |
| :------- | :------------------------------------------------ | :----------------- |
| 類別     | A類（L2違反）/ B類（L3違反）/ C類（Skillステップミス） | B類                |
| エラーパターン | 8字以内の抽象化表現                                | 「短編結末AI总结」 |
| 初回日付 | 初回出現日付                                    | 2026-05-10         |
| 累積回数 | 現在の累積回数                                      | 2/2                |
| 閾値     | デフォルトは2                                          | 2                  |
| 減格済み   | 既にL2/L3/Skillに書き込まれたか                            | はい                 |
| 減格目標   | どこに減格したか                                    | L3·短編創作        |

### 4.5.4 実行ロジック

ユーザーが悪いフィードバックを与えた時 → Agentがエラーパターンに抽象化 → 表で同種パターンを検索：
- 存在し閾値に達している場合：ユーザーに提示し、このパターンは既に減格されていると伝える。
- 存在し閾値に達していない場合：累積数+1。閾値に達したら、自動的に減格し確認を求める。
- 存在しない場合：新規エントリを作成し、回数を1に設定。

各回のクローズドループ整理時、Agentは本表の完全な状態を出力しなければならない。

## 4.6 実行層：スキル（Skill）――操作パイプライン

### 4.6.1 位置づけ

スキル層はシステムの**手と足**であり、すべての具体的な操作を担当します。三種のスキルを含みます：

| タイプ     | 例                                                         | 対応サイバネティックス概念     |
| :------- | :----------------------------------------------------------- | :----------------- |
| メタスキル   | `feedforward_startup`、`delivery_gate`、`control_loop_review` | フィードフォワード、自己モニタリング、クローズドループ |
| タスクスキル | `short_story_writer`、`code_reviewer` 等                     | L3駆動         |
| 保守スキル | `user_profile_maintainer`                                    | プロフィール保守           |

### 4.6.2 スキル設計仕様

各スキルは以下のフィールドを含まなければなりません（`short_story_writer`を例に）：

```
- トリガー条件：ユーザーが短編小説の執筆を求めた時
- 依存L3：[L3·短編創作] 感情定調優先、結末余白
- フィードフォワード注入：L2美的指向 + L3対応規律をロード
- 実行ステップ：
  1. テーマと字数を確認
  2. 感情基調を定める（冷感/暖感/サスペンス）
  3. 執筆（最初の三文で画面感を作る）
  4. ……
- 自己検査ステップ：delivery_gateの五関門スキャンを呼び出す
```

どのスキルもステップ中に「ユーザー嗜好」をハードコードしてはなりません。嗜好はフィードフォワード注入により取り込まれなければなりません。これによりL2のある美的原則を修正した時、すべてのスキルが自動的に適用され、個別に修正する必要がなくなります。

### 4.6.3 スキルバージョン管理

各スキル名はバージョン接尾辞を持ち（例：`_v1`）、更新時は新しいバージョンを作成し、旧版は照合のために保存します。バージョン管理によりいつでもロールバックでき、クローズドループ整理時の修正対比にも便利です。

## 4.7 データフロー全体：一つのタスクが各層をいかに駆け抜けるか

「短編小説を書く」を例に、完全なデータフローを展示します：

```
Task: 短編を書く
  │
  ├─[1. フィードフォワード起動] Skill: feedforward_startup
  │    ├─ L1読み：フィードフォワードルールを確認
  │    ├─ L2読み：「冷感叙事、AI总结拒否」を抽出
  │    ├─ L3読み：「短編：感情定調優先、結末余白」を抽出
  │    ├─ アーカイブ検索：前回の修正「画面感不足」
  │    └─ 「起動チェックリスト」を出力 → 確認待ち
  │
  ├─[2. タスク実行] Skill: short_story_writer
  │    ├─ フィードフォワードリスト中の回避項目を注入
  │    ├─ L3規律に従いコンテンツを組織化
  │    └─ 初稿完成
  │
  ├─[3. 引渡し自己検査] Skill: delivery_gate
  │    ├─ P0：硬性要求検査
  │    ├─ P1：AI痕跡 + 構造完全性
  │    ├─ P2：引渡し洁净性
  │    └─ 全部通過 → 最終稿出力
  │
  └─[4. ユーザーフィードバック→積分制御]
       ├─ 悪いフィードバック → パターンに抽象化
       ├─ 積分制御表に書き込み
       ├─ 累積≥2なら → L2/L3/Skillに減格
       └─ 次回タスクのフィードフォワード呼び出しを待つ
```

**クローズドループ整理（定期的に実行）**：
1. トリガー（ユーザーが「離陸」と言う）。
2. Agentが`control_loop_review`を呼び出す。
3. L1の四基準 + 根本原因分析標準プロセスにより、すべてのL2、L3、Skill、積分表を監査。
4. 整改案を提出。
5. `user_profile_maintainer`を呼び出しプロフィールを更新。

## 4.8 アーキテクチャの自己保護メカニズム

システム自体の「逸脱」や「肥大化」を防ぐため、三つの自己保護規則を内蔵しています。

1. **L1は自律修正不可**：AgentがL1を変更したい時は提案し、確認を待つ必要がある。
2. **L2/L3エントリ上限**：各クラス10件/3件上限を守り、超過したら時代遅れ項目の整理をユーザーに促す。
3. **Skillバージョン管理と廃止**：旧版スキルは無用と確認したらアーカイブまたは削除し、蓄積を防ぐ。

## 4.9 章結び

本章では、**サイバネティックス4概念を論理層として、記憶とスキルを物理層として、データフローを実行神経として構成する**完全なシステムアーキテクチャを展示しました。本質的には一つの「設計パターン」であり、Agentの記憶とスキルシステムの再計画であり、新たな機能を追加するものではありません。

**次章では、このアーキテクチャを実行可能なデプロイ命令に翻訳し、順を追ってあなたのAgent上に完全なサイバネティックスエンジンを構築していきます。**
