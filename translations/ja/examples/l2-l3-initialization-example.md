# L2/L3 初期化出力の例

このファイルでは、L2/L3の典型的な初期化対話を示します。ユーザーが洗練指示を送った後、Agentはユーザーに対する過去の理解に基づき、構造化されたL2とL3のドラフトを出力します。

> **注釈**: この例でのユーザー設定は架空のものであり、出力形式と洗練の粒度を示すためのものです。あなたのAgentは、あなたとの過去の対話に基づき、あなた自身のバージョンを生成します。

---

## Agentの洗練出力

私の現在の理解に基づき、以下のとおり洗練結果を提示します。

### [L2・ユーザー基礎論理]

- **コミュニケーションの好み**: 簡潔かつ直接的、形式的な前置き不要；境界線を明確に、「いえ」で即断
- **品質基準**: コードは検証方法を添付を優先；出力は再利用可能、1回の正確性では不十分
- **意思決定モード**: リスク先行、まず完全な問題を提示してから解決策；直観より二次元評価
- **学習モード**: 分野を超えた理論移転；知行合一、理解を必ず操作として実装

### [L3・シーン核心法則]

- **短編の創作**:
  - 情緒の定調をプロットの展開に優先
  - 結末は余白を残し、価値のまとめや道徳昇華をしない
  - 白描で映像感を構築し、比喩描写に頼らない

- **コード出力**:
  - まずテストや検証方法を書き、その後に実装
  - 機能の完全性をコメントの詳細に優先
  - 不確実な動作には「検証済」と付記し、適当に作り込まない

- **アーキテクチャ設計**:
  - まずシステム境界とデータフローを定義し、その後に技術スタックを選択
  - 拡張性と失敗モードを評価し、成功ルートだけ書かない
  - 出力にはリスク等級（P0-P3）と対応策を含める

- **問題のトラブルシューティング**:
  - まずログ/実行現場の根本原因を特定し、経験からの推測を行わない
  - 不確実な箇所には「検証済」と付記
  - 再利用可能なトラブルシューティング手順を提示し、結論だけ与えない

## ユーザーが行える審査アクション

1. **各項目の確認**: 各原則について自問する——「これは本当に私の安定した好みか？」
2. **修正**: 間違っている項目をAgentに伝え、どのようにすべきか指示
3. **欠落の追加**: ある次元/シーンで重要な原則が欠けていると感じれば、直接指摘
4. **不実なものの削除**: Agentが適当に作り込んだもの（あなたはその好みを持っていない）は、直接削除

## よくある修正の例

| Agentの原始的な洗練              | ユーザーの修正                                    | 説明                                           |
| :-------------------------------- | :------------------------------------------------ | :--------------------------------------------- |
| コミュニケーションの好み：簡潔かつ直接的 | コミュニケーションの好み：簡潔だが十分な文脈が必要 | ユーザーは文脈が必要であり、短さだけではないことが判明 |
| 短編：結末は余白を残す           | 短編：結末は余白を残すが、シリーズ物は温和な結末を許容 | ユーザーは例外条件を追加                       |
| コード出力：テスト優先           | コード出力：手軽なプロトタイプではテストを後回し可 | ユーザーはプロトタイプと正式な出力の異なる基準を区分 |
