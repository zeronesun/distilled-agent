# 第七章：常見問題與延伸思考

這是教程的最後一章，覆蓋兩個板塊：部署和運行中可能遇到的典型障礙及其解決方法，以及將這套框架擴展到更複雜場景的思路。

---

## 7.1 部署相關問題

### Q1：我寫入 L1 後，Agent 沒有按五條鐵律執行，怎麼辦？

**常見原因**：
- 記憶未置頂，被後續寫入的內容覆蓋或沖淡
- 上下文窗口因長對話被擠佔，L1 被截斷
- Agent 平台對記憶注入時機有延遲

**排查步驟**：
1. 讓 Agent 複述 L1 五條鐵律。若無法完整複述，說明記憶寫入或讀取有問題。
2. 若複述正確但仍不執行，在新會話開始時重申一句：`本次會話開始前，先回顧 memory 中 L1 元認知框架的五條鐵律。`
3. 若問題持續，考慮將 L1 的核心規則精簡為一兩句後寫入平台的「系統指令」或「system prompt」區域（如果支持），雙保險。

---

### Q2：Agent 說它沒有 `skill_manage` 功能或類似的技能管理工具

**兩種方案**：

**方案 A（推薦）**：用記憶模擬技能。
- 在 memory 中創建標記為 `[Skill-技能名]` 的記憶條目，內容就是技能的操作步驟。
- 在 L1 中加入規則：`每次需要執行某技能時，從 memory 中檢索對應的 [Skill-*] 條目並嚴格按步驟執行。`
- 缺點：佔用記憶空間；優點：邏輯完整。

**方案 B**：簡化架構，只用 L1 + L2 + L3 + 積分表。
- 放棄技能層的自動化，改為你手動在任務開始時提醒 Agent 遵循自檢步驟。
- 適用於輕度使用或短期協作。

---

### Q3：初始化 L2/L3 時，Agent 對我的提煉完全不準確

這並不意味著架構有問題，而恰恰說明 Agent 之前對你的理解有偏差——這正是本教程要解決的問題。

**處理方法**：
- 不要全盤接受 Agent 的提煉。逐條修正，把它當成一次「校準對話」。
- 修正後明確告訴 Agent：`你之前對我的理解有偏差，我已經修正了 L2/L3 的內容。請在記憶中以我的修正為准。`
- 後續每次閉環梳理中，Agent 會自動更新理解。通常經過 2-3 次復盤後，L2/L3 的準確度會顯著提升。

---

### Q4：積分控制表裡條目越來越多，怎麼辦？

這是系統正常運行的信號——說明你確實在持續協作，Agent 在持續學習。

**維護策略**：
1. 每次「起飛」復盤時，檢查積分表。
2. 對已經下沉且運行穩定的條目，可歸檔為「已固化」，減少活躍條目數。
3. 對歸類錯誤或不再適用的條目，手動刪除。

```
將積分控制表中「結尾AI總結」（已下沉，6個月內無復發）移到歸檔區，不再計入活躍條目。
```

---

### Q5：部署後發現前饋清單太冗長，影響效率

前饋清單的長度應與任務複雜度成正比。如果簡單任務也輸出長篇清單，調整方法：

**精簡指令**：
```
修改 feedforward_startup：對於簡單任務（如翻譯、簡短問答），前饋清單壓縮為一行——「避坑：xxx」。只有複雜任務才輸出完整啟動清單。是否簡單任務由你在執行時判斷並告知我。
```

你也可以隨時在任務開始時說「跳過清單直接做」，Agent 應按此執行。

## 7.2 運行中常見問題

### Q6：Agent 連續兩次我沒糾正的同類錯誤，但沒有觸發下沉

**直接排查**：
```
請輸出積分控制表當前完整狀態。
```

**可能原因**：
- Agent 對你的第一次反饋未正確歸類（如歸為 D 類偶發）。
- Agent 將兩次反饋歸為不同錯誤模式，而非同一條目。

**糾正方法**：
```
把[描述兩次反饋]統一歸為同一個錯誤模式：「[8字以內的模式名]」。請更新積分表並檢查是否達到下沉閾值。
```

---

### Q7：Agent 在交付閘門報告中總是「全部通過」，但我發現了問題

這說明閘門的檢查項不夠細，或者 Agent 執行閘門時敷衍了事。

**加強方法**：
1. 在發現問題的當下指出來：`delivery_gate 第三關沒有發現「值得注意的是」這句套話。請解釋原因，並更新閘門的檢測標準。`
2. 如果特定問題重複出現，在閘門中增加針對性的檢查規則：
```
在 delivery_gate 第三關的檢查項中增加：「是否包含『在某種程度上』、『可以說』等模糊限定詞？發現即刪除。」
```

---

### Q8：閉環梳理的輸出太模板化，缺乏真正有價值的根因分析

這可能是因為 Agent 在一個較長階段內沒有遇到真正的問題，也可能是流程設置導致 Agent 傾向於「填表」而非「思考」。

**激活方法**：
- 在「起飛」時追加具體指向：`本次復盤重點放在[某個具體問題]上，請深入根因分析。`
- 如果長期沒有問題，告訴 Agent：`本次閉環如果沒有發現任何需修正的問題，可以縮短為3句話簡報，不需要完整九步。`

---

### Q9：我擔心 Agent 的記憶越寫越多，最終超限

這是架構設計中已經考慮的問題。三個內置機制控制膨脹：

1. **層級分離本身即抗膨脹設計**：原則和規律的體積遠小於操作步驟和案例。
2. **L2/L3 條目上限**：L2 上限 10 條，L3 每個場景上限 3 條。超過時提示清理。
3. **閉環梳理中的淨化步驟**：每次復盤都檢查並遷移混淆內容。

如果你仍然擔心，可以定期手動清理：
```
列出 memory 中最近 30 天未被訪問過的條目，評估是否可以刪除。
```

## 7.3 延伸思考

### 7.3.1 從單 Agent 到多 Agent 協作

本架構的核心——控制論四概念 + 層級分離——天然適用於多 Agent 場景。

**擴展思路**：
- **主 Agent**（編排者）：持有 L1 元認知框架，負責任務分發和質量把控。
- **子 Agent**（執行者）：各自持有特定場景的 L3 規律和對應 Skill。
- **共享記憶區**：L2 用戶底層邏輯對所有 Agent 可見，確保不同角色都遵循統一的審美和原則。
- **統一積分表**：所有 Agent 的負面反饋匯集到同一積分表，由主 Agent 執行下沉決策。

這本質上是把「層級分離」中的「策略/執行」邊界，映射為「主 Agent / 子 Agent」的物理邊界。

---

### 7.3.2 引入其他理論框架

教程全程使用錢學森工程控制論作為底層框架。但 L1 不是封閉的——你的學習遷移能力可以通過「遷移提議」步驟持續為系統引入新理論。

已在評論區出現過的延伸嘗試：

| 引入理論              | 應用方向         | 可能對應 L1 的擴展                         |
| :-------------------- | :--------------- | :----------------------------------------- |
| 積極的投資管理        | 決策與資源分配   | 在 L1 增加「動態再平衡」原則                 |
| 貝葉斯推理            | 不確定性下的判斷 | 在積分控制中加入「先驗概率」修正反饋權重     |
| 凱利準則 + 香農信息論 | 信息價值評估     | 在自監控中加入「信息密度」檢查               |
| TRIZ 發明原理         | 矛盾解決         | 在根因分析中加入「物理矛盾 vs 技術矛盾」判定 |

引入新理論的方法：在「起飛」復盤時，當 Agent 提出遷移建議，或在你有明確想法時，主動說：
```
我想在 L1 中引入[理論名]，原則是[一句話描述]。請設計如何在現有的四概念框架下增加第⑥條鐵律，並寫明它如何影響 Skill 運行。
```

### 7.3.3 框架的適用邊界再思考

評論區中出現過一個值得認真對待的質疑：**「工程控制論對現在的 Agent 來說未必是必需，或者這只是在上下文裡給 LLM 加步驟，不算強約束。」**

經過本教程的完整展開，對這個質疑可以給出以下回答：

1. **「給 LLM 加步驟」本身就是有效的約束**。LLM 是 promptable 系統，結構化的上下文約束確實能改變其行為分佈。效果不來自硬編碼的強制性，而來自持續的結構化引導。

2. **架構的價值不在單次執行，而在長期穩定性**。單次來看，你不給 Agent 任何框架，只靠隨口指令也能拿到合格結果。但連續 50 次任務後，有架構的系統收斂得更好，漂移更小。這是一種保險，不是一種燃料。

3. **不是所有場景都需要**。輕度的、一次性的、高度隨機的對話不需要這套架構。它的適用邊界是：**長期協作 × 高精度需求 × 可歸納規律的任務類型**。如果你恰好處於這個區域內，那麼投入的搭建成本會在減少重複糾錯的收益中收回。

---

### 7.3.4 框架的自我終止條件

一個健康的系統應能識別「我什麼時候該被簡化」。

**終止或降階的信號**：
- 積分控制表連續 3 個月無新增條目（說明協作已完全穩定，或協作頻率大幅下降）。
- 每次「起飛」復盤都無實質發現。
- 你發現自己在協作中不再需要前饋清單，偏好已高度內化。

當這些信號出現時，可以降低閉環頻率（如從每周改為每月），或簡化架構（如去掉積分表，保留前饋和自監控）。

這是一個系統自我謙遜的設計選擇：**它追求的不是永遠存在，而是在被需要時有效運行，在不被需要時優雅縮減。**

教程到此結束。如果你已按第五章完成部署，現在的你只需要做三件事：

1. **正常用**——正常發任務、正常給反饋。
2. **定期復盤**——記得說「起飛」。
3. **必要時調整**——遇到問題，回看本章。
