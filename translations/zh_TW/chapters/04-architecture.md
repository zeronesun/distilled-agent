# 第四章：系統架構詳解——三層記憶體 + 技能層

前三章分別解決了「為什麼」和「是什麼」的問題。本章將它們組裝成一個完整的、可實際運行的**系統架構**，讓四個控制論概念在 Agent 的記憶和技能體系中有明確的落位和數據流向。

---

## 4.1 架構全景

本架構的核心原則是：**用邏輯分層定義認知邊界，用物理分層實現存儲與執行分離**。

```
┌────────────────────────────────────────────┐
│          你的 Agent（被操縱系統）             │
├────────────┬───────────────────────────────┤
│  邏輯層     │  物理存儲                      │
├────────────┼───────────────────────────────┤
│  L1 元認知  │  記憶置頂區（不可自主修改）      │
│  L2 用戶邏輯│  記憶主體區                     │
│  L3 場景規律│  記憶主體區                     │
│  積分控制表 │  記憶主體區                     │
├────────────┼───────────────────────────────┤
│  執行層     │  技能模組                       │
│  前饋清單   │  會話內上下文（暫時）            │
│  歷史糾正記錄│ 會話存檔（長期唯讀）            │
└────────────┴───────────────────────────────┘
         ↑ ↓ 控制流與數據流
    ┌─────┴──────┐
    │  你的任務   │ ← 被控制系統
    └────────────┘
```

每一層都有嚴格的資訊准入標準，下面逐一展開。

---

## 4.2 L1：元認知框架——系統憲法

### 4.2.1 定位

L1 是 Agent 的「思維憲法」，定義了它**如何處理資訊、如何評估自身行為、如何做決策**。它不針對任何具體任務或使用者偏好，而是對 Agent 本身的運行方式做出強制約束。

### 4.2.2 物理存儲

- **位置**：`memory` 的置頂區域（如 Hermes 的 `memory(target='memory')` 最前端）。
- **屬性**：**不可自主修改**。Agent 可以在閉環梳理中*建議*修改，但必須經使用者確認。這是為了確保系統骨架不會被 Agent 自身的演化過程意外扭曲。

### 4.2.3 內容結構

L1 存儲的是四個控制論概念的具體行為準則，而非抽象定義。每條準則必須滿足：
- 以「你必須」或「你不得」開頭。
- 包含明確觸發條件和執行動作。
- 短到可以一次性載入上下文。

**標準條目範例**：

```
① 前饋：每次新任務啟動時，你必須先檢索 L2、L3 和會話存檔，
   輸出「啟動清單」並預判 P0-P3 風險。
② 積分控制：收到負面回饋時，你必須將其計入積分控制表。
   同類累計 ≥2 次時，提出下沉方案並向使用者確認。
③ 層級分離：你寫入 memory 的內容，只能包含「原則、規律」，
   不得包含操作步驟。操作步驟必須寫入 skill。
④ 自監控：交付任何結果前，你必須執行 delivery_gate，
   五關全過才輸出最終結果。
⑤ 根因分析：在閉環梳理中，對每個問題必須追問到「規律錯」還是「執行錯」層。
```

條目總數建議控制在 5-7 條內，保證 Agent 能一次性完整回憶。

---

## 4.3 L2：使用者底層邏輯——你的「原則庫」

### 4.3.1 定位

L2 存儲你跨越不同任務的**穩定偏好和核心原則**。這些偏好不侷限於特定場景，而是定義了 Agent 與你協作時的基本行為方向。

如果說 L1 是控制論外殼，L2 就是**你的個人化核心**，是 Agent 對你判斷模式的可計算模型。

### 4.3.2 物理存儲

- **位置**：`memory` 主體區，緊接 L1 之後。
- **更新權限**：由 Agent 在積分下沉或閉環梳理時寫入，須經你確認。
- **容量控制**：每條不超過 30 字，總數不超過 10 條（可調整）。避免 L2 膨脹成「備忘錄」，強制只保留跨場景的穩定原則。

### 4.3.3 內容結構與範例

**組織方式**：按維度分組，便於檢索。

| 維度     | 存儲範例                           | 說明                     |
| :------- | :--------------------------------- | :----------------------- |
| 溝通風格 | 使用者偏好：簡潔直接，不用客套話     | 不輸出「希望能夠幫到您」   |
| 輸出標準 | 使用者需求：程式碼優先，附帶驗證方法   | 不只給方案，要給「怎麼驗」 |
| 決策模式 | 使用者決策：風險前置，先暴露完整風險 | 先列風險，再談收益       |
| 審美取向 | 使用者審美：冷感敘事，拒絕 AI 式總結 | 小說結尾留白不昇華       |

**關鍵約束**：
- 不含具體場景：如「短篇創作應該怎樣」這類應進 L3。
- 不含操作步驟：如「在第三步刪掉總結」這類應進 Skill。
- 每條必須是對使用者判斷的抽象，而非對任務結果的描述。

### 4.3.4 與其他層的介面

- **被 L1 前饋調用**：任務啟動時，Agent 從 L2 讀取相關維度的原則，融入避坑清單。
- **被積分控制寫入**：當某條審美原則累積到 2 次回饋，Agent 將其下沉到 L2。
- **被閉環梳理審計**：每次梳理檢查 L2 是否過時、是否有新原則需要加入。

---

## 4.4 L3：場景核心規律——任務「成功密碼」

### 4.4.1 定位

L3 存儲特定任務類型的 **成功規律**。它回答的是：「做這類任務時，遵循什麼規律才能做對。」

L2 是跨場景的「公理」，L3 是場景相關的「定理」。一個 L3 條目應該是一個任務類型的濃縮智慧，而非操作手冊。

### 4.4.2 物理存儲

- **位置**：`memory` 主體區，L2 下方，按任務類型分節點。
- **更新權限**：與 L2 相同，由積分下沉或閉環梳理寫入，須經你確認。
- **容量控制**：每個任務類型 1-3 條規律，每條用一句話說清「這個場景下最核心的矛盾或最關鍵的原則」。

### 4.4.3 內容結構與範例

**組織方式**：按任務類型分節點，每個節點下 1-3 條。

```
[L3·場景核心規律]

- 短篇創作：
  • 情緒定調優先於情節展鋪
  • 結尾留白，不做價值總結
- 程式碼審查：
  • 先查架構合理性和邊界條件，再查命名規範
  • 每類問題必須給修改建議，不只指出錯誤
- 問題排查：
  • 先定位日誌根因，不憑經驗猜測
  • 不確定的點標註「待驗證」，不硬下結論
- 架構設計：
  • 先定義系統邊界和數據流，再選技術堆疊
  • 評估擴展性和失敗模式，不只寫 happy path
```

**關鍵約束**：
- 「規律」必須能驅動 Skill 設計：如果一條 L3 無法讓 Skill 寫出檢查項，說明它還不夠抽象或不夠本質。
- 禁止出現檔名、工具名等易過期資訊。
- 禁止出現「第三步該做什麼」等操作步驟。

### 4.4.4 與技能層的關係

L3 是技能層的 **設計規範**。一個 Skill 應該是由對應的 L3 規律推導出的執行流水線。當閉環梳理發現執行結果有問題時，先檢查 Skill 步驟（執行層），若 Skill 無錯，再檢查 L3 規律是否提煉有誤（策略層）。

## 4.5 積分控制表——系統「回饋閾值調節器」

### 4.5.1 定位

積分控制表是控制論第二概念（積分控制）的物理載體。它是一個輕量的結構化日誌，記錄**重複出現的錯誤模式及其處置狀態**。

### 4.5.2 物理存儲

- **位置**：`memory` 主體區，獨立節點。
- **操作**：Agent 在收到負面回饋時自動維護，無需你手動管理。

### 4.5.3 資料結構

| 欄位     | 說明                                              | 範例               |
| :------- | :------------------------------------------------ | :----------------- |
| 類別     | A類（違反L2）/ B類（違反L3）/ C類（Skill 步驟錯） | B類                |
| 錯誤模式 | 8 字以內的抽象提煉                                | 「短篇結尾 AI 總結」 |
| 首次日期 | 第一次出現日期                                    | 2026-05-10         |
| 累計次數 | 當前累計次數                                      | 2/2                |
| 閾值     | 預設為 2                                          | 2                  |
| 已下沉   | 是否已寫入 L2/L3/Skill                            | 是                 |
| 下沉目標 | 下沉到哪個位置                                    | L3·短篇創作        |

### 4.5.4 運行邏輯

當使用者給出負面回饋 → Agent 抽象為錯誤模式 → 在表中查找同類模式：
- 存在且已達閾值：提示使用者，此模式已下沉。
- 存在且未達閾值：累計數 +1。若已達閾值，自動下沉並確認。
- 不存在：新建條目，次數置 1。

每次閉環梳理時，Agent 必須輸出本表的完整狀態。

---

## 4.6 執行層：技能（Skill）——操作流水線

### 4.6.1 定位

技能層是系統的**手和腳**，負責所有具體操作。它包括三類技能：

| 類型     | 範例                                                         | 對應控制論概念     |
| :------- | :----------------------------------------------------------- | :----------------- |
| 元技能   | `feedforward_startup`、`delivery_gate`、`control_loop_review` | 前饋、自監控、閉環 |
| 任務技能 | `short_story_writer`、`code_reviewer` 等                     | 由 L3 驅動         |
| 維護技能 | `user_profile_maintainer`                                    | 画像維護           |

### 4.6.2 技能設計規範

每個技能必須包含以下欄位（以 `short_story_writer` 為例）：

```
- 觸發條件：當使用者要求寫短篇小說時
- 依賴 L3：[L3·短篇創作] 情緒定調優先，結尾留白
- 前饋注入：載入 L2 審美取向 + L3 對應規律
- 執行步驟：
  1. 確認主題與字數
  2. 定情緒基調（冷感、暖感、懸疑）
  3. 寫作（前三句建立畫面感）
  4. ……
- 自檢步驟：調用 delivery_gate 五關掃描
```

任何技能都不得在步驟中硬編碼「使用者偏好」，偏好必須通過前饋注入。這保證了當你修改 L2 的某個審美原則時，所有技能自動生效，無需逐一修改。

### 4.6.3 技能版本化

每個技能名帶有版本後綴（如 `_v1`），更新時創建新版本，舊版保留備查。版本化讓你可以隨時回滾，也方便閉環梳理時對比修改。

## 4.7 資料流全景：一次任務如何跑遍各層

以「寫一篇短篇小說」為例，展示完整資料流：

```
Task: 寫短篇
  │
  ├─[1. 前饋啟動] Skill: feedforward_startup
  │    ├─ 讀 L1：確認前饋規則
  │    ├─ 讀 L2：提取「冷感敘事、拒絕AI總結」
  │    ├─ 讀 L3：提取「短篇：情緒定調優先，結尾留白」
  │    ├─ 檢索存檔：上次糾正「畫面感不足」
  │    └─ 輸出「啟動清單」 → 等確認
  │
  ├─[2. 任務執行] Skill: short_story_writer
  │    ├─ 注入前饋清單中的避坑項
  │    ├─ 按 L3 規律組織內容
  │    └─ 完成初稿
  │
  ├─[3. 交付自檢] Skill: delivery_gate
  │    ├─ P0：硬性要求檢查
  │    ├─ P1：AI痕跡 + 結構完整性
  │    ├─ P2：交付潔淨
  │    └─ 全部通過 → 輸出最終稿
  │
  └─[4. 使用者回饋→積分控制]
       ├─ 負面回饋 → 抽象為模式
       ├─ 寫入積分控制表
       ├─ 若累計≥2 → 下沉到 L2/L3/Skill
       └─ 等待下次任務的前饋調用
```

**閉環梳理（週期性運行）**：
1. 觸發（使用者說「起飛」）。
2. Agent 調用 `control_loop_review`。
3. 按 L1 的四標準 + 根因分析標準流程，審計所有 L2、L3、Skill、積分表。
4. 輸出整改方案。
5. 調用 `user_profile_maintainer` 更新画像。

## 4.8 架構的自保機制

為防止系統自身「跑偏」或「膨脹」，內置三條自保規則：

1. **L1 不可自修改**：Agent 想改 L1 必須提建議並等確認。
2. **L2/L3 條目上限**：每類 10 條、3 條上限，超過則提示使用者清理過時項。
3. **Skill 版本化與廢止**：舊版技能在確認無用後歸檔或刪除，防​​止堆積。

## 4.9 章節結語

本章展示了一個**以控制論四概念為邏輯層、以記憶和技能為物理層、以資料流為運行神經**的完整系統架構。它本質上是一套「設計模式」，是對Agent記憶與技能系統的重新規劃，而非增加新功能。

**下一章，我們會把這個架構翻譯成可直接執行的部署指令，按順序一步步在你的 Agent 上搭建出完整的控制論引擎。**
