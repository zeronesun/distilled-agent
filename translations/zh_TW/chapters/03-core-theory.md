# 第三章：核心理論詳解——四個控制論概念

---

本章深入拆解錢學森工程控制論中被我們提取的四個核心概念：**前饋、積分控制、層級分離、自監控**。

每個概念，我們都會完成以下四件事：
1. **原理解釋**：它在控制論中是什麼，為什麼重要。
2. **Agent 困境**：沒有這個概念時，Agent 會陷入什麼問題。
3. **落地設計**：如何用記憶與技能實現。
4. **一個可驗證的指標**：你怎麼判斷它已經在運作。

---

## 3.1 前饋（Feedforward）——把錯誤堵在發生之前

### 3.1.1 原理解釋

在控制論中，前饋是和回饋並列的兩種基本調控方式。

- **回饋**（Feedback）：測量輸出，發現偏差，再回去修正輸入。它永遠是「事後」的。
- **前饋**（Feedforward）：在干擾進入系統之前，根據預測，提前調整控制訊號。它是「事前」的。

工程技術裡最經典的前饋案例：鍋爐系統檢測到進水溫度下降，不等蒸汽壓力變化，直接加大火力。它測量的是「干擾源」，不是「輸出偏差」。

錢學森在《工程控制論》中反覆強調：**高效能系統必須同時包含前饋和回饋，而不能只依賴回饋。** 回饋負責糾正剩餘偏差，前饋負責消除主要干擾。

### 3.1.2 為什麼 AI Agent 需要前饋

沒有前饋時，Agent 的工作方式是純回饋：

```
你：寫一篇短篇。
Agent：（輸出）
你：結尾不要 AI 總結。
Agent：（修正）
下次：
你：寫一篇短篇。
Agent：（又出現 AI 總結）
你：結尾不要 AI 總結！
```

每次錯誤發生 → 你糾正 → 它修正。這種「等你罵了再改」的模式有幾個致命缺陷：
- **浪費對話輪次**：糾正就佔用了你的時間。
- **依賴你的警覺**：如果你某次忘了說，錯誤就直接交付。
- **跨 session 遺忘**：上次修正只存在於那次對話的上下文，新 session 想不起來。

前饋解決的就是這個：**不等你說「又錯了」，先把你可能在意的東西注入任務啟動階段。**

### 3.1.3 在 Agent 中的落地設計

前饋在 Agent 中實現為 **任務啟動時的「訊號預載入」**。

**具體機制**：
1. 每次開始一個新任務，Agent 主動從三個資料來源提取資訊：
   - **L2 使用者底層邏輯**：你的跨場景審美和原則（如「拒絕 AI 總結」）。
   - **L3 場景核心規律**：當前任務類型的成功規律（如「短篇要留白」）。
   - **Session 存檔/記憶中的糾正記錄**：上一次同類任務被你糾正了什麼。
2. 將上述資訊壓縮成一個 **「避坑清單」**。
3. 在正式輸出前，把避坑清單寫出來（讓你確認或直接執行）。

**範例**：當你讓 Agent 寫短篇，它會在動筆前輸出：

> 啟動清單：
> - 任務類型：短篇創作
> - 避坑項：①結尾留白，不總結、不昇華；②不出現「作為 AI」等套話；③冷感敘事，減少形容詞堆疊。
> - 上次問題：上次結尾出現了「人生就是這樣」，已規避。

然後它才開始寫作。這一層前饋，讓你從「糾錯者」變成了「確認者」。

### 3.1.4 如何驗證它已生效

觀察以下行為：
- Agent 在新任務開始時，是否主動輸出了一個「啟動清單」或「避坑清單」。
- 同類錯誤是否在第一次就顯著減少。
- 連續 5 次同類型任務中，你糾正的次數是否下降。

**前饋的目標不是零錯誤，而是讓錯誤不再重演。**

## 3.2 積分控制（Integral Control）——該改的改，不該改的別亂改

### 3.2.1 原理解釋

在控制論中，控制器通常由三項組成：

- **比例控制（P）**：偏差多大，動作就多大。
- **積分控制（I）**：偏差持續多久，動作逐步加大。
- **微分控制（D）**：偏差變化多快，提前抑制。

純比例控制的致命缺陷是：**對雜訊過於敏感**。一次偶然的偏差（比如感測器抖動），就會觸發一次不必要的大幅度修正，系統因此震盪——左一下、右一下，永遠穩定不下來。

**積分控制**正是解決這個問題的：它累積偏差，只有累積到一定量級才觸發動作。一次跳變被平均掉了；持續、穩定的偏差才會引發糾正。這就是「積分」的含義——**對誤差進行時間積分，超過閾值才出力**。

### 3.2.2 為什麼 AI Agent 需要積分控制

沒有積分控制時，Agent 對你的回饋響應方式往往是兩種極端：

- **過激響應**：你隨口說一次「這結尾不好」，它立刻把整個寫作風格全改了，甚至連你之前喜歡的部分也變掉。
- **遲鈍響應**：你連續五次說「不要加插件式總結」，它每次都應承，但下次照舊。

這兩種情況，根源都在於 **沒有設置「響應閾值」**。Agent 把你的一次表達當成永久規則，或者把你的重複表達當成每次獨立的偶發事件。

積分控制給出的方案非常簡潔：**同類回饋出現第二次，才把它作為穩定訊號，寫入長期記憶或修改技能。**

### 3.2.3 在 Agent 中的落地設計

積分控制落地為一個**帶閾值的回饋計數器**，存於記憶體，命名為「積分控制表」。

**結構**：

| 類別 | 錯誤模式（≤8字） | 首次日期 | 累計次數 | 閾值 | 已下沉 | 下沉目標    |
| :--- | :--------------- | :------- | :------- | :--- | :----- | :---------- |
| A類  | 結尾AI總結       | 0510     | 2        | 2    | 是     | L2·審美原則 |
| B類  | 節奏過快         | 0510     | 1        | 2    | 否     | L3·短篇規律 |

**運作規則**：
1. 當你給出負面回饋，Agent 將其歸類到某個錯誤模式（A類=審美原則，B類=場景規律，C類=技能步驟錯誤）。
2. 同類模式計數 +1。
3. 如果累計 ≥ 閾值（預設2次），且尚未下沉，Agent 自動執行下沉：
   - 將這個規則寫入對應的 L2、L3 或 Skill。
   - 更新「已下沉」狀態。
   - 向你確認：「『結尾AI總結』已出現2次，我已將其下沉至 L2 審美原則，是否同意？」
4. 如果累計次數 = 1，只做本次修正，不改長期規則。

**閾值設為2的意義**：
- 1 次是雜訊：可能只是你心情不好、或某次特定場景不適。
- 2 次是訊號：連續兩次同類回饋，說明這很可能是你的穩定偏好。
- 你可以根據自己的風格調成 3，但建議從 2 開始。

### 3.2.4 如何驗證它已生效

- 你故意給同一個類型回饋兩次（如兩次說「結尾囉嗦」），觀察 Agent 是否在第二次後主動提出要修改長期規則。
- 你給一個偶然回饋（只說一次「這個比喻我不喜歡」），觀察 Agent 是否沒有因此改變整體風格。

**積分控制的目標不是讓 Agent 變「固執」，而是讓它擁有穩定的偏好捕捉能力——該聽見的不會漏，該忽略的不過激。**

## 3.3 層級分離（Hierarchical Decomposition）——策略和執行的徹底分家

### 3.3.1 原理解釋

錢學森系統思想的核心之一是 **分層遞階控制**。一個複雜系統必須被分解為不同層級：上層定策略、定目標、定約束；下層做執行、做調節、做回饋。上下層之間只通過清晰的介面傳遞資訊，內部實作互不可見。

如果這種分離被打破——上層開始管執行細節，下層開始自己定策略——系統就會陷入混亂：牽一髮而動全身，改一處而全局震盪。

### 3.3.2 為什麼 AI Agent 需要層級分離

Agent 的記憶和技能，天然容易陷入以下幾種混淆：

- **記憶膨脹**：使用者的原則（「討厭 AI 結尾」）和具體的執行步驟（「寫完後搜尋『總結』並刪除」）混在一起。記憶裡既存「為什麼」又存「怎麼做」，體積暴漲，檢索困難。
- **技能僵化**：技能裡硬編碼了某次對話的偏好，但那個偏好可能是臨時的。下次使用時，技能已經過時，你卻不知道它為什麼那樣做。
- **修改成本高**：你要改變一個輸出風格，不得不去翻閱密密麻麻的記憶和技能，分不清哪個是該改的「原則」，哪個是無關的「歷史記錄」。

層級分離就是設置一條不可逾越的邊界：**原則進記憶，步驟進技能。**

### 3.3.3 在 Agent 中的落地設計

核心規則只有一條：

> **記憶體只存「為什麼」和「規律是什麼」，技能只存「怎麼做」。**

具體表現為：

| 資訊類型                     | 歸屬      | 範例                             |
| :--------------------------- | :-------- | :------------------------------- |
| 審美傾向（討厭 AI 式結尾）   | 記憶 L2   | 使用者偏好冷感敘事，拒絕總結式收束 |
| 任務規律（短篇要先定調）     | 記憶 L3   | 短篇創作核心規律：情緒定調優先   |
| 執行步驟（寫完後用閘門檢查） | 技能      | delivery_gate 五關檢查清單       |
| 某次具體的糾正記錄           | 存檔/檢索 | 2025-05-10 使用者說結尾囉嗦        |

**執行中的強制規則**：
- 每次閉環梳理時，Agent 必須主動掃描：
  - 記憶裡是否有操作步驟？有則遷移到技能。
  - 技能裡是否有原則性內容？有則遷移到記憶。
- 每次寫入新記憶時，Agent 應先判斷資訊屬於哪一類，再決定寫入位置。

### 3.3.4 如何驗證它已生效

- 你打開 Agent 的記憶，發現它清晰分區：原則、規律、場景映射——沒有混雜的步驟。
- 你打開 Agent 的技能，發現它只有「做什麼、檢查什麼」的清單，沒有「因為使用者喜歡什麼所以這樣做」的解釋。
- 當你改變一條審美原則（如「現在可以偶爾用華麗比喻了」），只需修改一條記憶，不用碰任何技能。

**層級分離的目標是把記憶從「大雜燴」變成「資料庫」，讓每一條資訊有唯一的歸屬、確定的修改路徑。**

## 3.4 自監控（Self-Monitoring）——交付之前，先過自己的閘門

### 3.4.1 原理解釋

任何控制系統，最後都必須有一個 **回饋迴路** 閉合。在工程系統裡，這意味著：系統測量自己的輸出，與期望值比較，產生誤差訊號去修正。

對 Agent 而言，這個「回饋迴路」就是你——使用者——來檢查輸出並糾正。但如果你能在交付給你之前，讓 Agent 自己先跑一遍這個迴路，就等於加了一層自動化質檢。這就是 **自監控**：系統內建一個品質檢測器，對自己的輸出進行預檢，只有通過預檢才能交付。

### 3.4.2 為什麼 AI Agent 需要自監控

Agent 最常見的交付問題：
- **遺漏硬性要求**：你要求輸出純文本，它給了帶註解的內部版本。
- **AI 痕跡過重**：「總結來說」「值得注意的是」「作為一個人工智慧」這些套話你看了就煩。
- **結構不完整**：說好的三個部分，它寫到兩個就停了。
- **交付物不乾淨**：檔案裡帶著 review notes、內部標記，你還得手動清理。

這些問題你不檢查就交付給你，你檢查到了又得再浪費一輪對話去糾正。自監控就是把這一輪對話的檢查成本，從你身上轉移到 Agent 自己身上。

### 3.4.3 在 Agent 中的落地設計

自監控落地為一個**交付前閘門技能（delivery_gate）**。Agent 在輸出任何最終結果之前，必須執行這個技能，逐項檢查。

**閘門檢查項（五關）**：

| 關口   | 優先級      | 檢查內容                                     | 不通過動作           |
| :----- | :---------- | :------------------------------------------- | :------------------- |
| 第一關 | P0 硬阻斷   | 硬性要求檢查：字數、格式、禁止項、必須包含項 | 強制修正，重新過閘   |
| 第二關 | P0 硬阻斷   | 事實準確性：程式碼邏輯、資料引用、檔名/路徑  | 不確定的標註「待驗證」 |
| 第三關 | P1 高優先級 | AI 痕跡：套話、道德總結、突然昇華            | 強制刪除             |
| 第四關 | P1 高優先級 | 結構完整性：邏輯閉環、必須步驟無遺漏         | 補齊                 |
| 第五關 | P2 優化建議 | 交付乾淨度：去除註解、review notes、多餘標記 | 淨化                 |

**執行步驟**：
1. 任務完成，準備交付。
2. Agent 內部呼叫 `delivery_gate`。
3. 逐關掃描，記錄通過/未通過。
4. 全部通過 → 輸出最終結果。
5. 任一項未通過 → 自動修正 → 重新過閘 → 直到全過才輸出。

### 3.4.4 如何驗證它已生效

- Agent 交付時，你檢查：是否還有 AI 套話？是否還有多餘註解？格式是否完全符合要求？
- Agent 在交付前，是否在 reasoning 或中間輸出中，體現出它在做檢查（如「自檢中：P0 硬阻斷通過，P1 AI 痕跡發現一處，已刪除」）。

**自監控的目標：讓你拿到的交付物已經是「乾淨」的，你只需確認，不再需要當質檢員。**

## 3.5 四概念的協作關係

四個概念不是互相獨立的零件，而是一個系統運作的四個相位：

1. **任務啟動前**：前饋注入訊號，把坑提前圈出來。
2. **任務執行中**：層級分離確保資訊流乾淨——上層給方向，下層給執行。
3. **使用者回饋時**：積分控制決定是否把回饋上升為長期規則。
4. **任務交付前**：自監控充當最後一個回饋迴路，攔截不合格輸出。

而在**週期性複盤**中，Agent 用這四條作為標準，去審計所有的記憶和技能，完成自我進化。這就構成了下一章將要展開的系統架構全景。
