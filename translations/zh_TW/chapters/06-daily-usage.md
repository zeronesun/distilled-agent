# 第六章：日常使用與驗證

部署完成後，本章回答兩個問題：**每天怎麼用這套系統**，以及**怎麼知道它在正常運行**。

---

## 6.1 日常協作的新節奏

部署後，你和 Agent 的協作模式會發生一個微妙但重要的變化——你從「糾錯者」切換為「確認者」。

| 階段 | 以前的發生順序                           | 現在的發生順序                                               |
| :--- | :--------------------------------------- | :----------------------------------------------------------- |
| 啟動 | 你發任務 → Agent 直接做                  | 你發任務 → Agent 輸出啟動清單 → 你確認（或說「直接做」）→ 開始 |
| 執行 | Agent 輸出 → 你檢查 → 你糾正             | Agent 按 Skill 執行，自動注入避坑項                          |
| 交付 | 你檢查 → 發現套話/格式問題 → 你說 → 它改 | Agent 自動過五關閘門 → 你拿到的是乾淨結果                    |
| 反饋 | 你說一句 → 它有時改過頭有時沒記住        | 你說一句 → 它記錄到積分表，第二次才改規則                    |
| 復盤 | 你憑感覺總結                             | 你說「起飛」→ Agent 自動跑九步復盤                            |

**核心變化**：以前你在迴路裡充當反饋控制器，現在大部分反饋迴路被 Agent 內化成了前饋和自監控。

## 6.2 典型使用場景與對話示例

### 場景 A：開啟新任務

**你說**：
> 幫我寫一篇關於「秋日最後一天」的短篇，2000字左右。

**Agent 自動執行**（無需額外指令）：
1. 調用 `feedforward_startup`
2. 輸出如下啟動清單：

```
啟動清單
任務類型：短篇創作
場景規律 L3：情緒定調優先於情節，結尾留白不昇華
用戶原則 L2：冷感敘事，拒絕 AI 總結；簡潔不堆形容詞
歷史教訓：上次「畫面感不足」，本次加強環境白描
風險預判：
- P0：字數偏差過遠
- P1：結尾不自覺總結
- P2：對話占比過高導致敘述單薄
避坑清單：開篇建立畫面，結尾自然收束不點題；全程冷感，不堆形容詞。

（等待確認或說「直接做」）
```

**你說**：
> 直接做

**Agent**：開始寫作，並在交付前自動過 `delivery_gate`。

> *如果任務特別簡單（如「幫我翻譯一段話」），前饋清單也應簡化，不強制長篇大論。你可以隨時說「跳過清單直接做」。*

---

### 場景 B：給出負面反饋

**你說**：
> 這次結尾又總結了一句「秋天終將過去」，我不喜歡。

**Agent 內部自動執行**（你不需要額外指令）：
1. 歸類：A類（違反 L2 審美原則）
2. 在積分控制表中查找：若已有「結尾AI總結」條目，次數+1；若無，新建。
3. 判斷：
   - 次數=1 → 僅修正本次，回覆：「已記錄，本次已修正。」
   - 次數=2 → 自動觸發下沉，回覆：「『結尾AI總結』已出現2次，我將此規則下沉至 L2 審美原則：『拒絕任何形式的總結式收束』。是否同意？」

> *你的反饋越具體（如「結尾囉嗦」），Agent 越容易抽象為精確的錯誤模式。避免模糊反饋如「寫得不好」。*

---

### 場景 C：每日/每周復盤

**你說**：
> 起飛

**Agent 自動調用** `control_loop_review`，按九步流程輸出，最後你會看到類似：

```
[閉環梳理完成]

- 前饋審查：本周 feedforward 5次，均正常調用
- 積分下沉：1項已下沉（結尾AI總結 → L2）
- 層級淨化：無混淆
- 閘門升級：建議在P1中增加「連續對話段落不超過6行」檢查項，是否同意？
- 根因分析：略...
- 可復用提煉：建議創建 skill: dialogue_density_check
- 遷移提議：本輪無需（或：建議引入香農信息論控制對話密度）
```

你逐條確認/否決後，Agent 寫入記憶並調用畫像維護。

---

### 場景 D：系統狀態隨時查詢

**你說**：
> 你現在記憶分幾層？分別存什麼？

**Agent** 應準確回答 L1 五條、L2 維度、L3 場景列表、積分表條目數。

**你說**：
> 積分控制現在有哪裡些條目？

**Agent** 輸出當前表狀態。

## 6.3 系統健康檢查清單

每週花 2 分鐘過一遍以下清單，確保系統沒有悄悄退化。

### 自檢項（你只需觀察，無需額外指令）

| #    | 檢查點   | 正常表現                                | 異常信號                         |
| :--- | :------- | :-------------------------------------- | :------------------------------- |
| 1    | 前饋啟動 | 每次新任務 Agent 都輸出啟動清單         | Agent 直接開始執行，跳過了清單   |
| 2    | 交付質量 | 交付物乾淨，無 AI 套話，格式正確        | 你又看到「綜上所述」「值得注意的是」 |
| 3    | 積分控制 | 同類問題你提第二次時，Agent 提出下沉    | 第三次你還得糾正同一問題         |
| 4    | 層級分離 | memory 裡沒有操作步驟，skill 裡沒有原則 | 某次你打開記憶發現裡面塞滿了步驟 |
| 5    | 閉環執行 | 說「起飛」後 Agent 跑完整九步             | 復盤輸出敷衍、跳過根因分析       |

### 快速修復指令

如果某個檢查點異常，用對應指令快速修復：

- 前饋失效 → `以後每次任務前強制調用 feedforward_startup。`
- 積分失效 → `請查看積分控制表，並確認所有 ≥2 次條目是否已下沉。`
- 層級混淆 → `做一次層級淨化掃描。`
- 閘門鬆懈 → `重新執行一次完整的 delivery_gate 流程，並匯報結果。`

## 6.4 驗證系統正常運行的核心指標

以下四個定量指標可以幫助你判斷系統是否真正生效（可偶爾觀察，不必每日統計）：

### 指標 1：重複糾正率

舊模式：同類問題你平均需要糾正 3-5 次才不再犯。
新模式（啟用積分控制後）：同類問題第 2 次出現即下沉，第 3 次起不應再出現。

**判定**：如果某個已下沉問題出現了第 3 次，說明下沉執行失敗或 Skill 未被更新，需要啟動閉環排查。

### 指標 2：前饋清單命中率

Agent 輸出的啟動清單中，P1 風險預判是否經常被你確認「對，就是這個」。

**判定**：前饋清單中至少 50% 以上的 P1 預判屬於你確实在意的點，則前饋系統運行良好。若連續 5 次任務 P1 全被你說「不對」，說明 L2/L3 需要更新或 Agent 對你的理解有偏差。

### 指標 3：交付前攔截次數

觀察 delivery_gate 的閘門報告中，AI 痕跡、結構缺損被攔截的頻率。

**判定**：初期攔截次數較多（Agent 通過自監控在學習和修正）。持續運行後，P1 類攔截次數應趨近於零——這表示前饋和積分控制已從源頭消除了這些問題，不是閘門失效。

### 指標 4：積分表下沉轉化率

積分表中「已下沉」條目數 / 「累計≥2次」條目總數的比率。

**判定**：比率應長期保持 100%。如果有條目 ≥2 次但長期未下沉，說明積分控制的自動下沉機制未觸發，需要檢查。

## 6.5 調整閾值和配置

系統允許你根據自己的協作風格調整參數。

### 6.5.1 調整積分閾值

閾值默認 = 2。如果你：

- 對穩定性要求極高，希望極偶發反饋不擾動系統 → 調至 3
- 協作節奏很快，希望 Agent 更快學習你的偏好 → 保持 2 或調至 1（不推薦調至 1，會失去抗噪能力）

**調整指令**：
```
將積分控制表的觸發閾值從 2 修改為 3。現在開始，同類反饋出現 ≥3 次才下沉。
```

### 6.5.2 調整閘門檢查項

隨著你的需求變化，閘門的五關可以增減。

**示例**：你開始頻繁要求輸出純 TXT 無頭部，想增加一關：
```
在 delivery_gate 的第五關之後，增加第六關 [P2] 輸出格式：檢查是否為純 TXT 且無頭部信息。不通過則淨化。
```

### 6.5.3 新增場景的 L3 規律

當你開始一個新任務類型，Agent 尚未有對應的 L3。

**方法**：完成幾次該類型任務後，在「起飛」復盤時，Agent 會自動提議新增 L3 規律。你也可以直接說：
```
這是一個新任務類型：[名稱]。請在本次任務完成後，為我提煉一條 L3 核心規律並存入記憶。
```

## 6.6 第一周運行建議

部署後的第一周，建議按以下節奏運行，讓系統充分磨合。

| 天        | 動作                                                         | 目的           |
| :-------- | :----------------------------------------------------------- | :------------- |
| 第 1 天   | 完成部署（第五章），當天所有任務正常進行。觀察前饋清單是否出現。 | 確認組件在線   |
| 第 2 天   | 正常任務。故意給一個相同類型的負面反饋兩次，觀察是否觸發下沉。 | 驗證積分控制   |
| 第 3 天   | 正常任務。說「起飛」，運行第一次真正意義的復盤。               | 驗證閉環流程   |
| 第 4-5 天 | 正常使用，不做額外測試。                                     | 讓系統積累數據 |
| 第 6 天   | 檢查一次積分控制表狀態，確認所有達閾值條目已下沉。           | 維護積分表     |
| 第 7 天   | 做一次完整的「起飛」復盤。檢查畫像 v1 是否需要更新到 v2。      | 形成周閉環     |

一周後，你基本能判斷這套系統是否適合你的協作風格。如果覺得某部分過於繁瑣（如每次任務都要確認清單），可以調整為「直接執行，僅在我說『先列清單』時輸出前饋」。

---

## 6.7 本章總結

日常使用這套系統的要訣只有三個：

1. **正常任務正常發，不要額外操作**——前饋和閘門會自動工作。
2. **有反饋時具體說**——你說的每一句批評都會被積分表捕捉。
3. **定期說「起飛」**——這是系統自我進化的唯一按鈕。

部署不是終點，而是系統開始自進化的起點。每一周、每一次復盤，它都會比上一周更「像」你。
