# 第二章：適用條件與準備工作

## 2.1 這套方法對 Agent 平台的最低要求

在動手之前，先確認你使用的 Agent 平台是否具備以下能力。這些是架構落地的物理基礎。

### 2.1.1 必需能力（缺一則無法完整實施）

| 能力               | 說明                                   | 用途                                                     |
| :----------------- | :------------------------------------- | :------------------------------------------------------- |
| **持久化記憶**     | 跨 session 儲存、讀取、更新的記憶系統       | 承載 L1 元認知框架、L2 使用者邏輯、L3 場景規律、積分控制表 |
| **程序性模組**     | 可建立、呼叫、更新的技能/Skill/GPTs 等 | 承載執行層（交付閘門、前饋啟動、閉環梳理等）             |
| **足夠上下文視窗** | 建議 100k tokens 以上                  | 容納 session 內的前饋清單、避坑記錄、任務執行過程             |

如果你的平台以上三項都具備，可以完整落地本教程的全部內容。

### 2.1.2 加分能力（有則更強，無則降階運作）

| 能力         | 說明                                         | 缺失時的替代方案                                 |
| :----------- | :------------------------------------------- | :----------------------------------------------- |
| **Session 檢索** | 搜尋歷史 session 中的內容和糾正記錄               | 在每次糾正後手動將關鍵教訓寫入記憶，作為前饋素材 |
| **工具呼叫** | Agent 可主動呼叫外部工具（搜尋、程式碼執行等） | 範圍受限但核心架構不受影響                       |

### 2.1.3 如果平台沒有 Skill 功能怎麼辦

一些平台（如基礎版 ChatGPT、簡單聊天介面）沒有技能模組。此時可採用 **「記憶模擬法」**：

- 將 Skill 的內容以結構化記憶條目存放，如 `[Skill-交付閘門] 內容：...`
- 在 L1 元認知框架中加入強制規則：「每次交付前，必須讀取並執行標記為 [Skill-*] 的所有記憶條目」
- 缺點：佔用記憶容量，且需 Agent 主動檢索；優點：邏輯完全保留，適用於輕度使用場景。

## 2.2 自我評估：你適合使用本教程嗎

### 2.2.1 強烈推薦的使用者畫像

- **高精度協作用者**：你對 Agent 的輸出有穩定品質要求（如寫作風格、程式碼規範）
- **重複任務使用者**：你頻繁與 Agent 進行同類任務（如每日/每週相似工作流程）
- **系統思維使用者**：你願意投入前期搭建成本，換取長期協作效率
- **邊界明確使用者**：你能清晰表達喜好和紅線，Agent 犯錯時你願意指出

### 2.2.2 不太適合的使用者

- 只進行一次性、無複用需求的隨機問答
- 使用場景跨越非常廣泛、難以歸納出穩定規律
- 不希望與 Agent 建立長期協作關係

## 2.3 開始前的準備工作

在正式部署之前，完成以下四項準備，可顯著降低後續調整成本。

### 2.3.1 盤點當前 Agent 狀態

執行以下兩條指令，記錄當前狀態作為基線：

```
列出你當前擁有的所有 skill，以及它們的內容摘要。
```

```
列出你 memory 中當前儲存的所有內容，並評估使用率。
```

**記錄要點**：
- 當前有多少個 skill？它們是否仍在使用？
- memory 使用率百分之幾？內容是否已經混雜？
- 是否有大量過時資訊？

### 2.3.2 列出你想固化的偏好與規則

用一個簡單的表格，列出你希望 Agent 長期遵守的原則。這一步幫你從模糊的「感覺」中提煉出可操作的規則：

| 維度     | 你的原則               | 範例                   |
| :------- | :--------------------- | :--------------------- |
| 溝通風格 | 簡潔直接，不用客套話   | 不輸出「希望能夠幫到您」 |
| 輸出格式 | 程式碼優先，附帶驗證方法 | 輸出方案時包含測試步驟 |
| 決策偏好 | 風險前置，先暴露問題   | 先列風險再談收益       |
| 審美取向 | 冷感敘事，拒絕 AI 總結 | 小說結尾留白，不昇華   |

不需要一次窮盡，4-6 條核心原則即可。後續 Agent 會在閉環中幫你不斷補充。

### 2.3.3 回想 3-5 個你頻繁執行的任務類型

- 例如：寫小說 / 程式碼審查 / 維運排查 / 架構設計 / 資料分析
- 每個類型試寫一句話，描述你認為這類任務「做對」的關鍵規律是什麼

**範例**：
- 短篇創作：「情緒定調優先於情節鋪展，結尾必須留白。」
- 程式碼審查：「先找架構問題，再看命名規範；每發現一類問題給修改建議。」
- 問題排查：「先看日誌定位根因，不等猜測；不確定時標註『待驗證』。」

這些將直接成為你的 L3 場景規律初始內容。

### 2.3.4 備份當前狀態

在開始部署前，**備份你當前的記憶和技能狀態**。具體方法取決於你的平台：

- 如果平台支援匯出，先匯出一份
- 如果不支援，手動複製當前 memory 和 skill 列表到本機文件
- 如果有版本管理（如部分平台的記憶快照），建立一個還原點

這一步的目的是：萬一部署過程中出現混亂，你可以還原到初始狀態重新開始。

## 2.4 準備完成的標誌

當你可以回答以下三個問題時，準備階段完成，可以進入第三章：

1. 我的 Agent 平台是否具備持久化記憶 + 技能模組？（是 / 否 / 部分具備）
2. 我梳理出了幾條核心偏好和原則？（建議 ≥4 條）
3. 我列舉出了幾個高頻任務類型，並對每個類型有了初步的「規律判斷」？（建議 ≥3 個）
