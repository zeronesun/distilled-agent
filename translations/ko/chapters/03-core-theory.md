# 제3장: 핵심 이론 상세 — 네 가지 제어론 개념

---

이 장에서는 첸쉐셍의 공학 제어론에서 추출한 네 가지 핵심 개념을 심층적으로 분석합니다: **전송 제어·적분 제어·계층 분리·자기 모니터링**.

각 개념마다 다음 네 가지를 완성합니다:
1. **원리 설명**: 제어론에서 무엇이며, 왜 중요한가.
2. **Agent의 곤경**: 이 개념이 부재할 때 Agent는 어떤 문제에 빠지는가.
3. **실행 설계**: 기억과 스킬로 어떻게 실현하는가.
4. **하나의 검증 가능한 지표**: 어떻게 그것이 이미 작동하고 있는지 판단하는가.

---

## 3.1 전송 제어(Feedforward) — 발생 전에 오류를 막는다

### 3.1.1 원리 설명

제어론에서 전송 제어는 피드백과 병행하는 두 가지 기본 제어 방식입니다.

- **피드백**(Feedback): 출력을 측정하여 편차를 발견하고 다시 돌아와 입력을 수정. 항상 "사후"입니다.
- **전송 제어**(Feedforward): 교란이 시스템에 들어가기 전에 예측에 따라 제어 신호를 미리 조정. "사전"입니다.

공학 기술에서 가장 전형적인 전송 제어의 예: 보일러 시스템이 급수 온도가 떨어짐을 감지하면 증기 압력이 변하기 기다리지 않고 직접 열량을 높입니다. 그것이 측정하는 것은 "교란원"이지 "출력 편차"가 아닙니다.

첸쉐셍은《공학 제어론》에서 반복적으로 강조합니다: **고성능 시스템은 반드시 전송 제어와 피드백을 동시에 포함해야 하며, 피드백에만 의존해서는 안 됩니다.** 피드백은 남은 편차를 수정하고, 전송 제어는 주요 교란을 제거합니다.

### 3.1.2 왜 AI Agent에 전송 제어가 필요한가

전송 제어가 없을 때 Agent의 작동 방식은 순수 피드백입니다:

```
당신: 단편 한 편을 써.
Agent: (출력)
당신: 결말에 AI 요약 하지 마.
Agent: (수정)
다음:
당신: 단편 한 편을 써.
Agent: (다시 AI 요약이 나타남)
당신: 결말에 AI 요약 하지 마!
```

매번 오류 발생 → 당신 정정 → 그것 수정. 이 "당신이 비난해야만 수정하는" 모드에는 몇 가지 치명적 결함이 있습니다:
- **대화 차원 낭비**: 정정이 당신의 시간을 차지합니다.
- **당신의 경각 의존성**: 어떤 때는 잊을 수 있으며, 오류가 그대로 인도됩니다.
- **세션 간 망각**: 이번 정정은 그 대화의 문맥에만 존재하고, 새 세션에서는 기억할 수 없습니다.

전송 제어는 이것을 해결합니다: **"또 틀렸습니다"라고 당신이 말하기 전에 먼저 당신이 신경 쓸 수 있는 것을 작업 시작 단계에 주입합니다.**

### 3.1.3 Agent에서의 실행 설계

전송 제어는 Agent에서 **"작업 시작 시 신호 사전 로딩"**으로 구현됩니다.

**구체적 메커니즘**:
1. 매번 새 작업을 시작할 때, Agent가 능동적으로 세 가지 데이터 소스에서 정보를 추출:
   - **L2 사용자 저수준 논리**: 당신의 장면 간 미학과 원칙(예: "AI 요약 거부").
   - **L3 장면 핵심 규율**: 현재 작업 유형의 성공 규율(예: "단편은 여백 남기기").
   - **세션 아카이브/기억의 정정 기록**: 이번에 해당 유형 작업에서 당신이 뭘 정정했는가.
2. 위 정보를 압축하여 **"회피 체크리스트"**로 만듭니다.
3. 정식 출력 전에 회피 체크리스트를 쓰고(당신이 확인하거나 직접 실행).

**예시**: Agent에 단편을 쓰라고 할 때, 그것은 쓰기 전에 다음을 출력할 것입니다:

> 시작 체크리스트:
> - 작업 유형: 단편 작문
> - 회피 항목: ①결말에 여백 남기고 요약·승화하지 않음; ②"AI로서"와 같은 상투어 등장하지 않음; ③콜드한 서사, 형용사 과다 쌓임 피하기.
> - 지난번 문제: 지난번 결말에 "인생이 바로 이렇다"가 나왔던 바, 이미 회피함.

그 후에야 비로소 글을 쓰기 시작합니다. 이 전송 제어 층은 당신을 "오류 정정자"에서 "확인자"로 바꿉니다.

### 3.1.4 어떻게 그것이 유효한지 검증하는가

다음 행동을 관찰합니다:
- 새 작업 시작 시 Agent가 능동적으로 "시작 체크리스트" 또는 "회피 체크리스트"를 출력하는가.
- 동일 유형 오류가 첫 번째에서부터 현저히 줄어드는가.
- 연속 5회 동일 유형 작업에서 당신이 정정한 횟수가 줄어드는가.

**전송 제어의 목표는 제로 오류가 아니라, 오류가 다시 발생하지 않게 하는 것입니다.**

## 3.2 적분 제어(Integral Control) — 고쳐야 할 것은 고치고, 고치지 말아야 할 것은 마구 바꾸지 않는다

### 3.2.1 원리 설명

제어론에서 컨트롤러는 일반적으로 세 항으로 구성됩니다:

- **비례 제어(Proportional Control, P)**: 편차가 얼마나 큰지, 동작도 그만큼 큰 것.
- **적분 제어(Integral Control, I)**: 편차가 얼마나 오래 지속되는지, 동작을 점진적으로 증대.
- **미분 제어(Derivative Control, D)**: 편차 변화가 얼마나 빠른지, 미리 억제.

순수 비례 제어의 치명적 결함은: **노이즈에 너무 민감**합니다. 우연한 편차(예: 센서 진동)가 한 번 나타나도 불필요한 큰 폭 수정을 트리거하여, 시스템이 진동 - 한 번 왼쪽, 한 번 오른쪽, 결코 안정되지 않습니다.

**적분 제어**는 바로 이 문제를 해결하는 것으로, 편차를 누적하여 일정 수준에 도달할 때만 동작을 트리거합니다. 한 번의 점프가 평균되어버리고, 지속적·안정적인 편차만이 수정을 유발합니다. 이것이 "적분"의 의미 — **오차의 시간 적분을 계산하고, 임계값을 초과하여야 비로소 힘을 냅니다.**

### 3.2.2 왜 AI Agent에 적분 제어가 필요한가

적분 제어가 없을 때, Agent가 당신의 피드백에 대한 응답 방식은 종종 두 가지 극단입니다:

- **과다 민감한 응답**: 당신이 우연히 "이 결말이 좋지 않다"라고 한 번 말하면, 그것이 즉시 전체 글쓰기 스타일을 바꾸고, 당신이 좋아했던 부분마저 변합니다.
- **지연된 응답**: 당신이 "플러그인식 요약을 추가하지 마"라고 다섯 번 말해도, 매번 허락은 하지만 다음 번 여전히 그대로입니다.

이 두 가지 상황의 근원은 **"응답 임계값"을 설정하지 않았기** 때문입니다. Agent가 당신의 표현 한 번을 영구 규칙으로 삼거나, 반복 표현을 매번 독립적인 우연적 사건으로 삼습니다.

적분 제어가 제시하는 방안은 매우 간결합니다: **동일 유형 피드백이 두 번째 나타나야만 비로소 그것을 안정 신호로 하여 장기 기억에 기록하거나 스킬을 수정합니다.**

### 3.2.3 Agent에서의 실행 설계

적분 제어는 **임계값이 있는 피드백 카운터**로 구현되며, 기억체에 저장되고 "적분 제어표"로 명명합니다.

**구조**:

| 클래스 | 오류 모드(≤8자) | 최초 일자 | 누적 회수 | 임계값 | 이미 퇴적 | 퇴적 대상 |
| :--- | :--------------- | :------- | :------- | :--- | :----- | :---------- |
| A class | 결말AI 요약 | 0510 | 2 | 2 | 예 | L2·미학 원칙 |
| B class | 리듬이 너무 빠름 | 0510 | 1 | 2 | 아니오 | L3·단편 규율 |

**작동 규칙**:
1. 당신이 부정적 피드백을 주면, Agent가 그것을 어떤 오류 모드에 올바르게 분류(A class = 미학 원칙, B class = 장면 규율, C class = 스킬 단계 오류).
2. 동일 모드 카운트 +1.
3. 누적 ≥ 임계값(기본 2회)이며 아직 퇴적하지 않았다면 Agent가 자동으로 퇴적 실행:
   - 이 규칙을 대응하는 L2·L3·Skill에 기록.
   - "이미 퇴적" 상태 업데이트.
   - 당신에게 확인: "'결말AI요약'이 2회 나타났으며, 저는 이미 그것을 L2 미학 원칙으로 퇴적했습니다. 동의합니까?"
4. 누적 회수 = 1이면, 이번 수정만 하고 장기 규칙은 바꾸지 않음.

**임계값을 2로 설정한 의의**:
- 1회는 노이즈: 당신 기분이 안 좋거나, 어떤 특정 장면에 적합하지 않았을 수 있음.
- 2회는 신호: 연속 2회 동일 유형 피드백은 이것이 아주 당신의 안정적 선호임을 설명함.
- 당신은 자신의 스타일에 따라 3으로 조정할 수 있으나, 2부터 시작하는 것을 권장.

### 3.2.4 어떻게 그것이 유효한지 검증하는가

- 고의적으로 동일 유형 피드백을 두 번(예: 두 번 "결말이啰嗦"이라고 말함) 주고, 두 번째 후에 Agent가 능동적으로 장기 규칙 수정을 제안하는지 관찰.
- 우연적 피드백(한 번 "이 비유가 마음에 들지 않는다"라고 말함)을 주었을 때, Agent가 그것 때문에 전체 스타일을 바꾸지 않는지 관찰.

**적분 제어의 목표는 Agent를 "완고"하게 만드는 것이 아니라, 안정적인 선호 포착 능력을 부여하는 것 — 들어야 할 것을 놓치지 않고, 무시해야 할 것은 과도 민감하지 않습니다.**

## 3.3 계층 분리(Hierarchical Decomposition) — 전략과 실행의 완전한 분리

### 3.3.1 원리 설명

첸쉐셍 시스템 사고의 핵심 중 하나는 **계층적·계단형 제어**입니다. 복잡한 시스템은 반드시 다른 계층으로 분해되어야 합니다: 상위계가 전략·목표·제약을 정하고; 하위계가 실행·조절·피드백을 합니다. 상하계 간은 명확한 인터페이스를 통해서만 정보 전달하며, 내부 구현은 서로 보이지 않습니다.

이런 분리가 깨지면 — 상위계가 실행 디테일을 관리하기 시작하고, 하위계가 스스로 전략을 정하기 시작하면 — 시스템은 혼란에 빠집니다: 한 데 움직이면 전체가 진동, 한 곳을 수정하면 전체가 진동합니다.

### 3.3.2 왜 AI Agent에 계층 분리가 필요한가

Agent의 기억과 스킬은 자연스럽게 다음과 같은 몇 가지 혼재에 빠지기 쉽습니다:

- **기억 팽창**: 사용자의 원칙("AI 결말 싫어")과 구체적 실행 단계("쓴 후 '종합'을 검색하여 삭제")가 혼재됨. 기억에는 "왜"와 "어떻게"를 모두 저장하여, 부피 폭증, 검색이 어렵습니다.
- **스킬의 경직**: 스킬에 어떤 대화의 선호가 하드코드되었으나, 그 선호는 일시적일 수 있습니다. 다음 사용 시, 스킬이 이미 시대에 뒤떨어졌으나, 당신은 그것이 그렇게 하는 이유를 알 수 없습니다.
- **수정 비용이 높음**: 당신이 출력 스타일 하나를 바꾸려면, 빽빽한 기억과 스킬을 뒤져보아야, 어느 것이 고쳐야 할 "원칙"이고 어느 것이 무관한 "역사 기록"인지 분별해야 합니다.

계층 분리는 넘을 수 없는 경계를 설정하는 것입니다: **원칙은 기억에, 단계는 스킬에 넣으십시오.**

### 3.3.3 Agent에서의 실행 설계

핵심 규칙은 단 하나뿐입니다:

> **기억체는 "왜"와 "규율은 무엇인지"만 저장하고, 스킬은 "어떻게"만 저장합니다.**

구체적 표현은 다음과 같습니다:

| 정보 유형 | 소속지 | 예시 |
| :--------------------------- | :-------- | :------------------------------- |
| 미학적 경향(AI식 결말 싫어) | 기억 L2 | 사용자는 콜드한 서사를 선호, 요약식 수용 거부 |
| 작업 규율(단편은 먼저 정조를 정해야 함) | 기억 L3 | 단편 작문 핵심 규율: 정서적 정조 우선 |
| 실행 단계(쓴 후 게이트 검사) | 스킬 | delivery_gate 다섯 관문 체크리스트 |
| 어떤 구체적 정정 기록 | 보관/검색 | 2025-05-10 사용자는 결말에啰嗦라고 말함 |

**실행 중 강제 규칙**:
- 매번 폐루프 정리 시, Agent는 능동적으로 스캔:
  - 기억 안에 실행 단계가 있는가? 기억이 있다면 스킬로 이동.
  - 스킬 안에 원칙적 내용이 있는가? 기억이 있다면 L2/L3로 이동.
- 매번 새 기억을 쓸 때, Agent는 먼저 정보가 어느 종류에 속하는지 판단한 후 다시 기록 위치 결정.

### 3.3.4 어떻게 그것이 유효한지 검증하는가

- Agent의 기억을 열어, 명확한 분리인 것을 확인: 원칙·규율·장면 매핑 — 혼재된 단계가 없음.
- Agent의 스킬을 열어, 그것에는 "무엇을·무엇을 검사"하는 체크리스트만 있고, "왜 사용자가 좋아하기 때문에 이렇게 한다"라는 설명이 없음.
- 당신이 미학적 원칙 하나(예: "이제 가끔 화려한 비유를 쓸 수 있음")를 바꿀 때, 기억 한 줄만 수정하면 되고, 어떤 스킬도 건드리지 않아도 됨.

**계층 분리의 목표는 기억을 "영양가 없는 잡탕"에서 "데이터베이스"로 바꾸어, 각 정보마다 유일한 소속·확정 수정 경로가 있게 하는 것입니다.**

## 3.4 자기 모니터링(Self-Monitoring) — 인도 전, 먼저 자기 게이트를 거친다

### 3.4.1 원리 설명

어떤 제어 시스템도 마지막에는 반드시 하나의 **피드백 회로**가 닫혀야 합니다. 공학 시스템에서 이것은: 시스템이 자신의 출력을 측정하고, 기대값과 비교하여 오차 신호를 생성해 수정합니다.

Agent에게 있어서, 이 "피드백 회로"는 바로 당신 — 사용자 — 가 출력을 검사하고 정정하는 것입니다. 그러나 당신에게 인도하기 전에, Agent가 스스로 먼저 이 회로를 한 번 돌리게 하면, 자동화된 품질검사 층 하나를 추가하는 셈입니다. 이것이 바로 **자기 모니터링**: 시스템에 내장 품질검사기를 두고, 자신의 출력에 대해 사전 검사를 하고, 통과해야만 인도합니다.

### 3.4.2 왜 AI Agent에 자기 모니터링이 필요한가

Agent의 가장 빈번한 인도 문제는 다음과 같습니다:
- **강제적 요구 누락**: 당신은 순 텍스트 출력을 요청했으니, 그것은 주석이 달린 내부 버전을 줍니다.
- **AI 흔적이 너무 짙음**: "종합적으로""주목할 만한 것은""AI로서" 같은 상투어를 보면 미움이 생깁니다.
- **구조 불완전**: 약속한 세 부분, 두 개만 쓰고 멈춥니다.
- **인도물이 깨끗하지 않음**: 파일에 review notes·내부 라벨이 들어 있어, 당신이 수동으로 청소해야 합니다.

이 문제들을 당신이 검사하지 않으면 인도해주고, 당신이 검사하면 다시 대화 차원 낭비로 정정해야 합니다. 자기 모니터링은 이번 대화 차원의 검사 비용을 당신에게서 Agent 자신에게 이전하는 것입니다.

### 3.4.3 Agent에서의 실행 설계

자기 모니터링은 **인도 전 게이트 스킬(delivery_gate)**로 구현됩니다. Agent는 최종 결과를 출력하기 전에 반드시 이 스킬을 실행하며 각 항목을 검사합니다.

**게이트 검사 항목(다섯 관문)**:

| 관문 | 우선순위 | 검사 내용 | 통과하지 않을 때 동작 |
| :----- | :---------- | :------------------------------------------- | :------------------- |
| 첫 번째 관문 | P0 하드 블록 | 하드적 요구 검사: 자수·형식·금지 항목·반드시 포함 항목 | 강제 수정, 다시 게이트 |
| 두 번째 관문 | P0 하드 블록 | 사실 정확성: 코드 논리·데이터 인용·파일명/경로 | 확실하지 않은 곳은 "검증 필요"로 표시 |
| 세 번째 관문 | P1 높은 우선순위 | AI 흔적: 상투어·도덕적 요약·갑작스러운 승화 | 강제 삭제 |
| 네 번째 관문 | P1 높은 우선순위 | 구조 완전성: 논리 폐루프·반드시 포함 단계 누락 없음 | 보완 |
| 다섯 번째 관문 | P2 최적화 건의 | 인도 깨끗도: 주석·review notes·불필요 라벨 제거 | 정화 |

**실행 단계**:
1. 작업 완료, 인도 준비.
2. Agent 내부에서 `delivery_gate` 호출.
3. 각 관문 스캔하며, 통과/미통과를 기록.
4. 전부 통과 → 최종 결과 출력.
5. 어떤 항목이든 미통과 → 자동 수정 → 다시 게이트 → 전부 통과해야만 출력.

### 3.4.4 어떻게 그것이 유효한지 검증하는가

- Agent 인도 시, 당신은 다음을 검사: 아직 AI 상투어가 있는가? 여전히 불필요한 주석이 있는가? 형식이 완전히 요구를 충족하는가?
- Agent 인도 전, reasoning 또는 중간 출력에서 그것이 검사를 하고 있음이 체현되는지(예: "자체 검사 중: P0 하드 블록 통과, P1 AI 흔적 하나 발견, 이미 삭제").

**자기 모니터링의 목표는 당신에게 딱 인도물이 이미 "깨끗한" 것을 전달하여, 확인만 하면 더 이상 품질검사를 할 필요가 없게 하는 것입니다.**

## 3.5 네 개념의 협력 관계

네 개념은 서로 독립적인 부품이 아니고, 시스템이 작동하는 네 가지 위상입니다:

1. **작업 시작 전**: 전송 제어가 신호를 주입하여 함정을 미리 표시합니다.
2. **작업 실행 중**: 계층 분리가 정보 흐름 깨끗하게 유지 — 상위계가 방향 제시, 하위계가 실행 담당.
3. **사용자 피드백 시**: 적분 제어가 피드백을 장기 규칙으로 승격할지 결정.
4. **작업 인도 전**: 자기 모니터링이 마지막 피드백 회로를 담당하여, 합격하지 않은 출력을 차단.

그리고 **주기적 복기**에서, Agent가 이 네 가지를 기준으로 삼아, 모든 기억과 스킬을 감사하여 자가 진화 완성을 합니다. 이것이 다음 장에서 전개할 시스템 아키텍처 전경입니다.
