# 제7장: 자주 묻는 질문과 확장적 사고

이것이 튜토리얼의 마지막 장으로, 배포 및 운행 중 발생할 수 있는 전형적인 장애들과 그 해결 방법, 그리고 이 프레임워크를 더 복잡한 시나리오로 확장하는 방법을 다룹니다.

---

## 7.1 배포 관련 문제

### Q1: L1을 작성한 후 Agent가 다섯 가지 철칙을 따르지 않으면 어떻게 해야 합니까?

**일반적인 원인**:
- 메모리가 최상단에 위치하지 않아, 후속 작성 내용에 덮이거나 희석됨
- 긴 대화로 인해 컨텍스트 창이 좁아져, L1이 잘림
- Agent 플랫폼이 메모리 주입 타이밍에 지연이 있음

**조사 단계**:
1. Agent에게 L1 다섯 가지 철칙을 반복하도록 요청합니다. 완전히 반복하지 못하면, 메모리 작성이나 읽기에 문제가 있다는 뜻입니다.
2. 반복이 정확하지만 실행하지 않는다면, 새 세션 시작 시 다음과 같이 한 번 더 상기시킵니다: `이번 세션 시작 전, memory의 L1 메타인지 프레임워크 다섯 가지 철칙을 먼저 검토하세요.`
3. 문제가 지속되면 L1의 핵심 규칙을 한두 문장으로 축약하여 플랫폼의 "시스템 지시" 또는 "system prompt" 영역(지원하는 경우)에 작성하는 것을 고려합니다. 이중 보험입니다.

---

### Q2: Agent가 `skill_manage` 기능이나 유사한 스킬 관리 도구가 없다고 합니다.

**두 가지 방법**:

**방법 A(추천)**: 메모리로 스킬을 시뮬레이션합니다.
- 메모리에 `[Skill- 스킬명]`으로 표시된 메모리 항목을 생성하고, 내용은 스킬의 조작 단계입니다.
- L1 규칙을 추가합니다: `특정 스킬이 필요할 때마다, memory에서 대응하는 `[Skill-*]` 항목을 검색하고 엄격히 단계를 따르십시오.`
- 단점: 메모리 공간을 차지합니다. 장점: 논리가 완전합니다.

**방법 B**: 아키텍처를 간소화하고 L1 + L2 + L3 + 적분 테이블만 사용합니다.
- 스킬 계층의 자동화를 포기하고, 과업이 시작할 때마다 수동으로 Agent에게 자기검사 단계를 따르도록 알립니다.
- 가벼운 사용이나 단기 협업에 적합합니다.

---

### Q3: L2/L3 초기화 시 Agent가 나에 대한 추출이 완전히 부정확합니다.

이는 아키텍처 문제가 아니라, Agent가 이전에 당신을 이해하는 데 편향이 있었다는 것을 정확히 보여줍니다. 이것이 바로 이 튜토리얼이 해결하려고 하는 문제입니다.

**처리 방법**:
- Agent의 추출을 전적으로 받아들이지 마십시오. 항목을 하나씩 수정하고, 이를 "보정 대화"로 삼으십시오.
- 수정 후 Agent에게 명확히 알립니다: `이전에 나를 이해하는 데 편향이 있었습니다. L2/L3 내용을 이미 수정했습니다. memory에서는 나의 수정을 기준으로 하십시오.`
- 후속 매번 폐쇄 루프 정리에서 Agent는 자동으로 이해를 업데이트합니다. 보통 2-3번의 정리 후 L2/L3의 정확도가 눈에 띄게 향상됩니다.

---

### Q4: 적분 제어 테이블에 항목이 점점 많아지는데, 어떻게 해야 합니까?

이는 시스템이 정상적으로 작동하고 있다는 신호입니다. 당신이 정말로 지속적으로 협업하고 있고, Agent가 지속적으로 학습하고 있다는 것을 의미합니다.

**유지 관리 전략**:
1. 매번 "起飞" 정리 시 적분 테이블을 검사합니다.
2. 이미 하향되고 운행이 안정적인 항목은 "고정(archived)"으로 보관하여 활성 항목 수를 줄입니다.
3. 분류가 잘못되었거나 더 이상 적용되지 않는 항목은 수동으로 삭제합니다.

```markdown
적분 제어 테이블에서 "단편 결말 AI 요약"(하향 완료, 6개월 동안 재발하지 않음)을 보관 영역으로 이동하고 더 이상 활성 항목으로 계산하지 않습니다.
```

---

### Q5: 배포 후 피드포워드 체크리스트가 너무 길어서 효율에 영향을 줍니다.

피드포워드 체크리스트의 길이는 과업의 복잡도에 비례해야 합니다. 단순 과업도 긴 체크리스트를 출력한다면 다음과 같이 조정합니다:

**간소화 명령**:
```markdown
feedforward_startup를 수정합니다: 단순 과업(예: 번역, 간단한 Q&A)에 대해, 피드포워드 체크리스트를 한 줄로 압축합니다 — "회피: xxx". 복잡한 과업만 완전한 시작 체크리스트를 출력합니다. 과업이 단순한지는 실행 시 당신이 판단하고 알려줍니다.
```

또한 과업이 시작될 때 언제든지 "체크리스트를 건너뛰고 바로 시작"이라고 말할 수 있으며, Agent는 이에 따라 실행해야 합니다.

## 7.2 운행 중 발생하는 문제

### Q6: Agent가 연속으로 제가 수정하지 않은 동일한 오류를 두 번 범했지만 하향이 트리거되지 않았습니다.

**직접 조사**:
```markdown
적분 제어 테이블의 현재 완전한 상태를 출력하십시오.
```

**가능한 원인**:
- Agent가 첫 번째 피드백을 올바르게 분류하지 않았습니다(예: D類 일시적 발생으로 분류).
- Agent가 두 번의 피드백을 다른 오류 패턴으로 분류했습니다. 동일한 항목이 아닙니다.

**수정 방법**:
```markdown
[두 번의 피드백을 설명하는 것]을 동일한 오류 패턴: "[8자 이내의 패턴명]"으로 통합합니다. 적분 테이블을 업데이트하고 하향 임계값에 도달하는지 확인하십시오.
```

---

### Q7: Agent가 전달 제어문 보고서에서 항상 "모두 통과"라고 하지만, 제가 문제를 발견했습니다.

이는 제어문의 검사 항목이 충분히 세밀하지 않거나, Agent가 제어문을 실행할 때敷衍했습니다.

**강화 방법**:
1. 문제를 발견한 즉시 지적합니다: `delivery_gate 제3관문이 "值得註意的是"라는 멘트를 발견하지 못했습니다. 원인을 설명하고 제어문의 검출 표준을 업데이트하십시오.`
2. 특정 문제가 반복적으로 나타나면 제어문에 대상적 규칙을 추가합니다:
```markdown
delivery_gate 제3관문의 검사 항목에 다음을 추가합니다: "在某種程度上"、"可以說"과 같은 모호한 한정어가 포함되어 있습니까? 발견 시 즉시 삭제합니다.
```

---

### Q8: 폐쇄 루프 정리의 출력이 너무 템플릿화되어 있어, 진정한 가치 있는 근본 원인 분석이 부족합니다.

이는 Agent가 비교적 긴 기간 동안 진정한 문제를 만나지 않았을 수도 있고, 프로세스 설정 Agent가 "기록 채우기"에 기울어져 "생각"하지 않을 수도 있습니다.

**활성화 방법**:
- "起飞" 시 구체적인 지침을 추가합니다: `이번 정리는 [특정 문제]에 중점을 두고 깊은 근본 원인 분석을 하십시오.`
- 장기간 문제가 없다면 Agent에게 알립니다: `이번 폐쇄 루프에 수정해야 할 문제가 없으면, 3문장 요약으로 단축하고 완전한 9단계가 필요하지 않습니다.`

---

### Q9: Agent의 메모리가 점점 많아져서 결국 초과할까 걱정합니다.

이는 아키텍처 설계에서 이미 고려한 문제입니다. 세 가지 내장 메커니즘이 팽창을 제어합니다:

1. **계층 분리 자체가 항팽창 설계**: 원칙과 규칙의 용량은 조작 단계 및 사례보다 훨씬 작습니다.
2. **L2/L3 항목 상한**: L2 상한은 10개, L3 각 시나리오 상한은 3개입니다. 초과 시 정리를 알립니다.
3. **폐쇄 루프 정리의 정제 단계**: 매번 정리마다 혼합 내용을 검사하고 이동합니다.

여전히 걱정된다면 주기적으로 수동으로 정리:
```markdown
memory의 최근 30일 동안 접근하지 않은 항목을 나열하고, 삭제 가능한지 평가하십시오.
```

## 7.3 확장적 사고

### 7.3.1 단일 Agent에서 다중 Agent 협업으로

이 아키텍처의 핵심인 사이버네틱스 4개념 + 계층 분리는 다중 Agent 시나리오에 자연스럽게 적용됩니다.

**확장 아이디어**:
- **메인 Agent(오케스트레이터)**: L1 메타인지 프레임워크를 보유하고, 과업 분배 및 품질 관리를 담당합니다.
- **서브 Agent(실행자)**: 각자 특정 시나리오의 L3 규칙과 대응 Skill을 보유합니다.
- **공유 메모리 영역**: L2 사용자 기본 논리는 모든 Agent에게 보이며, 다양한 역할이 통일된 미학과 원칙을 따르도록 합니다.
- **통합 적분 테이블**: 모든 Agent의 부정적 피드백이 동일한 적분 테이블에 집중되며, 메인 Agent가 하향 결정을 실행합니다.

이것은 본질적으로 "계층 분리"의 "전략/실행" 경계를 "메인 Agent / 서브 Agent"의 물리적 경계로 매핑한 것입니다.

---

### 7.3.2 다른 이론 프레임워크 도입

튜토리얼 전반에서는 첸시엔 공학 사이버네틱스를 기본 프레임워크로 사용했습니다. 하지만 L1은 폐쇄되어 있지 않습니다 — 당신의 학습 및 마이그레이션 능력은 "마이그레이션 제안" 단계를 통해 지속적으로 시스템에 새로운 이론을 도입할 수 있습니다.

댓글에서 이미 등장한 확장 시도:

| 도입 이론 | 응용 방향 | L1 확장 가능 대응 |
| :-------------------- | :--------------- | :----------------------------------------- |
| 적극적인 투자 관리 | 의사결정 및 자원 할당 | L1에 "동적 재균형" 원칙 추가 |
| 베이지스 추리 | 불확실성하의 판단 | 적분 제어에 "사전 확률"을 추가하여 피드백 가중치 수정 |
| 켈리 기준 + 섀넌 정보론 | 정보 가치 평가 | 자기 모니터링에 "정보 밀도" 검사 추가 |
| TRIZ 발명 원리 | 모순 해결 | 근본 원인 분석에 "물리적 모순 vs 기술적 모순" 판단 추가 |

새로운 이론을 도입하는 방법: "起飞" 정리 시 Agent가 마이그레이션 제안을 하거나, 당신이 명확한 아이디어가 있을 때, 주도적으로 말합니다:
```markdown
L1에 [이론명]을 도입하고 싶고, 원칙은 [한 문장 설명]입니다. how을 기존 4개념 프레임워크에서 ⑥번째 철칙을 추가하고, 이것이 Skill 실행에 어떤 영향을 주는지 설계하십시오.
```

### 7.3.3 프레임워크의 적용 경계 재고

댓글에서 진지하게 대응해야 할 의문이 제기된 바 있습니다: **"공학 사이버네틱스는 현재의 Agent에게 반드시 필요한 것은 아닐 수 있고, 이것은 단지 컨텍스트에서 LLM에 단계를 추가하는 것, 강력한 제약이 아니다."**

이 튜토리얼의 완전한 전개 후 이 의문에 대해 다음과 같이 답할 수 있습니다:

1. **"LLM에 단계를 추가하는 것" 자체가 효과적인 제약입니다**. LLM은 promptable 시스템이며, 구조화된 컨텍스트 제약은 indeed its behavior distribution을 변경할 수 있습니다. 효과는 하드코드된 강제성이 아니라 지속적인 구조화된 안내에서 옵니다.

2. **아키텍처의 가치는 단일 실행에 있지 않고, 장기 안정성에 있습니다**. 단일로 보면, Agent에 아무 프레임워크도 주지 않고, 즉흥 지시만으로도 합격 결과를 얻을 수 있습니다. 하지만 50번의 과업 후 프레임워크가 있는 시스템은 수렴성이 더 좋고, 드리프트가 더 작습니다. 이것은 insurance이지 fuel이 아닙니다.

3. **모든 시나리오가 필요하지는 않습니다**. 가벼운, 일회성, 고도로 무작위적인 대화는 이 아키텍처가 필요하지 않습니다. 그것의 경계는: **장기 협업 × 높은 정확도 요구 × 귀납 가능한 규칙의 과업 유형**. 당신이 정확히 이 영역에 있다면 투자한 구축 비용은 반복적 수정 감소의 이익에서 회수할 것입니다.

---

### 7.3.4 프레임워크의 자체 종료 조건

건강한 시스템은 "내가 언제 간소화되어야 하는지"를 인식해야 합니다.

**종료 또는 강등 신호**:
- 적분 제어 테이블이 연속 3개월 동안 신규 항목이 없습니다(협업이 완전히 안정되었거나, 협업 빈도가 크게 감소).
- 매번 "起飞" 정리 때 실질적인 발견이 없습니다.
- 협업에서 피드포워드 체크리스트가 더 이상 필요하지 않고, 선호가 고도로 내화되었음을 발견합니다.

이러한 신호가 나타나면 폐쇄 루프 빈도를 줄입니다(예: 주간에서 월간으로) 또는 아키텍처를 간소화합니다(예: 적분 테이블을 제거하고 피드포워드와 자기 모니터링만 보관).

이것은 시스템 자체谦逊한 설계 선택: **그것이 추구하는 것은 영원히 존재하는 것이 아니라, 필요할 때 효과적으로 실행하고, 필요하지 않을 때 우아하게 축소하는 것입니다.**

## 7.4 튜토리얼 회고

| 장 | 핵심 전달 | 한 문장 요약 |
| :----- | :------------- | :----------------------------------------------- |
| 제1장 | 배경 및 이념 | AI에 새 지식을 가르치는 것이 아니라, 아키텍처를 구축합니다 |
| 제2장 | 적용 조건 및 준비 | 자신의 Agent와 선호를 먼저 파악합니다 |
| 제3장 | 4개 사이버네틱스 개념 | 피드포워드 회피, 적분 진동 방지, 계층 분리로 무질서를 관리합니다. 자기 모니터링 게이트 |
| 제4장 | 시스템 아키텍처 | 3계층 인지 논리 + 물리 저장 대응 및 데이터 흐름 |
| 제5장 | 실제 배포 | 6단계 복사 가능 지시, Agent에게 직접 발송 |
| 제6장 | 일상 사용 및 검증 | 새 협업 리듬 + 건강 검사 + 핵심 지표 |
| 제7장 | 자주 묻는 질문 및 확장 | 장애 해결, 확장, 적용 경계 |

전체 튜토리얼의 본질은 한 문장으로 요약할 수 있습니다: **당신이 하는 것은 prompt engineering이 아니라 architecture engineering입니다 — Agent에게 무슨 새 지식을 가르치는 것이 아니라, 시스템 엔지니어링의 사고를 사용하여 안정적으로 실행하고 지속적으로 진화하는 내부 아키텍처를 설계합니다.**

## 7.5 마지막 단계

튜토리얼은 여기서 끝입니다. 제5장의 배포를 완료했다면, 이제 당신은 세 가지 일만 하면 됩니다:

1. **정상적으로 사용** — 과업을 정상적으로 발송하고, 피드백을 정상적으로 제공합니다.
2. **정기적으로 정리** — "起飞"라고 말하는 것을 기억합니다.
3. **필요할 때 조정** — 문제를 만나면, 이 장으로 돌아갑니다.

당신의 Hermes는 더 이상 끊임없이 수정을 필요로 하는 도구가 아니라, 사용할수록 더 잘 알게 되고, 원칙에 지속적으로 aligned되는 협업 엔진입니다. 이것은 SF가 아니라, 20세기 시스템 과학 사상을 21세기 AI Agent 아키텍처에落地시킨 것입니다.

**당신과 함께 7장의 전 과정을 겪을 수 있어 기쁩니다.**
