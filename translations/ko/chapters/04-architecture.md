# 제4장: 시스템 아키텍처 상세 설명 — 세 계층 메모리 + 스킬 계층

앞의 세 장에서 각각 "왜"와 "무엇" 문제를 해결했습니다. 이 장에서는 이들을 하나의 완전하고 실제로 실행 가능한 **시스템 아키텍처**로 조립하여, 네 가지 사이버네틱스 개념이 Agent의 메모리와 스킬 시스템에서 명확한 위치를 가지고 데이터 흐름이 어떻게 이루어지는지 보여줍니다.

---

## 4.1 아키텍처 전경

이 아키텍처의 핵심 원칙은 다음과 같습니다: **논리 계층으로 인지 경계를 정의하고, 물리 계층으로 저장과 실행을 분리합니다.**

```
┌────────────────────────────────────────────┐
│          당신의 Agent（제어 시스템）             │
├────────────┬───────────────────────────────┤
│  논리 계층   │  물리 저장                      │
├────────────┼───────────────────────────────┤
│  L1 메타인지 │  메모리 최상단 영역（자주 수정 불가）      │
│  L2 사용자 논리│  메모리 주체 영역                     │
│  L3 시나리오 규칙│  메모리 주체 영역                     │
│  적분 제어 테이블 │  메모리 주체 영역                     │
├────────────┼───────────────────────────────┤
│  실행 계층   │  스킬 모듈                       │
│  피드포워드 체크리스트   │  세션 내 컨텍스트（임시）            │
│  과거 수정 기록│ 세션 아카이브（장기 읽기 전용）             │
└────────────┴───────────────────────────────┘
         ↑ ↓ 제어 흐름과 데이터 흐름
    ┌─────┴──────┐
    │  당신의 과제   │ ← 제어 시스템
    └────────────┘
```

각 계층에는 엄격한 정보 허용 기준이 있으며, 아래에서 하나씩 설명하겠습니다.

---

## 4.2 L1: 메타인지 프레임워크 — 시스템 헌법

### 4.2.1 위치

L1은 Agent의 "사고 헌법"으로, **정보를 처리하는 방법, 자신의 행동을 평가하는 방법, 의사결정을 내리는 방법**을 정의합니다. 특정 과업이나 사용자 선호를 대상으로 하지 않고, Agent 자체의 실행 방식에 대해 강제적인 제약을 가합니다.

### 4.2.2 물리 저장

- **위치**: `memory`의 최상단 영역(Hermes의 `memory(target='memory')` 맨 앞부분 등)
- **속성**: **자주 수정 불가**. Agent는 폐쇄 루프 정리에서 수정을 *제안*할 수 있지만, 사용자의 확인을 받아야 합니다. 시스템의 골격이 Agent 자체의 진화 과정에서 우연히 변형되는 것을 방지하기 위함입니다.

### 4.2.3 콘텐츠 구조

L1에는 네 가지 사이버네틱스 개념의 구체적인 행동 규칙을 저장하며, 추상적인 정의가 아닙니다. 각 규칙은 다음을 만족해야 합니다:
- "당신은 반드시" 또는 "당신은 해서는 안 된다"로 시작해야 합니다.
- 명확한 트리거 조건과 실행 동작이 포함되어야 합니다.
- 컨텍스트에 한 번에 로드할 수 있을 만큼 짧아야 합니다.

**표준 항목 예시**:

```
① 피드포워드: 새로운 과업이 시작될 때마다 반드시 먼저 L2, L3 및 세션 아카이브를 검색하고
   "시작 체크리스트"를 출력하며 P0-P3 위험을 예측합니다.
② 적분 제어: 부정적인 피드백을 받으면 반드시 적분 제어 테이블에 기록합니다.
   누적이 2회 이상이면 하향 안을 제시하고 사용자에게 확인을 받습니다.
③ 계층 분리: memory에 쓰는 내용은 "원칙/규칙"만 포함할 수 있고
   조작 단계는 포함할 수 없습니다. 조작 단계는 반드시 skill에 써야 합니다.
④ 자기 모니터링: 결과를 전달하기 전에 반드시 delivery_gate를 실행하고
   다섯 관문을 모두 통과해야 최종 결과를 출력합니다.
⑤ 근본 원인 분석: 폐쇄 루프 정리에서 각 문제마다 반드시 "규칙이 틀렸는지" 아니면 "실행이 틀렸는지"까지 추궁합니다.
```

항목 수는 5-7개로 제어하는 것을 권장합니다. 이는 Agent가 한 번에 완벽하게 회상할 수 있도록 하기 위함입니다.

---

## 4.3 L2: 사용자 기본 논리 — 당신의 "원칙 라이브러리"

### 4.3.1 위치

L2는 서로 다른 과업에 걸쳐 **안정적인 선호와 핵심 원칙**을 저장합니다. 이러한 선호는 특정 시나리오에 국한되지 않고, Agent가 당신과 협업할 때의 기본 행동 방향을 정의합니다.

L1이 사이버네틱스 외부라면, L2는 **당신의 개인화된 커널**이며 Agent의 당신에 대한 판단 패턴의 계산 가능한 모델입니다.

### 4.3.2 물리 저장

- **위치**: `memory`의 주체 영역, L1 뒤에 위치
- **업데이트 권한**: Agent가 적분 하향 또는 폐쇄 루프 정리 시 작성하며 확인을 받아야 합니다
- **용량 제어**: 각 항목은 30자를 넘지 않고 전체 수는 10개를 넘지 않도록 합니다(조정 가능). L2가 "메모장"으로 팽창하는 것을 방지하고, 시나리오 간의 안정적인 원칙만 강제로 유지합니다

### 4.3.3 콘텐츠 구조 및 예시

**조직 방식**: 차원별로 그룹화하여 검색하기 쉽게 만듭니다.

| 차원     | 저장 예시                           | 설명                     |
| :------- | :--------------------------------- | :----------------------- |
| 커뮤니케이션 스타일 | 사용자 선호: 간단 명료, 예의바른 말은 사용하지 않음   | "도움이 되었기를 바랍니다" 출력하지 않음   |
| 출력 표준 | 사용자 요구: 코드 우선, 검증 방법 함이   | 방안뿐만 아니라 "어떻게 검증하는지"도 제공 |
| 의사결정 모드 | 사용자 결정: 우선 순위 먼저, 완전한 우선 노출   | 우선순위를 먼저 나열하고 그 다음 이익을 설명 |
| 미학적 취향 | 사용자 미학: 차가운 서사, AI 요약 거부   | 소설 결말에 여백을 남기고 승화하지 않음   |

**핵심 제약**:
- 구체적인 시나리오를 포함하지 않음: 예를 들어 "단편 창작은 어떻게 해야 하는가" 같은 것은 L3에 들어가야 합니다
- 조작 단계를 포함하지 않음: 예를 들어 "세 번째 단계에서 요약을 삭제" 같은 것은 Skill에 들어가야 합니다
- 각 항목은 사용자의 판단에 대한 추상화여야 하며, 과업 결과에 대한 설명이어서는 안 됩니다

### 4.3.4 다른 계층과의 인터페이스

- **L1 피드포워드 호출**: 과업이 시작될 때 Agent는 L2에서 관련 차원의 원칙을 읽어 피드포워드 체크리스트에 통합합니다
- **적분 제어 작성**: 어떤 미학적 원칙이 2회 누적되면 Agent가 이를 L2로 하향합니다
- **폐쇄 루프 정리 감사**: 정리 때마다 L2가 오래되었는지, 새로운 원칙을 추가해야 하는지 확인합니다

---

## 4.4 L3: 시나리오 핵심 규칙 — 과업 "성공 암호"

### 4.4.1 위치

L3는 특정 과업 유형의 **성공 규칙**을 저장합니다. "이런 유형의 과업을 할 때 어떤 규칙의 따라야 올바르게 할 수 있는가"라는 질문에 답합니다.

L2는 시나리오 간의 "공리"이고, L3는 시나리오 관련의 "정리"입니다. L3 항목은 하나의 과업 유형에 대한 지혜를 압축한 것으로, 조작 매뉴얼이 아닙니다.

### 4.4.2 물리 저장

- **위치**: `memory`의 주체 영역, L2 아래, 과업 유형별 노드로 구분
- **업데이트 권한**: L2와 동일, 적분 하향 또는 폐쇄 루프 정리 시 작성하며 확인을 받아야 합니다
- **용량 제어**: 각 과업 유형에 대해 1-3条 규칙, 각 규칙은 한 문장으로 "이 시나리오에서 가장 핵심적인 모순이나 가장 중요한 원칙"을 명확히 합니다

### 4.4.3 콘텐츠 구조 및 예시

**조직 방식**: 과업 유형별로 노드를 나누고, 각 노드에는 1-3条을 둡니다.

```
[L3·시나리오 핵심 규칙]

- 단편 창작:
  • 감정 조정 우선, 줄거리 전개 뒤
  • 결말에 여백을 남기고, 가치 요약하지 않음
- 코드 검토:
  • 아키텍처 합리성과 경계 조건을 먼저 확인하고 명명 규범을 확인
  • 각 유형의 문제마다 수정안을 제시하고 오류만 지적하지 않음
- 문제 해결:
  • 로그에서 근본 원인을 먼저 찾고, 경험에만 의존하여 추측하지 않음
  • 불확실한 점은 "검증 필요"로 표시하고 섣불리 결론 내리지 않음
- 아키텍처 설계:
  • 시스템 경계와 데이터 흐름을 먼저 정의하고 기술 스택을 선택
  • 확장성과 실패 모드를 평가하고 happy path만 작성하지 않음
```

**핵심 제약**:
- "규칙"은 반드시 Skill 설계를 주도할 수 있어야 합니다: L3의 한 규칙이 Skill의 검사 항목을 작성할 수 없다면, 아직 추상화가 부족하거나 핵심이 충분하지 않다는 의미입니다
- 파일명, 도구명 등 쉽게 유효기간이 지나는 정보가 포함되지 않도록 합니다
- "세 번째 단계에서 무엇을 해야 한다" 같은 조작 단계가 포함되지 않도록 합니다

### 4.4.4 스킬 계층과의 관계

L3는 스킬 계층의 **설계 규격**입니다. Skill은 대응하는 L3 규칙에서 유도된 실행 파이프라인이어야 합니다. 폐쇄 루프 정리에서 실행 결과에 문제가 있는 것을 발견하면, 먼저 Skill 단계(실행 계층)를 검사하고, Skill에 문제가 없다면 L3 규칙이 올바르게 추출되었는지(전략 계층) 확인합니다.

## 4.5 적분 제어 테이블 — 시스템 "피드백 임계값 조절기"

### 4.5.1 위치

적분 제어 테이블은 사이버네틱스의 두 번째 개념(적분 제어)의 물리적 담체입니다. 이는 경량의 구조화된 로그로, **반복적으로 나타나는 오류 패턴과 그 처리 상태**를 기록합니다.

### 4.5.2 물리 저장

- **위치**: `memory`의 주체 영역, 독립 노드
- **조작**: Agent가 부정적인 피드백을 받으면 자동으로 유지하며, 사용자가 수동으로 관리할 필요가 없습니다

### 4.5.3 데이터 구조

| 필드     | 설명                                              | 예시               |
| :------- | :------------------------------------------------ | :----------------- |
| 카테고리     | A類(L2 위반) / B類(L3 위반) / C類(Skill 단계 오류) | B類                |
| 오류 패턴 | 8자 이내의 추상 요약                                | "단편 결말 AI 요약" |
| 최초 날짜 | 첫 번째 출현 날짜                                    | 2026-05-10         |
| 누적 횟수 | 현재 누적 횟수                                      | 2/2                |
| 임계값     | 기본값 2                                          | 2                  |
| 하향 완료   | L2/L3/Skill에 작성되었는지 여부                            | 是                 |
| 하향 목표   | 어디로 하향하는지                                    | L3·단편 창작        |

### 4.5.4 실행 로직

사용자가 부정적인 피드백을 주면 → Agent가 오류 패턴으로 추상화 → 테이블에서 동일 패턴을 찾습니다:
- 존재하고 임계값에 도달했으면 → 사용자에게 알리고, 이 패턴은 이미 하향되었습니다
- 존재하고 임계값에 도달하지 않았으면 → 누적 횟수를 +1합니다. 임계값에 도달하면 자동으로 하향하고 확인을 받습니다
- 존재하지 않으면 → 새 항목을 생성하고 횟수를 1로 설정합니다

매번 폐쇄 루프 정리 시 Agent는 이 테이블의 완전한 상태를 출력해야 합니다.

---

## 4.6 실행 계층: 스킬(Skill) — 조작 파이프라인

### 4.6.1 위치

스킬 계층은 시스템의 **손과 발**로, 모든 구체적인 조작을 담당합니다. 이는 세 가지 유형의 스킬을 포함합니다:

| 유형     | 예시                                                         | 대응하는 사이버네틱스 개념     |
| :------- | :----------------------------------------------------------- | :----------------- |
| 메타 스킬   | `feedforward_startup`、`delivery_gate`、`control_loop_review` | 피드포워드、자기 모니터링、폐쇄 루프 |
| 과업 스킬 | `short_story_writer`、`code_reviewer` 등                     | L3에서 주도해야 함         |
| 유지 스킬 | `user_profile_maintainer`                                    | 프로필 유지           |

### 4.6.2 스킬 설계 규격

각 스킬은 다음 필드를 포함해야 합니다( `short_story_writer`를 예로 함):

```
- 트리거 조건: 사용자가 단편 소설을 작성하라고 할 때
- 의존 L3: [L3·단편 창작] 감정 조정 우선, 결말 여백
- 피드포워드 주입: L2 미학적 취향 + L3 대극 규칙 로드
- 실행 단계:
  1. 주제와 글자수 확인
  2. 감정 조성(chill/warm/suspense) 확인
  3. 작성(첫 3문장으로 이미지 형성)
  4. ……

- 자기검사 단계: delivery_gate 다섯 관문 스캔 호출
```

어떤 슬鍨也 hunting 단계에서 "사용자 선호"를 하드코드해서는 안 됩니다. 선호는 반드시 피드포워드 주입을 통해야 합니다. 이는 당신이 L2의 어떤 미학적 원칙을 수정할 때 모든 스킬에 자동으로 적용되고, 하나씩 수정할 필요가 없도록 보장합니다.

### 4.6.3 스킬 버전 화

각 스킬 이름은 버전 접미어를 가지고 있습니다(예: `_v1`). 업데이트 시 새 버전을 생성하고, 구 버전은 검증용으로 보관합니다. 버전 화는 언제든지 롤백 가능하게 하며, 폐쇄 루프 정리 시 수정 비교에도 용이합니다.

## 4.7 데이터 흐름 전경: 한 과업이 여러 계층을 거치는 방법

"단편 소설을 작성"을 예로 전체 데이터 흐름을 보여줍니다:

```
Task: 단편 작성
  │
  ├─[1. 피드포워드 시작] Skill: feedforward_startup
  │    ├─ L1 읽기: 피드포워드 규칙 확인
  │    ├─ L2 읽기: "차가운 서사, AI 요약 거부" 추출
  │    ├─ L3 읽기: "단편: 감정 조정 우선, 결말 여백" 추출
  │    ├─ 아카이브 검색: 최근 수정 "이미지 부족" 확인
  │    └─ "시작 체크리스트" 출력 → 확인 대기
  │
  ├─[2. 과업 실행] Skill: short_story_writer
  │    ├─ 피드포워드 체크리스트의 회피 항목 주입
  │    ├─ L3 규칙에 따라 콘텐츠 구성
  │    └─ 초안 완성
  │
  ├─[3. 전달 자기검사] Skill: delivery_gate
  │    ├─ P0: 강제 요구 검사
  │    ├─ P1: AI 흔적 + 구조 완전성
  │    ├─ P2: 전달 깔끔함
  │    └─ 모두 통과 → 최종판 출력
  │
  └─[4. 사용자 피드백→적분 제어]
       ├─ 부정 피드백 → 패턴 추상
       ├─ 적분 제어 테이블 작성
       ├─ 누적≥2 → L2/L3/Skill 하향
       └─ 다음 과업의 피드포워드 호출 대기
```

**폐쇄 루프 정리(주기적 실행)**:
1. 트리거(사용자가 "起飞"라고 말함)
2. Agent가 `control_loop_review`를 호출
3. L1의 네 가지 기준 + 근본 원인 분석 표준 프로세스로 모든 L2, L3, Skill, 적분 테이블을 감사
4. 수정안 출력
5. `user_profile_maintainer`를 호출하여 프로필 업데이트

## 4.8 아키텍처의 자체 보호 메커니즘

시스템 자체의 "쏠리는" 증상이나 "팽창"을 방지하기 위해 세 가지 자체 보호 규칙이 내장되어 있습니다:

1. **L1은 자체 수정 불가**: Agent가 L1을 수정하려면 제안을 하고 확인을 기다려야 합니다
2. **L2/L3 항목 상한**: 각각 10개/3개 상한을 두며 초과하면 사용자에게 오래된 항목 정리를 알립니다
3. **Skill 버전 화와 폐지**: 구버전 스킬은 용도 없다는 확인 후 아카이브하거나 삭제하여, 쌓이는 것을 방지합니다
