# 제2장: 적용 조건 및 준비 작업

## 2.1 이 방법이 Agent 플랫폼에 요구하는 최소 요구 사항

작업을 시작하기 전에, 사용하시는 Agent 플랫폼이 다음 능력을 갖추고 있는지 확인하십시오. 이들이 아키텍처 구현을 위한 물리적 기초입니다.

### 2.1.1 필수 능력(부재 시 완전 실행 불가)

| 능력 | 설명 | 용도 |
| :----------------- | :------------------------------------- | :------------------------------------------------------- |
| **지속 가능한 기억** | 세션 간 저장·읽기·업데이트 가능한 기억 시스템 | L1 메타인지 프레임워크·L2 사용자 논리·L3 장면 규율·적분 제어표 담당 |
| **절차적 모듈** | 생성·호출·업데이트 가능한 스킬/Skill/GPTs 등 | 실행계층(인도 게이트·전달 스타트·폐루프 정리 등) 담당 |
| **충분한 컨텍스트 윈도** | 100k 토큰 이상 권장 | 세션 내 전달 체크리스트·회피 기록·작업 실행 과정 수용 |

플랫폼이 위 세 항목 모두 갖추고 있다면, 이 튜토리얼의 전체 내용을 완전히 실행할 수 있습니다.

### 2.1.2 플러스 능력(있으면 더 강력, 없으면 하향식 실행)

| 능력 | 설명 | 부재 시 대체 방안 |
| :----------- | :------------------------------------------- | :----------------------------------------------- |
| **세션 검색** | 역사적 세션에서 내용과 정정 기록 검색 | 각 정정 후 핵심 교훈을 기억에 수동으로 기록하여 전달 제어 자료로 사용 |
| **도구 호출** | Agent가 능동적으로 외부 도구 호출(검색·코드 실행 등) | 범위는 제한되지만 핵심 아키텍처에 영향 없음 |

### 2.1.3 플랫폼에 Skill 기능이 없으면 어떻게 해야 하는가

일부 플랫폼(기본판 ChatGPT·단순 채팅 인터페이스 등)에는 스킬 모듈이 없습니다. 이때는 **"기억 모사법"**을 사용할 수 있습니다:

- Skill의 내용을 구조화된 기억 항목(예: `[Skill-인도 게이트] 내용: ...`)으로 저장.
- L1 메타인지 프레임워크에 강제 규칙 추가: "매번 인도 전, `[Skill-*]`로 표시된 모든 기억 항목을 읽고 엄밀히 따라야 함."
- 단점: 기억 공간을 차지하며 Agent가 능동적으로 검색해야 함. 장점: 논리가 완전히 유지되며 가벼운 사용 장면에 적합.

## 2.2 자가 평가: 당신은 이 튜토리얼을 사용하기에 적합한가

### 2.2.1 강력하게 추천되는 사용자 프로필

- **고정밀 협업 사용자**: Agent의 출력에 안정적인 품질 요구(글쓰기 스타일·코드 규범 등)
- **반복 작업 사용자**: Agent와 동일 유형 작업을 자주 수행(매일/매주 유사한 워크플로우)
- **시스템 사고 사용자**: 전기 구축 비용을 투자하여 장기 협업 효율을 얻을 의지
- **경계가 명확한 사용자**: 선호와 레드라인을 명확히 표현할 수 있으며, Agent가 실수할 때 지적할 의향

### 2.2.2 적합하지 않은 사용자

- 일회성·재사용 필요성이 없는 랜덤 질의응답만 수행
- 사용 장면이 매우 넓고 안정적인 규율로 귀납되기 어려움
- Agent와 장기 협업 관계를 맺기를 원하지 않음

## 2.3 시작 전 준비 작업

정식 배포 전, 다음 네 가지 준비를 완료하면 후속 조정 비용이 크게 줄어듭니다.

### 2.3.1 현재 Agent 상태 파악

다음 두 명령을 실행하여 현재 상태를 기록하고 기준선(baseline)으로 삼으십시오:

```
현재 보유한 모든 skill과 각 skill의 내용 요약을 나열하십시오.
```

```
memory에 현재 저장된 모든 내용을 나열하고, 사용률을 평가하십시오.
```

**기록 요점**:
- 현재 skill이 몇 개인가? 여전히 사용 중인가?
- memory 사용률이 몇 퍼센트인가? 내용이 이미 혼재되었는가?
- 오래된 정보가 많이 있는가?

### 2.3.2 당신이 고정하고 싶은 선호와 규칙 나열

간단한 테이블로 Agent가 장기적으로 준수하기를 바라는 원칙을 나열하십시오. 이 단계는 모호한 "느낌"에서 작업 가능한 규칙으로 추출하는 데 도움이 됩니다:

| 차원 | 당신의 원칙 | 예시 |
| :------- | :--------------------- | :--------------------- |
| 커뮤니케이션 스타일 | 간결하고 직접, 빈 말을 쓰지 않음 | "도움이 드리길 바랍니다"를 출력하지 않음 |
| 출력 형식 | 코드 우선, 검증 방법 포함 | 방안 출력 시 테스트 단계 포함 |
| 의사결정 선호 | 리스크 우선, 먼저 문제 노출 | 먼저 리스크 나열 후 이익 논의 |
| 미학적 담향 | 콜드한 서사, AI 요약 거부 | 소설 결말에 여백, 승화하지 않음 |

한 번에 완전할 필요 없으며, 4-6개 핵심 원칙이면 충분합니다. 이후 Agent가 폐루프에서 지속적으로 채워줄 것입니다.

### 2.3.3 3-5개 자주 수행하는 작업 유형 회상

- 예: 단편 소설 작성 / 코드 리뷰 / 운영 현황 확인 / 아키텍처 설계 / 데이터 분석
- 각 유형마다 한 문장으로 "해당 작업을 올바르게 하는 핵심 규율"을 기술하여보십시오

**예시**:
- 단편 작문: "정서적 정조가 줄거리보다 우선하며, 결말은 반드시 여백을 남겨야 합니다."
- 코드 리뷰: "먼저 아키텍처 문제를 찾고, 다음 명명 규범을 보며, 유형별 문제마다 수정 제안을 제시합니다."
- 문제 트러블슈팅: "먼저 로그로 근본 원인(Root Cause)을 찾고, 착각하지 않으며, 불확실할 때는 '검증 필요'로 표시합니다."

이것들이 곧 당신의 L3 장면 핵심 규율 초기 내용이 됩니다.

### 2.3.4 현재 상태 백업

배포 시작 전, **현재 기억과 스킬 상태를 백업하십시오.** 구체적 방법은 플랫폼에 따라 다릅니다:

- 플랫폼이 export를 지원하면 먼저 내보내기
- 지원하지 않으면 수동으로 현재 memory와 skill 목록을 로컬 문서에 복사
- 버전 관리가 있다면(일부 플랫폼의 기억 스냅샷 등), 복원 지점 생성

이 단계의 목적은 배포 과정에서 혼란이 발생할 경우, 초기 상태로 복원하여 다시 시작할 수 있게 하는 것입니다.

## 2.4 준비 완료의 표시

다음 세 가지 질문을 답할 수 있을 때, 준비 단계가 완료되며 제3장으로 진행할 수 있습니다:

1. 내 Agent 플랫폼은 지속 가능한 기억 + 스킬 모듈을 갖추고 있는가? (예 / 아니오 / 부분적 보유)
2. 나는 몇 가지 핵심 선호와 원칙을 도출했는가? (권장 ≥4개)
3. 나는 몇 개 고빈도 작업 유형을 열거하고, 각 유형에 대해 "판단"을 했는가? (권장 ≥3개)
