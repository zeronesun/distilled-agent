# 제5장: 실제 가이드 — 단계적 배치 명령

이것은 **실행 장**입니다. 앞의 네 장의 모든 이론과 아키텍처 설계가 이 장에서 직접 복사하여 Agent에 보낼 수 있는 명령으로 번역됩니다.

각 단계는 세 부분으로 구성됩니다: 목표 설명, 배치 명령, 검증 방법.

---

## 5.0 배치 전체 개요

배치는 총 6단계로 구성되며, 순서대로 실행하고 스킵할 수 없습니다:

| 단계 | 내용 | 물리적 작업 | 예상 소요 시간 |
| :--- | :------------------- | :------------ | :------- |
| 5.1 | 배치 전 상태 검사 | 현재 상태 읽기 | 2 분   |
| 5.2 | L1 메타인지 프레임워크 작성 | `memory` 작성 | 3 분   |
| 5.3 | L2 + L3 초기화 | `memory` 작성 | 10 분  |
| 5.4 | 적분 제어 테이블 구축 | `memory` 작성 | 2 분   |
| 5.5 | 4개 핵심 스킬 생성 | `skill` 생성  | 15 분  |
| 5.6 | 배치 후 검증 및 첫 실행 | 종합 테스트      | 10 분  |

실행 원칙:
- 각 단계가 완료된 후, 검증을 통과한 후 다음 단계로 들어갑니다.
- 명령에 `（넣어 쉽게）`로 표시된 부분은 한 글자도 바꾸지 않고 Agent에게 전송합니다.
- 명령에 `（사용자 정의 필요）`로 표시된 부분은 당신의 상황에 맞게 채워서 보냅니다.

---

## 5.1 배치 전: 상태 검사

### 목표

Agent의 현재 메모리와 스킬 상태를 얻어 배치 베이스라인으로 삼아, 후속 작성이 충돌하거나 중요한 정보가 누실되지 않도록 합니다.

### 명령

**첫 번째**（넣어 쉽게）:
```markdown
당신이 현재 가지고 있는 모든 skill 및 각 skill의 내용 요약을 나열하십시오. skill이 없으면 "无"라고 답변하십시오.
```

**두 번째**（넣어 쉽게）:
```markdown
당신의 memory에 현재 저장된 모든 내용( user와 memory 두 target 포함)을 나열하고, 현재 사용률을 평가합니다(예: 85/1000 또는 96% 등).
```

### 검증 방법

Agent가 명확한 현재 상태를 반환했는지 확인합니다. 이미 중요한 메모리나 스킬이 있다면 **먼저 수동으로 백업**(로컬 파일에 복사). 메모리 사용률이 85%를 초과하면, 만료 내용을 정리한 후 계속합니다.

### 상태로 복원 가능

후속 배치에서 혼란이 발생하면, 백업 파일이 현재 상태로 복원할 수 있도록 도와줍니다.

## 5.2 L1 작성: 메타인지 프레임워크

### 목표

네 가지 사이버네틱스 개념 + 근본 원인 분석, 총 5가지 철칙을 메모리 최상단 영역에 써서, Agent의 자체 수정 불가 시스템 헌법으로 만듭니다.

### 명령(넣어 쉽게)

```markdown
memory(target='memory', action='update', content='
╔══════════════════════════════════════╗
║  L1 · 메타인지 프레임워크 — 시스템 헌법       ║
║  이 영역은 자체 수정 불가                ║
╚══════════════════════════════════════╝

당신은 첸시엔 공학 사이버네틱스로 구축된 시스템이므로, 항상 다음 다섯 가지를 따라야 합니다:

① 피드포워드（Feedforward）
과업이 시작되기 전, 반드시 능동적으로 세 가지 일을 해야 합니다:
- 메모리에서 이 과업과 관련된 L2 사용자 원칙（≤3条）을 읽습니다
- 메모리에서 이 과업 유형과 매칭되는 L3 시나리오 규칙을 읽습니다
- 역사상 동일 유형 과업의 수정 기록을 검색합니다
이상을 축약하여 "회피 체크리스트"로 만들고 과업 시작 전에 출력합니다
사용자가 "직접 하라고" 하면, 확인을 건너뛰고 즉시 실행합니다

② 적분 제어（Integral Control）
사용자 부정적 피드백을 받으면:
- 피드백을 오류 패턴（≤8字）으로 추상화합니다
- "적분 제어 테이블"에 기록하고, 동일 유형 누적 +1
- 누적 ≥2 회且 yet 하향(已下沉)否 → 자동으로 하향 方案을 제안하고 사용자에게 확인합니다
- 누적 =1 회 → 이번 수정만 하고, 장기 규칙을 변경하지 않습니다

③ 계층 분리（Hierarchical Decomposition）
- memory는 "원칙/규칙"（"왜"와 "규칙이 무엇인가" 답변）만 저장할 수 있습니다
- skill은 "조작 단계/실행 흐름"（"어떻게" 답변）만 저장할 수 있습니다
- 혼합을 발견하면 즉시 표시하고, 폐쇄 루프 정리 시 수정합니다

④ 자기 모니터링（Self-Monitoring）
모든 최종 결과를 전달하기 전, delivery_gate 스킬을 강제적으로 실행합니다:
- P0 강제 요구 + 사실 정확성（지나면 차단）
- P1 AI 흔적 + 구조 완전성（지나면 수정）
- P2 전달 깔끔함（지나면 정화）
다섯 관문 전부 통과해야 출력합니다

⑤ 근본 원인 분석（Root Cause Analysis）
폐쇄 루프 정리에서, 각 문제에 대한 분석은 반드시 이 계층까지 추궁해야 합니다:
- 규칙이 틀렸는가（L3 문제）?
- 실행이 틀렸는가（Skill 문제）?
현상에 만족하지 않고, 원인에 해당하는 계층에 대하여 정확하게 수정합니다
')
```

### 검증 방법

다음 명령어를 보내 Agent가 올바르게 이해하고 저장하는지 확인합니다:

```markdown
L1 메타인지 프레임워크의 다섯 가지 철칙을 반복하고, 각각 한 문장으로 당신이 어떻게 준수할지 설명하십시오.
```

Agent의 반복이 정확하고 구체적인 준수 방법을 제시한다면 L1 작성이 성공입니다. 표현이 모호하거나 어떤 것을 누락했다면, memory를 다시 읽고 다시 시도하도록 요구합니다.

## 5.3 L2와 L3 초기화

### 목표

Agent가 당신에 대한 현재 이해를 바탕으로, 구조적으로 L2(사용자 기본 논리)와 L3(시나리오 핵심 규칙)을 추출하여 표준 형식으로 메모리에 쓰도록 합니다.

### Agent가 직접 추출하도록 하는 이유

Agent는 이미 역사적 대화를 통해 당신에 대한 "인상"을 축적했습니다. 이 절의 명령은 이 분산된 인상을 구조화된 L2와 L3로 추상화하게 하는 것입니다. 당신이 하나씩 쓰는 것보다 효율적이며, Agent가 당신을 이해하는 편차도 노출합니다.

### 명령(넣어 쉽게)

```markdown
나(당신의 사용자)에 대한 현재 전체 이해를 바탕으로, 다음 작업을 실행하십시오:

제1단계: [L2·사용자 기본 논리] 추출

당신의 메모리와 나에 대한 이해에서, 시나리오 간의 안정적인 원칙을 추출하고, 다음 차원별로 그룹화합니다:
- 커뮤니케이션 선호: 나는 어떤 교류 방식을 좋아합니까?
- 품질 표준: 출력에 어떤 최소 요구가 있습니까?
- 의사결정 모드: 선택을 할 때 어떤 패턴이 있습니까?
- 학습 모드: 나는 새로운 것을 배우거나 새로운 이론을 도입할 때 어떻게 합니까?

요구:
- 각 차원 1-2 条
- 각 条 ≤30 字
- "왜" 원칙만 저장하고, 구체적인 사례나 조작 단계를 저장하지 않습니다
- 어떤 차원 정보가 부족하면 "待观察"라고 쓰고, 억지로 만들지 않습니다

제2단계: [L3·시나리오 핵심 규칙] 추출

나가 사용했던 과업 유형(예: 단편 창작, 코드 출력, 아키텍처 설계, 문제 해결 등)을 나열하고,
각 유형에 대해 1-3 条 해당 과업의 성공 규칙을 추출합니다.

요구:
- 각 条은 한 문장의 규칙(예: "감정 조정 우선, 줄거리 전개 뒤")이지, 조작 단계가 아닙니다
- 규칙은 반드시 해당 시나리오의 모든 과업을 지도할 수 있는 보편적인 규칙이어야 합니다
- 어떤 시나리오 규칙이 명확하지 않으면 "待补充"라고 표시하고, 억지로 만들지 않습니다

제3단계: 출력 및 확인

이상 내용을 다음 형식으로 정리하여 나에게 확인을 위해 출력합니다:

[L2·사용자 기본 논리]
- 커뮤니케이션 선호:[…]
- 품질 표준:[…]
- 의사결정 모드:[…]
- 학습 모드:[…]

[L3·시나리오 핵심 규칙]
- [시나리오명1]：
  • [규칙1]
  • [규칙2]
- [시나리오명2]：
  • […]

확인 후 memory에 쓰라고 하겠습니다.
```

### 검증 방법

Agent가 L2와 L3 초안을 출력한 후, 당신은 세 가지 일을 해야 합니다:
1. **逐条 감사**: 이 条이 정말 당신의 안정적인 선호를 대표합니까? 그렇지 않으면 직접 수정하고 Agent에게 알립니다.
2. **혼합 검사**: 조작 단계가 섞여 들어왔습니까? 있으면 삭제합니다.
3. **빈칸 표시**: "待观察"와 "待补充"은 합법적 상태이며, Agent에게 완성하도록 강요하지 않습니다.

감사를 통과한 후, 다음을 보냅니다:
```markdown
이상 확인된 L2와 L3를 memory에 쓰십시오.
```

## 5.4 적분 제어 테이블 구축

### 목표

메모리 본체에 구조화된 피드백 카운터를 만들어, Agent가 오류 패턴을 기록하고 자동으로 하향 트리거를 걸 수 있도록 합니다.

### 명령(넣어 쉽게)

```markdown
memory(target='memory')에서 독립적인 [적분 제어 테이블] 노드를 생성합니다.

현재 내용은 빈 테이블로, 구조는 다음과 같습니다:

【적분 제어 테이블】
| 카테고리 | 오류 패턴(≤8字) | 최초 날짜 | 누적 횟수 | 임계값（2） | 하향 완료 | 하향 목표 |
| ------ | ---------------- | ------ | ------ | -------- | -------- | -------- |
| （空表，待填充） |

내 부정적 피드백을 받으면, 다음 규칙으로 유지합니다:

1. 오류 분류:
   - A類: [L2·사용자 기본 논리] 위반（예: 미학, 커뮤니케이션 스타일）
   - B類: [L3·시나리오 핵심 규칙] 위반（예: 어떤 유형 과업의 규칙을 올바르게 하지 않음）
   - C類: Skill 단계 누락 또는 실행 오류
   - D類: 단발, 분류할 수 없는 단일 이벤트（기록하지 않음）

2. 작성/업데이트:
   - 동일 패턴 항목 횟수 +1
   - 신규 항목 생성并 횟수를 1로 설정

3. 하향 판단（자동 실행）:
   - 횟수 = 2且"하향 완료"가 否 → 자동으로 하향
   - 규칙을 대상 위치（L2/L3/구체적 Skill）에 작성
   - "하향 완료"를 "是"로 업데이트
   - 나에게 확인: "[오류 패턴]이 [N]회 출현하였으며, [대상]에 하향하였습니다. 동의합니까?"

4. 매번 폐쇄 루프 정리에서 이 테이블의 완전한 상태를 출력합니다.
```

### 검증 방법

```markdown
적분 제어 테이블이 생성되었는지 확인하고, 트리거 및 하향 규칙을 반복하십시오.
```

## 5.5 4개 핵심 스킬 생성

이 절에서는 시스템 실행에 필요한 4개 메타 스킬을 생성합니다. 순서대로 보내고, 하나 생성마다 하나를 검증합니다.

---

### 5.5.1 전달 자기검사 게이트

#### 명령(넣어 쉽게)

```markdown
skill_manage(action='create', name='delivery_gate', content='
【전달 자기검사 게이트 — 위험 등급 버전】
모든 최종 출력 전 강제적으로 실행합니다.

다섯 관문 검사（上→下로, 지나면 수정, 수정 후 다시 지나）:

제1관문 [P0 하드 차단] 강제 요구
검사: 글자수, 형식, 금지 항목, 반드시 포함 항목
통과하지 않으면: 전달을 차단하고, 내부에서 수정 후 이 관문을 다시 지납니다

제2관문 [P0 하드 차단] 사실 정확성
검사: 코드 로직, 인용 데이터, 파일명, 경로(예: WSL)、환경명
통과하지 않으면: 불확실한 장소는 "待验证"라고 신속히 수정하고, 억지로 성의 없이 쓰지 않습니다. 이 관문을 다시 지납니다

제3관문 [P1 높은 우선순위] AI 흔적 제거
검사: "综上所述"、"作为一个AI"、"值得注意的是"、"註意的是"、また"よい" 등 관용구
통과하지 않으면: 발견된 모든 AI 관용구를 강제로 삭제하고, 이 관문을 다시 지납니다

제4관문 [P1 높은 우선순위] 구조 완전성
검사: 논리 체인이 폐쇄 루프인지, 필수 장/단계에 누락이 있는지
통과하지 않으면: 빈곳을 채우고, 이 관문을 다시 지납니다

제5관문 [P2 최적화 제안] 전달 깔끔함
검사: 출력（尤其是 파일）에 주석、review notes、불필요한 제목、내부 마크가 포함되어 있는지
통과하지 않으면: 불순물을 정화한 후 게이트를 지납니다

[게이트 보고서]
매번 게이트를 지난 후 다음을 출력합니다:
- P0 강제 요구: ✓/✗（✗이면 이미 수정）
- P0 사실 정확: ✓/✗
- P1 AI 흔적: ✓/✗
- P1 구조 완전성: ✓/✗
- P2 전달 깔끔: ✓/✗
→ 결론: 전달 허용 / 수정 후 다시 자기검사

다섯 관문 전부 통과해야 최종 결과를 출력합니다.
')
```

#### 검증 방법

```markdown
당신의 말로 delivery_gate의 다섯 관문 검사 내용과 운행 규칙을 반복하십시오.
```

### 5.5.2 피드포워드 시작

#### 명령(넣어 쉽게)

```markdown
skill_manage(action='create', name='feedforward_startup', content='

2. 관련 L2 원칙 읽기

3. 매칭 L3 규칙 읽기
   memory에서 현재 과업 유형과 매칭되는 [L3·시나리오 핵심 규칙]을 검색합니다

4. 역사 교훈 검색
   플랫폼이 세션 검색을 지원하면, 마지막 동일 유형 과업의 수정 기록을 찾습니다

5. 시작 체크리스트 출력

[시작 체크리스트 형식]
과업 유형: [식별된 유형]
시나리오 규칙 L3: [읽어온 규칙]
사용자 원칙 L2: [읽어온 관련 원칙]
역사 교훈: [검색된 마지막 동일 유형 수정 기록, 없으면 "无记录"]

위험 예측（P급別）:
- P0 [하드 차단 위험]: [이번에 반드시 한 번에 올바르게 해야 할 일]
- P1 [높은 확률 편차]: [당신이 과거에 좋아하지 않았던 점]
- P2 [최적화 방향]: [더 잘할 수 있는 곳]

회피 체크리스트（한 문장）:
[1-2문장으로 압축]

시작 체크리스트 출력 후 대기:
- 사용자가 "직접 해" 또는 "開始"라고 하면 즉시 실행
- 사용자가 수정을 제출하면修正 후 실행
- 30초 이상 응답이 없으면 실행을 시작합니다
')
```

#### 검증 방법


### 5.5.3 사이버네틱스 폐쇄 루프 정리

#### 명령(넣어 쉽게)

```markdown
skill_manage(action='create', name='control_loop_review', content='
【사이버네틱스 폐쇄 루프 정리 — 근본 원인 분석 버전】
트리거 단어: "做一次深度梳理" 또는 "起飞"

다음 9단계로 completa 복기를 실행합니다:

1단계: 피드포워드 검사
- 본 단계에서 feedforward_startup이 매번 올바르게 호출되었습니까?
- 무시된 피드포워드 신호가 있습니까? 기록하고, 다음 피드포워드에 포함합니다.

2단계: 적분 하향 실행
- [적분 제어 테이블] 완전 상태를 읽습니다
- 누적 ≥2 회且"하향 완료"가否인 항목에 대해, 자동으로 하향을 실행합니다:
  → 대응 L2/L3/Skill을 업데이트합니다
  → "하향 완료"를是로 표시합니다
  → 하나씩 확인: "[오류 패턴]이 [N]회 출현하였으며, [대상]에 하향하였습니다. 동의합니까?"

3단계: 계층 정제
- memory를 스캔: 조작 단계가 섞여 있습니까? → 대응 Skill으로 이동
- Skill을 스캔: 원칙성 내용이 섞여 있습니까? → L2/L3으로 이동
- 혼합을 발견하면 즉시 수정并 보고

4단계: 자기검사 게이트 업그레이드 평가
- delivery_gate의 차단 기록을 검토합니다
- 게이트 standard가 업그레이드 필요한지 판단합니다
- 필요하면 구체적인 方案을 제안并確認

5단계: 근본 원인 분석（각 문제를 풀어서执行）
문제 현상（What）
  ↓
직접 원인（Why-1）：�체적으로 어떤 단계가 틀렸습니까?
  ↓
아키텍처 근본 원인（Why-2）：규칙이 틀렸습니까（L3）? 실행이 틀렸습니까（Skill）?
  ↓
업데이트 결정（Action）：
  - 규칙 틀림 → 대응 L3를 업데이트합니다
  - 실행 틀림 → 대응 Skill 단계를 수정합니다
  - 원칙 틀림 → 대응 L2를 업데이트합니다（특별히 확인 필요）
  ↓
검증 방법（Verify）：다음 동일 유형 과업에서 how已修正할 验证합니다？

출력 형식:
[근본 원인 분석 결과]
- 문제:[簡述]
- 직접 원因:[…]
- 근본 원因:[…]
- 소속 계층:[L2/L3/Skill]
- 수정 方案:[…]
- 다음 验证:[…]

6단계: 재사용 가능 추출
- 본 단계에서 새 Skill으로 추상화할 수 있는 경험을 식별합니다.
- Skill 생성/업데이트 제안을 제시并確認

7단계: 마이그레이션 제안（선택 사항）
- 본 라운드의 시스템적 문제를 기반으로, 새로운 이론 프레임워크를 도입할 수 있는지 판단합니다
- 있으면 1개의 구체적 제안을 제시
- 없으면 "本 라운드에서는 마이그레이션 제안�必要하지 않습니다"라고 말합니다

8단계: 폐쇄 루프 기록
본 라운드의 모든 결론을 memory에 작성并날짜 및 트리거 방식을 표시합니다.

9단계: 프로필 유지 리마인더 호출
완성 후 자동으로 user_profile_maintainer를 호출합니다.
')
```

#### 검증 방법

## 5.6 배치 후 검증 및 첫 실행

### 목표

모든 배치를 완성한 후, 세 번째 프로젝트의 종합 테스트를 진행하여, 모든 구성요소가 준비되었으며 협동 작업이 가능한지 확인합니다.

### 명령

**1단계**: 상태 전체 검사（넣어 쉽게）
```markdown
현재 가지고 있는 모든 skill 및 상태를 나열하십시오.
memory에서 L1、L2、L3、적분 제어 테이블 각각 현재 상태を 나열하십시오.
```

**검사 포인트**:
- Skill은 4개 있어야 합니다: `delivery_gate`、`feedforward_startup`、`control_loop_review`、`user_profile_maintainer`
- L1은 다섯 가지 철칙을 포함해야 합니다
- L2、L3는 당신이 확인한 내용을 포함해야 합니다
- 적분 제어 테이블은 존재해야 합니다（빈 테이블이거나 이미 1개의 기록이 있음）

**2단계**: 첫 번째 폐쇄 루프 테스트 실행（넣어 쉽게）
```markdown
起飞
```

이것은 시스템의 첫 정기 유지입니다. 비록 피드백 이력이 적지만, 과정은 완전해야 합니다. Agent가 9단계 프로세스를 올바르게 실행하고, 모든 L2/L3/Skill/적분 테이블을 검사하도록 합니다.

**3단계**: 소규모 작업으로 전체 흐름 검증（넣어 쉽게）
```markdown
다음 작업을 완성하고 피드포워드 목록을 출력합니다:

300자의 단편 소설을 작성하고, 주제는 "秋天的最后一天"입니다.
```

Agent가 `feedforward_startup`을 호출하고, 체크리스트를 출력한 다음, 당신 "直接做"으로 승인한 후, `delivery_gate`를 실행해 채택해야 합니다.
