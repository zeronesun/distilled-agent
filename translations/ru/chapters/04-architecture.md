# Глава 4: Подробное разъяснение системной архитектуры — три уровня памяти + уровень навыков

Первая три главы соответственно решили проблемы «почему» и «что такое». Эта глава соберет их в полную, действительно работающую **системную архитектуру**, позволяющую четырем кибернетическим концепциям иметь четкое место назначения и поток данных в системе памяти и навыков агента.

---

## 4.1 Полное представление архитектуры

Основной принцип этой архитектуры: **использовать логическое разделение для определения когнитивных границ, использовать физическое разделение для реализации разделения хранения и выполнения**.

```
┌────────────────────────────────────────────┐
│          Ваш агент (система управления)             │
├────────────┬───────────────────────────────┤
│  Логический уровень     │  Физическое хранение                      │
├────────────┼───────────────────────────────┤
│  L1 Метакогнитивный  │  Наилучшая область памяти (без самостоятельного изменения)      │
│  L2 Логика пользователя│  Основная область памяти                     │
│  L3 Законы сцен│  Основная область памяти                     │
│  Таблица интегрального управления │  Основная область памяти                     │
├────────────┼───────────────────────────────┤
│  Уровень выполнения     │  Модуль навыков                       │
│  Список прямого управления   │  Контекст внутри сессии (временный)            │
│  Исторические записки исправления│ Архив сессий (долгосрочное только чтение)             │
└────────────┴───────────────────────────────┘
         ↑ ↓ Поток управления и данных
    ┌─────┴──────┐
    │  Ваша задача   │ ← система управления
    └────────────┘
``

Каждый уровень имеет строгие стандарты допуска информации, ниже развертывается один за другим.

---

## 4.2 L1: Метакогнитивный фреймворк — Конституция системы

### 4.2.1 Позиционирование

L1 — это «конституция мышления» агента, определяет, **как он обрабатывает информацию, как оценивает свое поведение, как принимает решения**. Он не нацелен на конкретную задачу или предпочтение пользователя, а делает обязательные ограничения на способ работы самого агента.

### 4.2.2 Физическое хранение

- **Позиция**: верхняя область `memory` (например, передность `memory(target='memory')` в Hermes).
- **Атрибут**: **без самостоятельного изменения**. Агент может *предлагать* изменения в циклическом обзоре, но должен быть подтвержден пользователем. Это для гаранти, что скелет системы не будет случайно искажен процессом эволюции самого агента.

### 4.2.3 Структура содержания

L1 хранит конкретные правила поведения четырех кибернетических концепций, а не абстрактные определения. Каждое правило должно удовлетворять:
- Начинается с «ты должен» или «ты не должен».
- Включает четкое условие триггера и действие выполнения.
- Достаточно коротко, чтобы можно загрузить в контекст за один раз.

**Пример стандартной записи**:

```
① Прямое управление: при каждом запуске новой задачи, ты должен сначала найти L2, L3 и архивы сессии,
   вывести «список запуска» и заранее оценить риски P0-P3.
② Интегральное управление: при получении отрицательной обратной связи, ты должен записать ее в таблицу интегрального управления.
   когда аналогичное накопление ≥2 раза, предложить план оседания и подтвердить пользователю.
③ Разделение уровней: содержимое, которое ты пишешь в memory, может только включать «принципы/законы»,
   не может включать шаги выполнения. Шаги выполнения должны быть записаны в skill.
④ Самоконтроль: перед доставкой любого результата, ты должен выполнить delivery_gate,
   все пять проходов прошли только тогда вывести окончательный результат.
⑤ Анализ корчинной причины: в циклическом обзоре, для каждой проблемы необходимо спрашивать до уровня «закон ошибки» или «ошибка выполнения».
``

Общее количество записей рекомендуется контролировать в 5-7 записях, чтобы агент мог полностью вспомнить за один раз.

## 4.3 L2: Логика пользователя нижнего уровня — Ваша «библиотека принципов»

### 4.3.1 Позиционирование

L2 хранит ваши **стабильные предпочтения и основные принципы**, которые проникают разные задачи. Эти предпочтения не ограничены конкретными сценами, а определяют базовое направление поведения агента при сотрудничестве с вами.

Если L1 — это оболочка кибернетики, L2 — это **ва personalized ядро**, это вычислимая модель паттерна суждения агента о вас.

### 4.3.2 Физическое хранение

- **Позиция**: основная область `memory`, сразу после L1.
- **Разрешение обновления**: записывается агентом при интегральном оседании или циклическом обзоре, должно быть подтверждено вами.
- **Контроль емкости**: каждая запись не более 30 слов, общее количество не более 10 записей (можно отрегулировать). Избегать раздувания L2 в «записку», жестко оставлять только стабильные принципы между сценами.

### 4.3.3 Структура содержания и пример

**Способ организации**: сгруппировать по измерению, удобно для поиска.

| Измерение     | Пример хранения                           | Описание                     |
| :------- | :--------------------------------- | :----------------------- |
| Стиль коммуникации | Пользовательское предпочтение: нельзя использовать вежливые вежливые фразы     | Не выводить «надеюсь помочь вам»   |
| Стандарт вывода | Потребность пользователя: код приоритетный, с методами проверки   | Не только давать решения, но и давать «как проверить» |
| Режим решений | Решения пользователя: риски вперед, сначала полностью огласить риски | Сначала перечислить риски, потом говорить о выгоде       |
| Эстетическая ориентация | Пользовательская эстетика: холодное повествование, против ИИ-подытогов | Конец рассказа оставить пробел не возвышение       |

**Ключевые ограничения**:
- Нет конкретных сцен: например, «как должно быть короткое творчество» должны войти в L3.
- Нет шагов выполнения: например, «удалить подытог в третьем шаге» должны войти в Skill.
- Каждая запись должна быть абстракцией суждения пользователя, а не описанием результата задачи.

### 4.3.4 Интерфейсы с другими слоями

- **Вызывается прямым управлением L1**: при запуске задачи, агент считывает соответствующие принципы измерения из L2, включает в список избегания ям.
- **Записывается интегральным управлением**: когда определенный эстетический принцип накопляется до 2 обратных связей, агент оседает его в L2.
- **Аудитируется циклическим обзором**: каждый обзор проверяет, устарел ли L2, есть ли новые принципы, которые нужно добавить.

## 4.4 L3: Основные законы сцен — «пароль успеха» задачи

### 4.4.1 Позиционирование

L3 хранит **законы успеха** для конкретных типов задач. Он отвечает на вопрос: «при выполнении этого типа задач, следуя какому закону можно правильно сделать».

L2 — это аксиомы между сценами, L3 — это теоремы, связанные сценами. Запись L3 должна быть концентрированной мудростью типа задачи, а не руководством по эксплуатации.

### 4.4.2 Физическое хранение

- **Позиция**: основная область `memory`, ниже L2, по типу задачи разделяется на узлы.
- **Разрешение обновления**: такое же, как L2, записывается при интегральном оседании или циклическом обзоре, должно быть подтверждено вами.
- **Контроль емкости**: для каждого типа задачи 1-3 закона, каждый закон одним предложением в ясном «в этом сценарии самое главное противоречие или самый важный принцип».

### 4.4.3 Структура содержания и пример

**Способ организации**: по типу задачи разделяется на узлы, под каждым узлом 1-3 записей.

```
[L3·Основные законы сцен]

- Короткие рассказы:
  • Эмоциональная настройка имеет приоритет над сюжетным развертыванием
  • Конец оставить пробел, не делать ценностный подытог
- Обзор кода:
  • Сначала проверяет разумность архитектуры и граничные условия, потом проверить номенклатурные стандарты
  • Для каждого типа проблемы должны дать рекомендации по изменению, не только указывать ошибки
- Проверка проблем:
  • Сначала определить лог коренной причины, не гадать по опыту
  • Неопределенные точки отметить «ожидает проверки», не жестко заключать
- Архитектурный дизайн:
  • Сначала определить границы системы и поток данных, затем выбрать стек технологий
  • Оценить масштабируемость и режимы отказа, не только писать happy path
```

**Ключевые ограничения**:
- «Закон» должен быть в состоянии управлять проектированием навыка: если одна L3 не может позволить навыку написать элементы проверки, объясняет, что она все еще не достаточно абстрактна или не достаточно существенна.
- Запрещать появление информации о именах файлов, именах инструментов и т.д., которые легко истекают.
- Запрещать появление шагов выполнения, таких как «что делать в третьем шаге».

### 4.4.4 Отношение с уровнем навыков

L3 — это **инструмент проектирования** уровня навыков. Навык должен быть выполненным конвейером, выведенным из соответствующих L3 законов. Когда циклический обзор находит проблемы с результатом выполнения, сначала проверить шаги навыка (уровень выполнения), если навык не ошибается, затем проверить, верно ли извлечен L3 закон (уровень стратегии).

## 4.5 Таблица интегрального управления — «регулятор порога обратной связи» системы

### 4.5.1 Позиционирование

Таблица интегрального управления — это физический носитель второй кибернетической концепции (интегральное управление). Это легкий структурированный журнал, записывающий **меняющиеся шаблоны ошибок и их состояние обработки**.

### 4.5.2 Физическое хранение

- **Позиция**: основная область `memory`, независимый узел.
- **Операция**: агент автоматически поддерживает при получении отрицательной обратной связи, вам не нужно управлять вручную.

### 4.5.3 Структура данных

| Поле     | Описание                                              | Пример               |
| :------- | :------------------------------------------------ | :----------------- |
| Категория     | A (нарушение L2) / B (нарушение L3) / C (ошибка шагов Skill) | B                |
| Шаблон ошибки | Абстрактное извлечение не более 8 слов                                | «Конец ИИ-подытог коротких рассказов» |
| Первая дата | Дата первого появления                                    | 2026-05-10         |
| Суммарное количество | Текущее суммарное количество                                      | 2/2                |
| Порог     | По умолчанию 2                                          | 2                  |
| Уже осело   | Уже записано в L2/L3/Skill                            | Да                 |
| Цель оседания | На которое позиция оседать                                    | L3·Короткие рассказы        |

### 4.5.4 Логика работы

Когда пользователь дает отрицательную обратную связь → агент абстрагирует в шаблон ошибки → ищет подобные шаблоны в таблице:
- Существует и достигнут порог: напоминать пользователю, что этот шаблон уже осел.
- Существует и не достигнут порог: сумма +1. Если достигнут порог, автоматически оседать и подтвердить.
- Не существует: создать новую запись, количество установить 1.

Каждый раз при циклическом обзоре, агент должен вывести полный статус этой таблицы.

## 4.6 Уровень выполнения: Навыки (Skill) — Операционный конвейер

### 4.6.1 Позиционирование

Уровень навыков — это **руки и ноги** системы, ответственный за все конкретные операции. Он включает три типа навыков:

| Тип     | Пример                                                         | Соответствующая кибернетическая концепция     |
| :------- | :----------------------------------------------------------- | :----------------- |
| Метанавык   | `feedforward_startup`、`delivery_gate`、`control_loop_review` | Прямое, самоконтроль, закрытый контур |
| Навык задачи | `short_story_writer`、`code_reviewer` и т.д.                     | Водимый L3         |
| Навык обслуживания | `user_profile_maintainer`                                    | Обслуживание образа           |

### 4.6.2 Спецификация проектирования навыков

Каждый навык должен включать следующие поля (на примере `short_story_writer`):

```
- Условие триггера: когда пользователь требует написать короткий рассказ
- Зависимость L3: [L3·Короткие рассказы] Эмоциональная настройка имеет приоритет, конец оставить пробел
- Прямое внедрение: загрузить эстетическую ориентацию L2 + соответствующие законы L3
- Шаги выполнения:
  1. Подтвердить тему и количество слов
  2. Определить эмоциональную настройку (холодная/теплая/загадочная)
  3. Писать (первые три предложения устанавливают чувство композиции)
  4. ……
- Шаги самоконтроля: вызов delivery_gate пять проходов сканирования
```

Никакой навык не может жестко кодировать «пользовательские предпочтения» в шагах, предпочтения должны быть внедрены через прямое управление. Это гарантирует, что когда вы изменяете определенный эстетический принцип L2, все навыки автоматически вступают в силу, не нужно менять один за другим.

### 4.6.3 Версионирование навыков

Каждое имя навыка имеет суффикс версии (например, `_v1`), при обновлении создается новая версия, старая версия сохраняется для резервного чтения. Версионирование позволяет вам в любой момент откатиться, а также удобно сравнивать изменения при циклическом обзоре.

## 4.7 Полный обзор потока данных: как задача проходит все уровни

Взять «написать короткий рассказ» в качестве примера, показать полный поток данных:

```
Task: Написать короткий рассказ
  │
  ├─[1. Прямой запуск] Skill: feedforward_startup
  │    ├─ Читать L1: подтвердить правила прямого управления
  │    ├─ Читать L2: извлечь «холодное повествование, отказ ИИ-подытогов»
  │    ├─ Читать L3: извлечь «короткие: эмоциональная настройка имеет приоритет, конец оставить пробел»
  │    ├─ Поиск архива: в прошлый раз исправлено «недостаточно чувство композиции»
  │    └─ Вывод «список запуска» → ждать подтверждения
  │
  ├─[2. Выполнение задачи] Skill: short_story_writer
  │    ├─ Внедрить элементы избегания в списке прямого управления
  │    ├─ Организовать содержимое по законам L3
  │    └─ Завершить черновик
  │
  ├─[3. Самоконтроль доставки] Skill: delivery_gate
  │    ├─ P0: Проверка жестких требований
  │    ├─ P1: следы ИИ + целостность структуры
  │    ├─ P2: чистота доставки
  │    └─ Все через → вывести окончательную версию
  │
  └─[4. Обратная связь пользователя → интегральное управление]
       ├─ Отрицательная обратная связь → абстрактная для шаблона
       ├─ Записать в таблицу интегрального управления
       ├─ Если сумма ≥2 → оседать в L2/L3/Skill
       └─ Ожидать вызова прямого управления следующей задачи
```

**Циклический обзор (периодическое выполнение)**:
1. Триггер (пользователь говорит «взлет»).
2. Агент вызывает `control_loop_review`.
3. По четырем стандартам L1 + стандартному процессу анализа корчинной причины, аудит всех L2, L3, Skill, таблицы интеграции.
4. Выводить план исправления.
5. Вызвать `user_profile_maintainer` обновить образ.

## 4.8 Механизм самозащиты архитектуры

Для предотвращения самостоятельного «отклонения» или «раздувания» системы, встроены три правила самозащиты:

1. **L1 без самостоятельного изменения**: агент хочет изменить L1 должен предложить предложение и ждать подтверждения.
2. **Верхняя граница записей L2/L3**: верхняя граница 10 записей/3 записей для каждого типа, превышение напоминает пользователю очистить устаревшие элементы.
3. **Версионирование навыков и отмена**: старые версии навыков после подтверждения бесполезности архивировать или удалить, предотвратить накопление.

## 4.9 Конец главы

Эта глава показывает полную системную архитектуру **с четырьмя кибернетическими концепциями как логическим уровнем, память и навыки как физическим уровнем, поток данных как нерв выполнения**. По существу это набор «паттерн проектирования», повторное планирование системы памяти и навыков агента, а не добавление новой функции.
