# Пример вывода инициализации L2/L3

Этот файл представляет типичный диалог инициализации L2/L3. После отправки инструкции кодирования (извлечения), Агент, основываясь на своём историческом понимании пользователя, выводит структурированные черновики L2 и L3.

> **Примечание**: Пользовательские предпочтения в этом примере вымышлены и служат только для иллюстрации формата вывода и степени детализации извлечения. Ваш Агент сгенерирует версию, индивидуальную для вас, основываясь на ваших исторических беседах.

---

## Вывод извлечения от Агента

Основываясь на моём текущем понимании вас, вот результат извлечения:

### [L2·Подвоящая пользовательская логика]

- **Предпочтения общения**: краткий и прямой, без лишних формальностей; чёткие границы, "нет" = остановка
- **Стандарты качества**: код должен сопровождаться методом проверки; вывод должен быть многоразовым, а не просто однажды правильным
- **Режим принятия решений**: предварительная оценка риска, сначала раскрыть полную проблему, затем представить решение; оценка второго порядка важнее интуиции
- **Режим обучения**: междисциплинарная теоретическая миграция; единство знания и действия, понимание должно трансформироваться в действие

### [L3·Фундаментальные правила сценария]

- **Короткое творчество**:
  - Эмоции приоритетнее развития сюжета
  - Оставлять пространство в конце, не делать выводов или нравоучений
  - Создавать образ визуальным описанием, не полагаться на метафоры

- **Вывод кода**:
  - Сначала писать тесты или метод проверки, затем реализацию
  - Функциональная полнота важнее обилия комментариев
  - Отмечать "проверить" для неопределённых поведений, не выдумывать

- **Архитектурное проектирование**:
  - Сначала определить границы системы и поток данных, затем выбирать технологический стек
  - Оценивать расширяемость и режимы отказов, не ограничиваться обычным путём
  - Вывод включает классификацию рисков (P0-P3) и их решения

- **Решение проблем**:
  - Сначала локализовать корневую причину в логах или месте выполнения, не гадать по опыту
  - Отмечать "проверить" для неопределённостей
  - Предоставлять повторяемые этапы решения, не только заключение

## Действия проверки, которые может выполнить пользователь

1. **Подтвержение строка за строкой**: Для каждого принципа спросите себя — "действительно ли это моё устойчивое предпочтение?"
2. **Исправление отклонений**: Скажите Агенту напрямую, какое правило неверное, каким оно должно быть
3. **Восполнение пропусков**: Если вы считаете, что измерение/сценарий пропускает ключевой принцип, скажите напрямую
4. **Удаление ложного**: Если Агент выдумал (у вас нет такого предпочтения), удалите его

## Общие примеры исправлений

| Исходное извлечение Агента | Исправление пользователя | Объяснение |
| :----------------- | :--------------------------------------- | :--------- |
| Предпочтения общения: краткий и прямой | Предпочтения общения: краткий, но с достаточным контекстом | Пользователь обнаружил, что ему нужен контекст, а не только краткость |
| Короткое творчество: оставлять пространство в конце | Короткое творчество: оставлять пространство в конце; но серийные истории могут иметь мягкий финал | Пользователь добавил исключение |
| Вывод кода: тесты в приоритете | Вывод кода: для быстрых прототипов тесты можно добавить после | Пользователь различает стандарты прототипов и формальных выводов |

## Инструкция записи

После проверки:
> Записать подтверждённые выше L2 и L3 в память (memory).

Агент сохранит их в основной области памяти, в отдельных узлах [L2·Подвоящая пользовательская логика] и [L3·Фундаментальные правила сценария].
