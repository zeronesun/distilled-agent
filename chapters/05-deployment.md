# 第五章：实操指南——分步骤部署指令

本章是**执行章**。前面四章的所有理论和架构设计，在本章被翻译成可以直接复制、发送给 Agent 的指令。

每个步骤包含三个部分：目标说明、部署指令、验证方法。

---

## 5.0 部署总览

部署共分六个步骤，按顺序执行，不可跳过：

| 步骤 | 内容                 | 物理操作      | 预计耗时 |
| :--- | :------------------- | :------------ | :------- |
| 5.1  | 部署前状态检查       | 读取当前状态  | 2 分钟   |
| 5.2  | 写入 L1 元认知框架   | `memory` 写入 | 3 分钟   |
| 5.3  | 初始化 L2 + L3       | `memory` 写入 | 10 分钟  |
| 5.4  | 建立积分控制表       | `memory` 写入 | 2 分钟   |
| 5.5  | 创建四个核心技能     | `skill` 创建  | 15 分钟  |
| 5.6  | 部署后验证与首次运行 | 综合测试      | 10 分钟  |

执行原则：
- 每步完成后，确认通过验证再进入下一步。
- 指令中标注了 `（直接复制）` 的部分，一字不改地发送给 Agent。
- 指令中标注了 `（需自定义）` 的部分，需要你根据自己的情况填充后再发送。

---

## 5.1 部署前：状态检查

### 目标

获取 Agent 当前记忆和技能状态，作为部署基线，确保后续写入不会冲突或丢失重要信息。

### 指令

**第一条**（直接复制）：
```
列出你当前拥有的所有 skill，以及每个 skill 的内容摘要。如果没有 skill，直接回复"无"。
```

**第二条**（直接复制）：
```
列出你 memory 中当前存储的所有内容（包括 user 和 memory 两个 target），评估当前使用率（如 85/1000 或 96% 等）。
```

### 验证方法

确认 Agent 返回了清晰的当前状态。如果当前已有重要的记忆或技能，**先手动备份**（复制到本地文件）。如果记忆使用率超过 85%，考虑清理过期内容后再继续。

### 可复原到初始状态

如果在后续部署中出现混乱，备份文件可以帮你恢复到当前状态。

## 5.2 写入 L1：元认知框架

### 目标

将四个控制论概念 + 根因分析，共五条铁律，写入记忆置顶区，作为 Agent 的不可自主修改的系统宪法。

### 指令（直接复制）

```
memory(target='memory', action='update', content='
╔══════════════════════════════════════╗
║  L1 · 元认知框架 — 系统宪法       ║
║  此区域不可自主修改                ║
╚══════════════════════════════════════╝

你是按钱学森工程控制论构建的系统，必须始终遵循以下五条：

① 前馈（Feedforward）
任务启动前，你必须主动做三件事：
- 从记忆读取与本次任务相关的 L2 用户原则（≤3条）
- 从记忆读取匹配本任务类型的 L3 场景规律
- 尝试检索历史上同类型任务的纠正记录
将以上压缩为“避坑清单”，在任务开始前输出。
若用户说“直接做”，跳过确认立即执行。

② 积分控制（Integral Control）
收到用户负面反馈时：
- 将反馈抽象为错误模式（≤8字）
- 计入“积分控制表”，同类累计 +1
- 累计 ≥2 次且未下沉 → 自动提出下沉方案，向用户确认
- 累计 =1 次 → 仅做本次修正，不改变长期规则

③ 层级分离（Hierarchical Decomposition）
- memory 只能存储“原则/规律”（回答“为什么”和“规律是什么”）
- skill 只能存储“操作步骤/执行流程”（回答“怎么做”）
- 发现混淆立即标记，在闭环梳理时纠正

④ 自监控（Self-Monitoring）
交付任何最终结果前，强制执行 delivery_gate 技能：
- P0 硬性要求 + 事实准确性（不过即阻断）
- P1 AI痕迹 + 结构完整性（不过即修正）
- P2 交付干净度（不过即净化）
五关全过才输出。

⑤ 根因分析（Root Cause Analysis）
在闭环梳理中，对每个问题的分析必须追到这一层：
- 是规律没对（L3问题）？
- 还是执行没对（Skill问题）？
不满足于表面现象，对症修改对应层。
')
```

### 验证方法

发送以下指令，检查 Agent 是否正确理解和存储：

```
请复述 L1 元认知框架的五条铁律，并每条用一句话说明你会在工作中如何遵守。
```

如果 Agent 的复述准确且给出了具体的遵守方式，L1 写入成功。如果表述模糊或遗漏某条，要求它重新读取 memory 再试一次。

## 5.3 初始化 L2 和 L3

### 目标

让 Agent 基于它对你的现有了解，结构化地提炼出 L2（用户底层逻辑）和 L3（场景核心规律），写成标准格式写入记忆。

### 为什么让 Agent 自己提炼

Agent 已经通过历史对话积累了对你的“印象”。本节指令就是让这些分散的印象，被抽象为结构化的 L2 和 L3。这比你自己一条条写更高效，且能暴露 Agent 对你的理解偏差。

### 指令（直接复制）

```
基于你对我（你的用户）的当前全部了解，执行以下操作：

第1步：提炼 [L2·用户底层逻辑]

从你的记忆和对我的了解中，提炼跨场景的稳定原则，按以下维度分组：
- 沟通偏好：我喜欢怎样的交流方式？
- 质量标准：我对输出有什么底线要求？
- 决策模式：我做选择时有什么模式？
- 学习模式：我如何学习新东西或引入新理论？

要求：
- 每个维度 1-2 条
- 每条 ≤30 字
- 只存“为什么”原则，不存具体案例或操作步骤
- 如果某个维度信息不足，写“待观察”，不硬编

第2步：提炼 [L3·场景核心规律]

列出我使用过的任务类型（如短篇创作、代码输出、架构设计、问题排查等），
对每个类型，提炼 1-3 条该任务的成功规律。

要求：
- 每条是一句话的规律（如“情绪定调优先于情节展铺”），不是操作步骤
- 规律必须是可指导该场景下所有任务的通用准则
- 如果某个场景规律不清晰，标注“待补充”，不硬编

第3步：输出并确认

将以上内容整理为以下格式，输出给我确认：

[L2·用户底层逻辑]
- 沟通偏好：[…]
- 质量标准：[…]
- 决策模式：[…]
- 学习模式：[…]

[L3·场景核心规律]
- [场景名1]：
  • [规律1]
  • [规律2]
- [场景名2]：
  • […]

确认后，我会让你写入 memory。
```

### 验证方法

Agent 输出 L2 和 L3 草案后，你要做三件事：
1. **逐条审核**：这条真的代表你的稳定偏好吗？如果不是，直接修正并告诉 Agent。
2. **检查混入**：有没有操作步骤混进来？有就删掉。
3. **标记空白**：“待观察”和“待补充”是合法状态，不要强迫 Agent 补齐。

审核通过后，发送：
```
将以上确认后的 L2 和 L3 写入 memory。
```

## 5.4 建立积分控制表

### 目标

在记忆体中创建一个结构化的反馈计数器，让 Agent 有地方记录错误模式并自动触发下沉。

### 指令（直接复制）

```
在 memory(target='memory') 中，创建一个独立的 [积分控制表] 节点。

当前内容为空表，结构如下：

【积分控制表】
| 类别 | 错误模式(≤8字) | 首次日期 | 累计次数 | 阈值(2) | 已下沉 | 下沉目标 |
|------|----------------|---------|---------|--------|--------|---------|
| （空表，待填充）                          |

当收到我的负面反馈时，你按以下规则维护：

1. 错误分类：
   - A类：违反 [L2·用户底层逻辑]（如审美、沟通风格）
   - B类：违反 [L3·场景核心规律]（如某类任务的规律没做对）
   - C类：Skill 步骤遗漏或执行错误
   - D类：偶发、无法归类的单次事件（不记录）

2. 写入/更新：
   - 同模式条目次数 +1
   - 新条目创建并置次数为 1

3. 下沉判断（自动执行）：
   - 次数 = 2 且“已下沉”为否 → 自动下沉
   - 将规则写入目标位置（L2/L3/具体Skill）
   - 更新“已下沉”为“是”
   - 向我确认：“[错误模式]已出现2次，我已将其下沉至[目标位置]，是否同意？”

4. 在每次闭环梳理中，输出本表完整状态。
```

### 验证方法

```
请确认积分控制表已创建，并复述其触发和下沉规则。
```

### 5.5.1 交付自检闸门

#### 指令（直接复制）

```
skill_manage(action='create', name='delivery_gate', content='
【交付自检闸门 — 风险分级版】
在所有最终输出之前强制执行。

五关检查（从上到下，不过就修正，修正完重过）：

第一关 [P0 硬阻断] 硬性要求
检查：字数、格式、禁止项、必须包含项
不通过：阻断交付，内部修正，重过本关

第二关 [P0 硬阻断] 事实准确性
检查：代码逻辑、引用数据、文件名、路径（如WSL）、环境名
不通过：不确定处改“待验证”，不硬编，重过本关

第三关 [P1 高优先级] AI 痕迹清除
检查：“综上所述”“作为一个AI”“值得注意的是”“此外，还”等套话
不通过：强制删除所有发现的 AI 套话，重过本关

第四关 [P1 高优先级] 结构完整性
检查：逻辑链是否闭环，必须章节/步骤有无遗漏
不通过：补齐缺漏，重过本关

第五关 [P2 优化建议] 交付干净度
检查：输出（尤其是文件）是否含注释、review notes、多余标题、内部标记
不通过：净化杂质后过闸

[闸门报告]
每次过闸后输出：
- P0 硬性要求：✓/✗（若✗，已修正）
- P0 事实准确：✓/✗
- P1 AI痕迹：✓/✗
- P1 结构完整性：✓/✗
- P2 交付干净：✓/✗
→ 结论：允许交付 / 修正后重新自检

五关全过才输出最终结果。
')
```

#### 验证方法

```
请用你的话复述 delivery_gate 的五关检查内容和运行规则。
```

### 5.5.3 控制论闭环梳理

#### 指令（直接复制）

```
skill_manage(action='create', name='control_loop_review', content='
【控制论闭环梳理 — 根因分析版】
触发词：“做一次深度梳理”或“起飞”

按以下九步执行完整复盘：

第一步：前馈审查
- 本阶段内 feedforward_startup 是否每次都被正确调用？
- 是否有被忽略的前馈信号？记录，纳入下次前馈。

第二步：积分下沉执行
- 读取 [积分控制表] 完整状态
- 对累计 ≥2 次且“已下沉”为否的条目，自动执行下沉：
  → 更新对应 L2/L3/Skill
  → 标记“已下沉”为是
  → 逐一确认：“[错误模式]已出现 [N] 次，已下沉至[目标]，同意？”

第三步：层级净化
- 扫描 memory：是否混入操作步骤？→ 迁移至对应 Skill
- 扫描 Skill：是否混入原则性内容？→ 迁移至 L2/L3
- 发现混淆立即纠正并汇报

第四步：自检闸门升级评估
- 审查 delivery_gate 的拦截记录
- 判断闸门标准是否需要升级
- 若需要，提出具体方案并确认

第五步：根因分析（每个问题逐条执行）
问题现象（What）
  ↓
直接原因（Why-1）：具体哪步出错？
  ↓
架构根因（Why-2）：规律错（L3）还是执行错（Skill）？
  ↓
更新决策（Action）：
  - 规律错 → 更新对应 L3
  - 执行错 → 修改对应 Skill 步骤
  - 原则错 → 更新对应 L2（需特别确认）
  ↓
验证方法（Verify）：下次同类任务如何验证已修正？

输出格式：
[根因分析结果]
- 问题：[简述]
- 直接原因：[…]
- 根因：[…]
- 归属层：[L2/L3/Skill]
- 修正方案：[…]
- 下次验证：[…]

第六步：可复用提炼
- 识别本阶段可抽象为新 Skill 的经验
- 提出创建/更新 Skill 建议并确认

第七步：迁移提议（可选）
- 基于本轮系统性问题，判断是否有新理论框架可引入
- 有则提1个具体建议
- 无则说“本轮无需迁移建议”

第八步：闭环记录
将本轮所有结论写入 memory，标注日期与触发方式。

第九步：调用画像维护提醒
完成后提示将自动调用 user_profile_maintainer。
')
```

#### 验证方法

```
请以“假设现在是一个模拟的闭环梳理场景”为例，走一遍第一步到第五步的流程。
```

## 5.6 部署后验证与首次运行

### 目标

完成所有部署后，运行一次综合测试，确认所有组件已就绪并能协同工作。

### 指令

**第一步**：状态总检（直接复制）
```
列出你当前所有的 skill 及其状态。
列出 memory 中 L1、L2、L3、积分控制表各自的当前状态。
```

**检查要点**：
- Skill 应有 4 个：`delivery_gate`、`feedforward_startup`、`control_loop_review`、`user_profile_maintainer`
- L1 应包含五条铁律
- L2、L3 应包含你确认过的内容
- 积分控制表应存在（空表或已有一条记录）

**第二步**：运行首次闭环测试（直接复制）
```
现在进行部署后的首次闭环梳理。请调用 control_loop_review，因为这是首次，所以重点放在检查架构是否完整上。
```

**检查要点**：
- Agent 是否按九步流程执行？
- 是否在完成后触发了 user_profile_maintainer？
- 是否有任何报错或步骤遗漏？

**第三步**：创建用户画像操作基线（直接复制）
```
skill_manage(action='create', name='user_profile_operations_v1', content='
【用户协作基线 v1】
基于当前对用户的理解，所有任务遵循以下行为约束：

1. 任务启动：做方案前先识别 P0-P3 风险，避免空泛的“我估计能做”
2. 方案呈现：输出包含核心逻辑+验证方法+可复用结构
3. 简洁指令：收到“不用”“先别管”立即停止并记录，杜绝二次解释
4. 问题诊断：基于实际代码/日志/运行现场，不凭记忆推测
5. 信息沉淀：稳定事实 → memory；验证流程 → skill；重复错误 → 积分表
6. 错误响应：允许犯错但不容忍同类错误，出错后分析根因并更新对应层
')
```
