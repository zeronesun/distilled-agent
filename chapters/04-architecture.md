# 第四章：系统架构详解——三层记忆体 + 技能层

前三章分别解决了“为什么”和“是什么”的问题。本章将它们组装成一个完整的、可实际运行的**系统架构**，让四个控制论概念在 Agent 的记忆和技能体系中有明确的落位和数据流向。

---

## 4.1 架构全景

本架构的核心原则是：**用逻辑分层定义认知边界，用物理分层实现存储与执行分离**。

```
┌────────────────────────────────────────────┐
│          你的 Agent（被操纵系统）             │
├────────────┬───────────────────────────────┤
│  逻辑层     │  物理存储                      │
├────────────┼───────────────────────────────┤
│  L1 元认知  │  记忆置顶区（不可自主修改）      │
│  L2 用户逻辑│  记忆主体区                     │
│  L3 场景规律│  记忆主体区                     │
│  积分控制表 │  记忆主体区                     │
├────────────┼───────────────────────────────┤
│  执行层     │  技能模块                       │
│  前馈清单   │  会话内上下文（临时）            │
│  历史纠正记录│ 会话存档（长期只读）             │
└────────────┴───────────────────────────────┘
         ↑ ↓ 控制流与数据流
    ┌─────┴──────┐
    │  你的任务   │ ← 被控制系统
    └────────────┘
```

每一层都有严格的信息准入标准，下面逐一展开。

---

## 4.2 L1：元认知框架——系统宪法

### 4.2.1 定位

L1 是 Agent 的“思维宪法”，定义了它**如何处理信息、如何评估自身行为、如何做决策**。它不针对任何具体任务或用户偏好，而是对 Agent 本身的运行方式做出强制约束。

### 4.2.2 物理存储

- **位置**：`memory` 的置顶区域（如 Hermes 的 `memory(target='memory')` 最前端）。
- **属性**：**不可自主修改**。Agent 可以在闭环梳理中*建议*修改，但必须经用户确认。这是为了确保系统骨架不会被 Agent 自身的进化过程意外扭曲。

### 4.2.3 内容结构

L1 存储的是四个控制论概念的具体行为准则，而非抽象定义。每条准则必须满足：
- 以“你必须”或“你不得”开头。
- 包含明确触发条件和执行动作。
- 短到可以一次性加载进上下文。

**标准条目示例**：

```
① 前馈：每次新任务启动时，你必须先检索 L2、L3 和会话存档，
   输出“启动清单”并预判 P0-P3 风险。
② 积分控制：收到负面反馈时，你必须将其计入积分控制表。
   同类累计 ≥2 次时，提出下沉方案并向用户确认。
③ 层级分离：你写入 memory 的内容，只能包含“原则/规律”，
   不得包含操作步骤。操作步骤必须写入 skill。
④ 自监控：交付任何结果前，你必须执行 delivery_gate，
   五关全过才输出最终结果。
⑤ 根因分析：在闭环梳理中，对每个问题必须追问到“规律错”还是“执行错”层。
```

条目总数建议控制在 5-7 条内，保证 Agent 能一次性完整回忆。

---

## 4.3 L2：用户底层逻辑——你的“原则库”

### 4.3.1 定位

L2 存储你跨越不同任务的**稳定偏好和核心原则**。这些偏好不限于特定场景，而是定义了 Agent 与你协作时的基本行为方向。

如果说 L1 是控制论外壳，L2 就是**你的个性化内核**，是 Agent 对你判断模式的可计算模型。

### 4.3.2 物理存储

- **位置**：`memory` 主体区，紧接 L1 之后。
- **更新权限**：由 Agent 在积分下沉或闭环梳理时写入，须经你确认。
- **容量控制**：每条不超过 30 字，总数不超过 10 条（可调整）。避免 L2 膨胀成“备忘录”，强制只保留跨场景的稳定原则。

### 4.3.3 内容结构与示例

**组织方式**：按维度分组，便于检索。

| 维度     | 存储示例                           | 说明                     |
| :------- | :--------------------------------- | :----------------------- |
| 沟通风格 | 用户偏好：简洁直接，不用客套话     | 不输出“希望能够帮到您”   |
| 输出标准 | 用户需求：代码优先，附带验证方法   | 不只给方案，要给“怎么验” |
| 决策模式 | 用户决策：风险前置，先暴露完整风险 | 先列风险，再谈收益       |
| 审美取向 | 用户审美：冷感叙事，拒绝 AI 式总结 | 小说结尾留白不升华       |

**关键约束**：
- 不含具体场景：如“短篇创作应该怎样”这类应进 L3。
- 不含操作步骤：如“在第三步删掉总结”这类应进 Skill。
- 每条必须是对用户判断的抽象，而非对任务结果的描述。

### 4.3.4 与其他层的接口

- **被 L1 前馈调用**：任务启动时，Agent 从 L2 读取相关维度的原则，融入避坑清单。
- **被积分控制写入**：当某条审美原则累积到 2 次反馈，Agent 将其下沉到 L2。
- **被闭环梳理审计**：每次梳理检查 L2 是否过时、是否有新原则需要加入。

---

## 4.4 L3：场景核心规律——任务“成功密码”

### 4.4.1 定位

L3 存储特定任务类型的 **成功规律**。它回答的是：“做这类任务时，遵循什么规律才能做对。”

L2 是跨场景的“公理”，L3 是场景相关的“定理”。一个 L3 条目应该是一个任务类型的浓缩智慧，而非操作手册。

### 4.4.2 物理存储

- **位置**：`memory` 主体区，L2 下方，按任务类型分节点。
- **更新权限**：与 L2 相同，由积分下沉或闭环梳理写入，须经你确认。
- **容量控制**：每个任务类型 1-3 条规律，每条用一句话说清“这个场景下最核心的矛盾或最关键的原则”。

### 4.4.3 内容结构与示例

**组织方式**：按任务类型分节点，每个节点下 1-3 条。

```
[L3·场景核心规律]

- 短篇创作：
  • 情绪定调优先于情节展铺
  • 结尾留白，不做价值总结
- 代码审查：
  • 先查架构合理性与边界条件，再查命名规范
  • 每类问题必须给修改建议，不只指出错误
- 问题排查：
  • 先定位日志根因，不凭经验猜测
  • 不确定的点标注“待验证”，不硬下结论
- 架构设计：
  • 先定义系统边界与数据流，再选技术栈
  • 评估扩展性和失败模式，不只写 happy path
```

**关键约束**：
- “规律”必须能驱动 Skill 设计：如果一条 L3 无法让 Skill 写出检查项，说明它还不够抽象或不够本质。
- 禁止出现文件名、工具名等易过期信息。
- 禁止出现“第三步该做什么”等操作步骤。

### 4.4.4 与技能层的关系

L3 是技能层的 **设计规范**。一个 Skill 应该是由对应的 L3 规律推导出的执行流水线。当闭环梳理发现执行结果有问题时，先检查 Skill 步骤（执行层），若 Skill 无错，再检查 L3 规律是否提炼有误（策略层）。

## 4.5 积分控制表——系统“反馈阈值调节器”

### 4.5.1 定位

积分控制表是控制论第二概念（积分控制）的物理载体。它是一个轻量的结构化日志，记录**重复出现的错误模式及其处置状态**。

### 4.5.2 物理存储

- **位置**：`memory` 主体区，独立节点。
- **操作**：Agent 在收到负面反馈时自动维护，无需你手动管理。

### 4.5.3 数据结构

| 字段     | 说明                                              | 示例               |
| :------- | :------------------------------------------------ | :----------------- |
| 类别     | A类（违反L2）/ B类（违反L3）/ C类（Skill 步骤错） | B类                |
| 错误模式 | 8 字以内的抽象提炼                                | “短篇结尾 AI 总结” |
| 首次日期 | 第一次出现日期                                    | 2026-05-10         |
| 累计次数 | 当前累计次数                                      | 2/2                |
| 阈值     | 默认为 2                                          | 2                  |
| 已下沉   | 是否已写入 L2/L3/Skill                            | 是                 |
| 下沉目标 | 下沉到哪个位置                                    | L3·短篇创作        |

### 4.5.4 运行逻辑

当用户给出负面反馈 → Agent 抽象为错误模式 → 在表中查找同类模式：
- 存在且已达阈值：提示用户，此模式已下沉。
- 存在且未达阈值：累计数 +1。若已达阈值，自动下沉并确认。
- 不存在：新建条目，次数置 1。

每次闭环梳理时，Agent 必须输出本表的完整状态。

---

## 4.6 执行层：技能（Skill）——操作流水线

### 4.6.1 定位

技能层是系统的**手和脚**，负责所有具体操作。它包括三类技能：

| 类型     | 示例                                                         | 对应控制论概念     |
| :------- | :----------------------------------------------------------- | :----------------- |
| 元技能   | `feedforward_startup`、`delivery_gate`、`control_loop_review` | 前馈、自监控、闭环 |
| 任务技能 | `short_story_writer`、`code_reviewer` 等                     | 由 L3 驱动         |
| 维护技能 | `user_profile_maintainer`                                    | 画像维护           |

### 4.6.2 技能设计规范

每个技能必须包含以下字段（以 `short_story_writer` 为例）：

```
- 触发条件：当用户要求写短篇小说时
- 依赖 L3：[L3·短篇创作] 情绪定调优先，结尾留白
- 前馈注入：加载 L2 审美取向 + L3 对应规律
- 执行步骤：
  1. 确认主题与字数
  2. 定情绪基调（冷感/暖感/悬疑）
  3. 写作（前三句建立画面感）
  4. ……
- 自检步骤：调用 delivery_gate 五关扫描
```

任何技能都不得在步骤中硬编码“用户偏好”，偏好必须通过前馈注入。这保证了当你修改 L2 的某个审美原则时，所有技能自动生效，无需逐一修改。

### 4.6.3 技能版本化

每个技能名带有版本后缀（如 `_v1`），更新时创建新版本，旧版保留备查。版本化让你可以随时回滚，也方便闭环梳理时对比修改。

## 4.7 数据流全景：一次任务如何跑遍各层

以“写一篇短篇小说”为例，展示完整数据流：

```
Task: 写短篇
  │
  ├─[1. 前馈启动] Skill: feedforward_startup
  │    ├─ 读 L1：确认前馈规则
  │    ├─ 读 L2：提取“冷感叙事、拒绝AI总结”
  │    ├─ 读 L3：提取“短篇：情绪定调优先，结尾留白”
  │    ├─ 检索存档：上次纠正“画面感不足”
  │    └─ 输出“启动清单” → 等确认
  │
  ├─[2. 任务执行] Skill: short_story_writer
  │    ├─ 注入前馈清单中的避坑项
  │    ├─ 按 L3 规律组织内容
  │    └─ 完成初稿
  │
  ├─[3. 交付自检] Skill: delivery_gate
  │    ├─ P0：硬性要求检查
  │    ├─ P1：AI痕迹 + 结构完整性
  │    ├─ P2：交付洁净
  │    └─ 全部通过 → 输出最终稿
  │
  └─[4. 用户反馈→积分控制]
       ├─ 负面反馈 → 抽象为模式
       ├─ 写入积分控制表
       ├─ 若累计≥2 → 下沉到 L2/L3/Skill
       └─ 等待下次任务的前馈调用
```

**闭环梳理（周期性运行）**：
1. 触发（用户说“起飞”）。
2. Agent 调用 `control_loop_review`。
3. 按 L1 的四标准 + 根因分析标准流程，审计所有 L2、L3、Skill、积分表。
4. 输出整改方案。
5. 调用 `user_profile_maintainer` 更新画像。

## 4.8 架构的自保机制

为防止系统自身“跑偏”或“膨胀”，内置三条自保规则：

1. **L1 不可自修改**：Agent 想改 L1 必须提建议并等确认。
2. **L2/L3 条目上限**：每类 10 条/3 条上限，超过则提示用户清理过时项。
3. **Skill 版本化与废止**：旧版技能在确认无用后归档或删除，防止堆积。

本章展示了一个**以控制论四概念为逻辑层、以记忆和技能为物理层、以数据流为运行神经**的完整系统架构。它本质上是一套“设计模式”，是对Agent记忆与技能系统的重新规划，而非增加新功能。
