# 第七章：常见问题与延伸思考

这是教程的最后一章，覆盖两个板块：部署和运行中可能遇到的典型障碍及其解决方法，以及将这套框架扩展到更复杂场景的思路。

---

## 7.1 部署相关问题

### Q1：我写入 L1 后，Agent 没有按五条铁律执行，怎么办？

**常见原因**：
- 记忆未置顶，被后续写入的内容覆盖或冲淡
- 上下文窗口因长对话被挤占，L1 被截断
- Agent 平台对记忆注入时机有延迟

**排查步骤**：
1. 让 Agent 复述 L1 五条铁律。若无法完整复述，说明记忆写入或读取有问题。
2. 若复述正确但仍不执行，在新会话开始时重申一句：`本次会话开始前，先回顾 memory 中 L1 元认知框架的五条铁律。`
3. 若问题持续，考虑将 L1 的核心规则精简为一两句后写入平台的“系统指令”或“system prompt”区域（如果支持），双保险。

---

### Q2：Agent 说它没有 `skill_manage` 功能或类似的技能管理工具

**两种方案**：

**方案 A（推荐）**：用记忆模拟技能。
- 在 memory 中创建标记为 `[Skill-技能名]` 的记忆条目，内容就是技能的操作步骤。
- 在 L1 中加入规则：`每次需要执行某技能时，从 memory 中检索对应的 [Skill-*] 条目并严格按步骤执行。`
- 缺点：占用记忆空间；优点：逻辑完整。

**方案 B**：简化架构，只用 L1 + L2 + L3 + 积分表。
- 放弃技能层的自动化，改为你手动在任务开始时提醒 Agent 遵循自检步骤。
- 适用于轻度使用或短期协作。

---

### Q3：初始化 L2/L3 时，Agent 对我的提炼完全不准确

这并不意味着架构有问题，而恰恰说明 Agent 之前对你的理解有偏差——这正是本教程要解决的问题。

**处理方法**：
- 不要全盘接受 Agent 的提炼。逐条修正，把它当成一次“校准对话”。
- 修正后明确告诉 Agent：`你之前对我的理解有偏差，我已经修正了 L2/L3 的内容。请在记忆中以我的修正为准。`
- 后续每次闭环梳理中，Agent 会自动更新理解。通常经过 2-3 次复盘后，L2/L3 的准确度会显著提升。

---

### Q4：积分控制表里条目越来越多，怎么办？

这是系统正常运行的信号——说明你确实在持续协作，Agent 在持续学习。

**维护策略**：
1. 每次“起飞”复盘时，检查积分表。
2. 对已经下沉且运行稳定的条目，可归档为“已固化”，减少活跃条目数。
3. 对归类错误或不再适用的条目，手动删除。

```
将积分控制表中“结尾AI总结”（已下沉，6个月内无复发）移到归档区，不再计入活跃条目。
```

---

### Q5：部署后发现前馈清单太冗长，影响效率

前馈清单的长度应与任务复杂度成正比。如果简单任务也输出长篇清单，调整方法：

**精简指令**：
```
修改 feedforward_startup：对于简单任务（如翻译、简短问答），前馈清单压缩为一行——“避坑：xxx”。只有复杂任务才输出完整启动清单。是否简单任务由你在执行时判断并告知我。
```

你也可以随时在任务开始时说“跳过清单直接做”，Agent 应按此执行。

## 7.2 运行中常见问题

### Q6：Agent 连续两次我没纠正的同类错误，但没有触发下沉

**直接排查**：
```
请输出积分控制表当前完整状态。
```

**可能原因**：
- Agent 对你的第一次反馈未正确归类（如归为 D 类偶发）。
- Agent 将两次反馈归为不同错误模式，而非同一条目。

**纠正方法**：
```
把[描述两次反馈]统一归为同一个错误模式：“[8字以内的模式名]”。请更新积分表并检查是否达到下沉阈值。
```

---

### Q7：Agent 在交付闸门报告中总是“全部通过”，但我发现了问题

这说明闸门的检查项不够细，或者 Agent 执行闸门时敷衍了事。

**加强方法**：
1. 在发现问题的当下指出来：`delivery_gate 第三关没有发现“值得注意的是”这句套话。请解释原因，并更新闸门的检测标准。`
2. 如果特定问题重复出现，在闸门中增加针对性的检查规则：
```
在 delivery_gate 第三关的检查项中增加：“是否包含‘在某种程度上’、‘可以说’等模糊限定词？发现即删除。”
```

---

### Q8：闭环梳理的输出太模板化，缺乏真正有价值的根因分析

这可能是因为 Agent 在一个较长阶段内没有遇到真正的问题，也可能是流程设置导致 Agent 倾向于“填表”而非“思考”。

**激活方法**：
- 在“起飞”时追加具体指向：`本次复盘重点放在[某个具体问题]上，请深入根因分析。`
- 如果长期没有问题，告诉 Agent：`本次闭环如果没有发现任何需修正的问题，可以缩短为3句话简报，不需要完整九步。`

---

### Q9：我担心 Agent 的记忆越写越多，最终超限

这是架构设计中已经考虑的问题。三个内置机制控制膨胀：

1. **层级分离本身即抗膨胀设计**：原则和规律的体积远小于操作步骤和案例。
2. **L2/L3 条目上限**：L2 上限 10 条，L3 每个场景上限 3 条。超过时提示清理。
3. **闭环梳理中的净化步骤**：每次复盘都检查并迁移混淆内容。

如果你仍然担心，可以定期手动清理：
```
列出 memory 中最近 30 天未被访问过的条目，评估是否可以删除。
```

## 7.3 延伸思考

### 7.3.1 从单 Agent 到多 Agent 协作

本架构的核心——控制论四概念 + 层级分离——天然适用于多 Agent 场景。

**扩展思路**：
- **主 Agent**（编排者）：持有 L1 元认知框架，负责任务分发和质量把控。
- **子 Agent**（执行者）：各自持有特定场景的 L3 规律和对应 Skill。
- **共享记忆区**：L2 用户底层逻辑对所有 Agent 可见，确保不同角色都遵循统一的审美和原则。
- **统一积分表**：所有 Agent 的负面反馈汇集到同一积分表，由主 Agent 执行下沉决策。

这本质上是把“层级分离”中的“策略/执行”边界，映射为“主 Agent / 子 Agent”的物理边界。

---

### 7.3.2 引入其他理论框架

教程全程使用钱学森工程控制论作为底层框架。但 L1 不是封闭的——你的学习迁移能力可以通过“迁移提议”步骤持续为系统引入新理论。

已在评论区出现过的延伸尝试：

| 引入理论              | 应用方向         | 可能对应 L1 的扩展                         |
| :-------------------- | :--------------- | :----------------------------------------- |
| 积极的投资管理        | 决策与资源分配   | 在 L1 增加“动态再平衡”原则                 |
| 贝叶斯推理            | 不确定性下的判断 | 在积分控制中加入“先验概率”修正反馈权重     |
| 凯利准则 + 香农信息论 | 信息价值评估     | 在自监控中加入“信息密度”检查               |
| TRIZ 发明原理         | 矛盾解决         | 在根因分析中加入“物理矛盾 vs 技术矛盾”判定 |

引入新理论的方法：在“起飞”复盘时，当 Agent 提出迁移建议，或在你有明确想法时，主动说：
```
我想在 L1 中引入[理论名]，原则是[一句话描述]。请设计如何在现有的四概念框架下增加第⑥条铁律，并写明它如何影响 Skill 运行。
```

### 7.3.3 框架的适用边界再思考

评论区中出现过一个值得认真对待的质疑：**“工程控制论对现在的 Agent 来说未必是必需，或者这只是在上下文里给 LLM 加步骤，不算强约束。”**

经过本教程的完整展开，对这个质疑可以给出以下回答：

1. **“给 LLM 加步骤”本身就是有效的约束**。LLM 是 promptable 系统，结构化的上下文约束确实能改变其行为分布。效果不来自硬编码的强制性，而来自持续的结构化引导。

2. **架构的价值不在单次执行，而在长期稳定性**。单次来看，你不给 Agent 任何框架，只靠随口指令也能拿到合格结果。但连续 50 次任务后，有架构的系统收敛得更好，漂移更小。这是一种保险，不是一种燃料。

3. **不是所有场景都需要**。轻度的、一次性的、高度随机的对话不需要这套架构。它的适用边界是：**长期协作 × 高精度需求 × 可归纳规律的任务类型**。如果你恰好处于这个区域内，那么投入的搭建成本会在减少重复纠错的收益中收回。

---

### 7.3.4 框架的自我终止条件

一个健康的系统应能识别“我什么时候该被简化”。

**终止或降阶的信号**：
- 积分控制表连续 3 个月无新增条目（说明协作已完全稳定，或协作频率大幅下降）。
- 每次“起飞”复盘都无实质发现。
- 你发现自己在协作中不再需要前馈清单，偏好已高度内化。

当这些信号出现时，可以降低闭环频率（如从每周改为每月），或简化架构（如去掉积分表，保留前馈和自监控）。

这是一个系统自我谦逊的设计选择：**它追求的不是永远存在，而是在被需要时有效运行，在不被需要时优雅缩减。**
